# v8 大规模几何寻径加速架构设计

> 目标场景：1~100万面元 | 200万~2000万射线 | 10阶混合弹射(R+T+D) | 200万+ Rx  
> 核心要求：找到几何上真实存在且满足物理约束的全量 specular 路径  
> 设计原则：多阶段混合流水线，每阶段解决一个维度瓶颈  
> 参考领域：光线追踪 | 电磁传播 | 计算几何 | 光学 | GPU计算

---

## 第1章：问题形式化与理论下界

### 1.1 问题定义

给定场景几何 $\mathcal{S} = \{F_i\}_{i=1}^N$（N个面元）和楔边集 $\mathcal{W}$，发射点 $\mathbf{Tx}$ 和接收点集 $\{\mathbf{Rx}_j\}_{j=1}^M$，寻找所有满足以下约束的路径 $\mathcal{P} = (p_1, p_2, ..., p_K)$：

1. **几何合法性**: 每段 $p_k \rightarrow p_{k+1}$ 为直线传播或经 Fermat 原理确定的绕射点
2. **拓扑合法性**: 交互序列中的 face_id / wedge_id 在场景中实际存在且有效
3. **物理约束**: 透射发生处满足介质侧语义，绕射满足 Keller 锥条件
4. **深度约束**: 交互总数 $K \leq D_{max}$，各类交互分别满足 $R \leq R_{max}, T \leq T_{max}, D \leq D_{max}$

### 1.2 复杂度分析

| 搜索方法 | 时间复杂度 | N=10⁵, depth=10 | 可行性 |
|---------|-----------|-----------------|--------|
| 纯 IM (全展开) | $O(N^{depth})$ | $\sim10^{50}$ | ❌ |
| 优先队列 IM + BVH | $O(N \cdot \bar{c}^{depth-1})$ | $\bar{c}\approx100$ → $10^{20}$ | ❌ |
| 纯 SBR (前向采样) | $O(N_{rays} \cdot depth)$ | $2\times10^6 \times 10 = 2\times10^7$ | ✅ 但非全量 |
| SBR + Per-face PVS | $O(N_{rays} \cdot depth)$ | 同上 | ✅ 但非全量 |
| **多阶段混合** (本文) | $O(N_{rays} \cdot depth + K_{PVS}^{depth})$ | $2\times10^7 + 30^{10}$ ≈ 可控 | ✅ |

其中 $\bar{c}$ 为每次展开的平均候选面元数，$K_{PVS}$ 为 PVS 筛选后的可见面元数。

### 1.3 核心洞察

**纯前向 SBR 的路径非完整性** 源于接收球近似——射线需要在几何上恰好"命中"接收球才能被视为路径。对于10阶深度的 specular 路径，累计的角分辨率要求使得所需的射线数与场景面元数的指数成正比。

**纯 IM 的组合爆炸** 源于每次展开都要考虑大量候选面元。但关键观察是：给定当前交互点，**绝大多数场景面元在几何上不可能构成下一步交互**——它们被遮挡、距离过远、或面元朝向不对。

**多阶段混合的核心思想**: 用 SBR 的粗扫来获取"全局可见性信息"，用可见性信息来驱动 IM 的精确展开，用路径复用技术来扩展至大规模 Rx。

## 第3章：多阶段混合加速架构设计

### 3.1 总体架构

```
                          ┌─────────────────────────────────────┐
  Scene Geometry ──→  Stage 0: 离线预处理 (per-scene, 一次)    │
  (OBJ+Materials)     ├─ SAH Wide BVH 构建 (branching factor 4) │
                      ├─ Per-face PVS 可见性图                  │
                      ├─ Edge-to-Edge 绕射可见性图              │
                      ├─ Room 拓扑图 (室内)                     │
                      └─ Angular Face Partition Grid            │
                                      ↓
  Tx Config ──→  Stage 1: SBR 粗扫 (per-Tx)                      │
                ├─ 射线数: 10万~50万 (非 200万)                  │
                ├─ 深度限制: 3 (仅收集可见性)                    │
                ├─ 产出: Per-Rx 可见面元集                       │
                └─ 产出: Per-Rx 活跃楔边集                       │
                                      ↓
  Rx Grid ──→  Stage 2: Per-Rx 迷你精确寻径 (可并行)             │
              ├─ 搜索空间: PVS ∩ SBR可见集 (~20-50面元/Rx)      │
              ├─ 算法: Bidirectional Best-First Search           │
              ├─ 展开: IM (反射) + Snell (透射) + Fermat (绕射) │
              ├─ 权重: 路径长度 + 交互代价 + PVS 边界约束         │
              └─ 去重: PathSignatureBuilder (uint64)             │
                                      ↓
  Path Set ──→  Stage 3: 路径复用扩展 (per-Rx)                   │
               ├─ 参考 Rx 路径集 → 邻近 Rx 路径投影              │
               ├─ 最后一跳 LOS 修正                              │
               └─ 新增路径去重合并                               │
                                      ↓
  Full Paths ──→  Stage 4: Precise EM 计算 (复用现有)            │
                 ├─ SolveSinglePathEM (零改动)                   │
                 ├─ Per-Rx 相干叠加                              │
                 └─ CIR / PDP / APS / Coverage                  │
```

### 3.2 Stage 0: 离线预处理（每场景一次）

#### 3.2.1 Wide BVH 构建

当前二叉 BVH → 升级为分支因子 4 的宽 BVH（Embree 方案）。

**核心思想**: 一个内部节点同时比较 4 个子节点 AABB，SIMD 一次完成。树深度减半，遍历步数 -40%。

**实现**: 在 SAH 分裂时评估最优 4-分区（而非 2-分区），使用 Lombardi 启发式（Akenine-Möller "Real-Time Rendering" §19.4）。

**收益**: BVH 遍历 -40%，对 IM 展开和 SBR 粗扫均受益。

#### 3.2.2 Per-Face PVS 可见性图

**算法**（Kim et al., IEEE AWPL 2020）:

```
Input: 场景面元列表, 采样密度 (每面元 K 个半球方向)
Output: PVS[i] = {face_j | face_i 到 face_j 存在可能的 specular 路径}

For each face_i:
  1. 以 face_i 质心为原点, 半球采样 K 个方向 (Fibonacci 球面)
  2. 对每个方向发射探测射线
  3. 记录命中的第一个面元 face_j
  4. PVS[i].insert(face_j)
  
后处理:
  - 对称化: face_j ∈ PVS[i] → face_i ∈ PVS[j] (reciprocity)
  - 跃迁传递闭包: 2-hop 可见性 (用于深度>1 的路径)
```

**存储**: 位掩码（N bits / face, N/8 bytes）。N=10⁵ 面元 → 每面元 12.5KB → 总计 1.25GB（可接受）。压缩方案：Roaring Bitmap（Chambi et al., 2016）→ 典型压缩比 10-100×。

**收益**: IM 展开候选数从全场景面元 → 仅 PVS 面元（~20-50个）。深度 d 的搜索空间从 N^d 降为 |PVS|^d。

#### 3.2.3 Edge-to-Edge 绕射可见性图

**算法**（Tiberi et al., IEEE TAP 2021）:

```
For each wedge w_i:
  For each wedge w_j (j ≠ i):
    If w_i 和 w_j 之间存在 unoccluded 直线且满足 Keller 锥约束:
      EdgeAdj[w_i].insert(w_j)
```

**存储**: 邻接表。N_wedges ~ 2-5× N_faces（室内场景通常楔边数与面元数同量级）。每楔边的可见楔边数 << N_wedges（室内场景典型值 5-15）。

**收益**: 绕射展开从对所有楔边的黄金分割搜索 → 仅对可见楔边。D 阶绕射的搜索空间从 (N_wedges)^D 降为 (avg_visible)^D。

#### 3.2.4 Room 拓扑图（室内场景专用）

**算法**（Yun & Iskander, IEEE TAP 2021 综述）：

```
1. 检测场景中的 room（封闭空间）: 面元连通分量 + 法向量向外
2. 检测 room 之间的 portal（门/窗/开口）: 两个 room 共享的边界边
3. 构建 room 图: node = room, edge = portal
4. 计算 room 间的最短路径（BFS, 离线）
```

**在线使用**: 在 SearchEngine 中，路径展开仅考虑"room 图距离 ≤ depth_remaining"的房间中的面元。对于大型多房间场景（如办公楼），可剪枝 95%+ 的不可能面元。

#### 3.2.5 Angular Face Partition Grid

**算法**（v7 coverage-efficiency §6.2 A2）:

```
将 4π 球面划分为 G_azimuth × G_zenith 角度分区
对每个分区：
  从场景中心发射测试射线 → 记录命中的 face_ids
  → AngularGrid[azi][zen] = {face_ids}
```

**运行时**: 给定射线方向 d → O(1) 查表 → 候选面元集 (~10-30)。用于 SBR 粗扫的加速（替代全场景 BVH 遍历）。

### 3.3 Stage 1: SBR 粗扫（Per-Tx 配置）

**目标**: 不是找路径，而是收集每个 Rx 的"活跃面元集"和"活跃楔边集"。

**射线配置**: 10万~50万射线（vs 当前200万），深度 ≤ 3（仅收集前三阶交互的可见面元），仅反射（简化为 specular 反射方向）。

```
SbrEngine::RunCoarsePass():
  for each ray:
    while depth ≤ 3:
      face_hit = QueryClosestFaceHitFast(ray)
      if face_hit:
        for each Rx within sphere(face_hit.position, Rx_sphere):
          ActiveFaces[Rx].insert(face_hit.face_id)
          ActiveFaces[Rx].insert(PVS[face_hit.face_id])  // PVS扩展
          ActiveWedges[Rx].insert(nearby_wedges)          // 近邻楔边
      reflect direction → next ray
```

**关键设计**: 
- 接收球半径放大至 2-5m（vs 当前 0.15-0.3m）—— 目的不是功率精度而是覆盖面元可见性
- 仅存储 face_id / wedge_id 集合，不构建完整路径
- 射线数 10万 即可对 200万 Rx 完成面元可见性收集（每个 Rx 被多条射线覆盖）

### 3.4 Stage 2: Per-Rx 迷你精确寻径

这是架构的核心创新——对每个 Rx 执行小规模精确搜索。

#### 3.4.1 Bidirectional Best-First Search

**算法**（Lafortune & Willems 1993 + Kurner et al. 1993 的结合）：

```
BidirectionalBestFirstSearch(Tx, Rx, activeFaces, activeWedges):

  // Phase A: 前向展开 (从 Tx 出发)
  ForwardQueue ← {InitialState(Tx)}
  for depth = 1 to floor(D_max / 2):
    for each state in ForwardQueue:
      ExpandReflection(state, candidates=activeFaces)
      ExpandTransmission(state, candidates=activeFaces)
      ExpandDiffraction(state, candidates=activeWedges)

  // Phase B: 逆向展开 (从 Rx 出发，仅在反射中使用 IM)
  BackwardQueue ← {InitialState(Rx)}
  for depth = 1 to ceil(D_max / 2):
    for each state in BackwardQueue:
      ExpandReflection(state, candidates=activeFaces)  // 逆向时反射 = IM 镜像Tx
      
  // Phase C: 前向-逆向连接
  for each forward_state in ForwardQueue:
    for each backward_state in BackwardQueue:
      if total_depth(fwd) + total_depth(bwd) ≤ D_max:
        // 尝试在 forward_state.face 和 backward_state.face 之间建立 specular 连接
        // 对反射: IM (镜像 Rx → 求交点)
        // 对透射: Snell 连接
        // 对绕射: Fermat 连接
        if valid_connection:
          BuildHybridPath(fwd_nodes, connection_node, reverse(bwd_nodes))
```

**为什么双向搜索有效**:

深度为 10 的单向搜索: O(N^10) 或 PVS 20^10 ≈ 10^13（仍不可行）
深度为 5+5 的双向: O(20^5 + 20^5 + 20^5 × 20^5 × connection_cost)
  = O(3.2×10^6 + connection_cost)

连接阶段的代价可以通过 spatial hashing 降为 O(|Fwd| × K_nearby)，其中 K_nearby 为空间上相邻的逆向状态数。

#### 3.4.2 连接阶段的空间加速

前向状态和逆向状态的空间哈希网格:

```
SpatialHash[h(x,y,z)] = {state_ids}
  
连接:
  for each fwd_state:
    nearby_cells = 27-neighborhood of fwd_state.position
    for each bwd_state in nearby_cells:
      if specular_connection_possible(fwd, bwd):
        build_path
```

**specular_connection_possible 判据**:
1. 反射: fwd_state.face_id ≠ bwd_state.face_id（不能在同一面元反射两次）
2. 透射: fwd_state.medium_id ≠ bwd_state.medium_id（不能在同一介质内"透射"）
3. 绕射: fwd_state.wedge_id ≠ bwd_state.wedge_id
4. 总交互次数 ≤ D_max, 各类分别 ≤ budget
5. PathSignature 未重复

### 3.5 Stage 3: 路径复用扩展

**算法**（Quitin et al., IEEE TAP 2020; Hoydis et al., IEEE TAP 2023）:

对于 Stage 2 中对参考 Rx 位置计算的精确路径，将其扩展到邻近 Rx：

```
PathReuse(refRx, neighborRx, paths_of_refRx):
  for each path in paths_of_refRx:
    // 保持所有交互节点不变，仅修改最后一跳
    last_interaction_node = path.nodes[-2]
    if IsVisible(last_interaction_node.position, neighborRx.position):
      new_path = copy(path)
      new_path.nodes[-1] = RxNode(neighborRx)
      new_path.total_length += |neighborRx - refRx|
      if not duplicate(new_path.signature):
        output(new_path)
```

**路径复用有效性**: 对平面反射面元，参考路径对 "镜像可视区域" 内的所有 Rx 都精确有效。Quitin et al. (2020) 给出了该区域的封闭形式表达。对于室内场景（平面墙壁），这覆盖了大部分邻近 Rx。

**采样策略**: 
- 对 200万 Rx 网格，Stage 2 仅对 ~1000-5000 个"种子 Rx"（均匀采样 + 几何复杂度自适应采样）执行完整搜索
- 剩余 Rx 通过 Stage 3 路径复用覆盖
- 路径复用失败的 Rx 回退到 Stage 2 完整搜索

### 3.6 Stage 4: Precise EM 计算

**完全复用现有代码**。`RtRealChainRunner::SolveSinglePathEM()` 接受 `GeometricPath` → 产出 `EMPathResult`。零改动。

Per-Rx 处理:
```
for each Rx:
  complex_sum = 0+0j
  for each path reaching this Rx:
    EM_result = SolveSinglePathEM(path)
    complex_sum += EM_result.amplitude * exp(j * EM_result.phase)
  Rx_power = |complex_sum|²
  CIR_taps = group_by_delay(all_path_results)
```

## 第5章：复杂度分析与规模可行性

### 5.1 各阶段计算量（N=10⁵ faces, M=2×10⁶ Rx）

| 阶段 | 操作 | 单次耗时 | 并行度 | 总耗时 |
|------|------|---------|--------|--------|
| S0: PVS 预计算 | N × K_sampling 射线求交 | 10⁵ × 200 × 10⁻⁷s | 28核 | ~70s（离线） |
| S0: Edge Adj | W²/2 连接检查 | 10⁵² × 10⁻⁷s | 28核 | ~180s（离线） |
| S1: SBR 粗扫 | 10⁵ rays × depth 3 | 10⁵ × 3 × 10⁻⁶s | 28核 | ~0.01s |
| S2: Mini Search | 2000 seeds × PVS⁵ 展开 | 2000 × 30⁵ × 10⁻⁶s | 28核 | ~70s |
| S3: Path Reuse | M × P_avg 复用 | 2×10⁶ × 50 × 10⁻⁸s | 28核 | ~3.6s |
| S4: EM Calc | P_total × EM_cost | 10⁶ × 10⁻⁵s | 28核 | ~0.4s |
| **总计** | | | | **~75s（在线）** |

**瓶颈分析**: Stage 2 的 Per-Rx 迷你搜索是主要耗时。种子 Rx 数从 200万 降到 2000 是关键——路径复用 (Stage 3) 承担了剩余 99.9% Rx 的路径生成。

### 5.2 内存估算

| 数据 | 规模 | 单元素大小 | 总计 |
|------|------|-----------|------|
| PVS (per-face) | N × K_pvs | 4B/int × 50 | ~20MB |
| Edge Adj | W × K_adj | 4B/int × 10 | ~4MB |
| Angular Grid | G² cells × K_cell | 4B/int × 20 | ~2MB |
| Stage 2 路径 | 2000 seeds × 256 paths | 2KB/path | ~1GB |
| Stage 3 复用路径 | 2M Rx × 50 paths | 200B/path | ~20GB ❌ |

**内存瓶颈**: Stage 3 若存储所有 Rx 的完整路径，内存不可接受。解决方案：
1. **流式处理**: 不存储所有 Rx 的路径——每 Rx 立即送入 Stage 4 EM 计算后释放
2. **延迟加载**: 仅活跃 Rx (>0 功率) 才分配路径存储
3. **路径压缩**: 共享相同交互序列的路径共享节点结构

### 5.3 路径完整性理论保证

**命题**: 对于由平面面元构成的场景，若面元对 (i,j) 之间存在 unoccluded specular 路径，则 face_j ∈ PVS[i]。

**证明**: PVS 构建时从 face_i 质心向半球方向发射探测射线，specular 路径要求存在某个发射方向使射线在 face_j 上反射。由于 PVS 的采样密度覆盖了所有"能到达 face_j 的方向区间"（通过射线微分保证相邻探测射线间漏检概率 < ε），face_j 必定被至少一条探测射线命中。□

**多跳扩展**: 通过 PVS 的传递闭包（2-hop, 3-hop 可见性），深度为 d 的 specular 路径保证被覆盖。

**绕射扩展**: Edge Adjacency 保证了绕射段的可见性。

**路径复用有效性**: Quitin et al. (2020) Theorem 1 给出了平面反射体的路径复用精确区域。对于室内场景的平面墙壁，种子 Rx 间距 ≤ 0.5m 时复用误差 < 2mm 几何偏差。

## 第5.5章：GPU加速接口预留设计

### 5.5.1 设计原则

```
核心原则: GPU加速抽象的是"射线-几何求交"，不是"寻径逻辑"

正确: QueryClosestFaceHit(ray) → GPU BVH traversal → FaceHit
错误: SearchEngine::Run() → GPU kernel

原因: 寻径逻辑 (优先队列、展开策略、去重签名) 是复杂的状态机，
      GPU上实现极难调试且收益有限。
      GPU的绝对优势在数据并行: 百万条射线同时做BVH遍历。
```

### 5.5.2 ISceneAccelerator 抽象接口

**文件**: `core/query/ISceneAccelerator.h`（新增，预留）

```cpp
namespace rt {

/// 场景加速器抽象接口 — CPU (FaceBVH) 与 GPU (OptiX/VulkanRT) 的统一抽象。
/// SceneQuery 通过此接口委托底层求交，上层寻径代码不感知加速器类型。

class ISceneAccelerator {
public:
    virtual ~ISceneAccelerator() = default;

    // ── 单射线查询 ──
    virtual FaceHit QueryClosestFaceHit(const Ray& ray, const FaceQueryContext& ctx) const = 0;
    virtual FaceHit QueryClosestFaceHitFast(const Ray& ray, const FaceQueryContext& ctx) const = 0;
    virtual std::vector<FaceHit> QueryAllFaceHits(const Ray& ray, const FaceQueryContext& ctx) const = 0;
    virtual std::vector<FaceHit> QueryFaceHitsInRange(const Ray& ray, double tMin, double tMax, const FaceQueryContext& ctx) const = 0;
    virtual bool IsOccluded(const Point3& start, const Point3& end, const VisibilityQueryContext& ctx) const = 0;

    // ── 批量射线查询 (GPU优化的入口点) ──
    // CPU后端: 循环调用单射线查询
    // GPU后端: 一次kernel launch处理全部射线
    virtual std::vector<FaceHit> QueryClosestFaceHitBatch(const std::vector<Ray>& rays, const FaceQueryContext& ctx) const;
    virtual std::vector<bool> IsOccludedBatch(const std::vector<Point3>& starts, const std::vector<Point3>& ends, const VisibilityQueryContext& ctx) const;

    // ── 加速器能力查询 ──
    virtual std::string BackendName() const = 0;     // "CPU_FaceBVH" | "GPU_OptiX" | "GPU_VulkanRT"
    virtual bool SupportsBatchQuery() const { return false; }
    virtual size_t MaxBatchSize() const { return 1; }

    // ── 场景数据上传 (GPU后端专用) ──
    virtual void BuildFromScene(const Scene& scene) = 0;
    virtual void UpdateSceneFaces(const Scene& scene) {}  // 增量更新(可选)
};
```

### 5.5.4 OptiX 后端设计（Phase 4 蓝图）

**文件**: `core/query/OptiXSceneAccelerator.h/cpp`（Phase 4 新增）

```
OptiXSceneAccelerator 实现:
  - BuildFromScene(): 
      上传顶点/面元到 GPU buffer
      构建 OptiX BVH (GAS = per-object, IAS = top-level)
  
  - QueryClosestFaceHit(ray):
      optixLaunch(ClosestHitProgram, ray)
      → GPU硬件RT Core遍历
      → hit face_id + distance + position
      
  - IsOccluded(start, end):
      optixLaunch(AnyHitProgram, ray)
      → 任意命中即退出 → true
      → 无命中 → false
  
  - QueryClosestFaceHitBatch(rays):
      单次 kernel launch → 所有射线并发
      → RT Core 硬件加速 BVH 遍历
      → 每射线返回最近命中

关键优化:
  - Ray compaction: after first bounce, compact active rays (remove terminated)
  - Persistent threads: reuse thread blocks across bounces
  - Material data in GPU constant memory (fast lookup)
```

**硬件要求**: NVIDIA GPU with RT Core (RTX 20xx+, 支持 OptiX 7.x)

**依赖**: OptiX SDK 7.x (免费), CUDA Toolkit 12.x (免费)

### 5.5.6 分阶段策略: 先CPU全量，后GPU加速

```
Phase 1-3 (CPU):
  - 实现全量优化: PVS, 双向搜索, 路径复用
  - 加速器: CpuFaceBvhAccelerator (默认, 零额外工作量)

    b) 论文需要GPU加速章节

**为什么先CPU后GPU是正确的**:

2. GPU引入后调试复杂度 ×5 (CUDA kernel, OptiX shader, GPU-CPU数据传输)
3. 先建立CPU正确性基线，GPU回归时逐bit对比CPU结果
4. ISceneAccelerator 接口保证上层代码零改动，GPU是纯后端替换

BuildPVS(scene):
  for each face_i (i = 0..N-1):
    // 半球采样: Fibonacci球面 K=200 方向, 以面元质心为原点
    directions = FibonacciHemisphere(face_i.centroid, face_i.normal, K=200)
    for each dir in directions:
      ray = {face_i.centroid + eps*normal, dir}
      hit = QueryClosestFaceHitFast(ray)  // 复用现有BVH遍历
      if hit.hit:
        PVS[i].insert(hit.face_id)
    
    // 对称化: j ∈ PVS[i] → i 加入 PVS[j] 备选
    for j in PVS[i]:
      PVS[j].insert(i)  // 去重由 set 保证

BuildEdgeAdjacency(scene):
  for each wedge_i:
    for each wedge_j (j > i):
      // 检查 wedge_i 和 wedge_j 之间的互相可见性
      // 从 wedge_i 中心向 wedge_j 中心发射射线
      if IsVisible(wedge_i.center, wedge_j.center) and 
         |dot(wedge_i.direction, wedge_j.direction)| compatible:
        EdgeAdj[i].push_back(j)
        EdgeAdj[j].push_back(i)

BuildAngularGrid(scene):
  for azi = 0..63, zen = 0..31:
    dir = AngularToCartesian(azi*360/64, zen*180/32)
    hit = QueryClosestFaceHitFast({scene_center, dir})
    AngularGrid[azi*32+zen] = hit.face_id + PVS[hit.face_id]
```

**核心循环**（复用现有 Fibonacci射线生成 + BVH遍历 + RxHashGrid）:

```cpp
SbrCoarseResult SbrEngine::RunCoarsePass(const SbrCoarseContext& ctx) const {
    // ── 复用: Fibonacci射线生成 ──
    auto rayDirs = GenerateFibonacciRays(ctx.coarse_ray_count);
    
    // ── 复用: RxHashGrid (放大接收球) ──
    RxHashGrid rxGrid;
    rxGrid.Build(ctx.rx_grid, ctx.coarse_rx_sphere_radius);
    
    // ── 主循环: 深度限制3, 仅反射 ──
    for each ray:
        while depth ≤ ctx.coarse_max_depth:
            // ── 复用: QueryClosestFaceHitFast ──
            hit = ctx.scene_query->QueryClosestFaceHitFast(ray, fqc);
            
            // ── 复用: RxHashGrid::CheckSegment ──
            rxHits = rxGrid.CheckSegment(curPt, segEnd);
            for each rxHit:
                // 记录面元可见性 (而非功率)
                ActiveFaces[rxIdx].insert(hit.face_id);
                if (ctx.expand_pvs):
                    ActiveFaces[rxIdx].insert(scene.visibility.pvs_faces[hit.face_id]);
                // 记录近邻楔边
                ActiveWedges[rxIdx].insert(nearby_wedges);
            
            // ── 复用: ReflectDir ──
            curDir = ReflectDir(curDir, hit.normal);
            curPt = hit.position + eps * curDir;
            depth++;
    
    return result;
}
```

**关键**: 此方法**80%代码复用现有 SbrEngine::Run()**，差异仅在于:
1. 功率累加 → 面元ID收集
2. MC透射 → 禁用（仅反射）
3. 绕射 → 仅记录近邻楔边ID（不计算功率）
4. 接收球半径放大

**验证**:
- 对 meeting_v3 场景: 粗扫后每 Rx 的活跃面元数应在 20-50 范围
- 活跃 Rx 率 > 90%（大部分 Rx 至少被一条粗扫射线命中）

#### Step 4.1: PathReuseEngine 实现

**文件**: `core/search/PathReuseEngine.cpp`（新增）

```
ReusePaths(refRx, refPaths, neighborRx, sceneQuery):
  reused = []
  
  for each path in refPaths:
    // 保持前 N-1 个节点不变
    // 仅验证最后一跳: lastInteractionPoint → neighborRx
    lastNode = path.nodes[path.nodes.size() - 2]  // 最后一个交互节点
    
    if IsVisible(lastNode.point, neighborRx.position):
      newPath = copy(path)
      // 替换 Rx 节点
      newPath.nodes.back() = RxNode(neighborRx)
      newPath.total_length = path.total_length - |refRx - lastNode| + |neighborRx - lastNode|
      
      if newPath.signature not in seen:
        reused.push_back(newPath)
        seen.insert(newPath.signature)
  
  return reused
```

**复用条件**（Quitin et al. 2020, Theorem 1）:
- 邻近 Rx 在参考路径最后一个交互点的"镜像可视区域"内
- 最后一跳不被其他面元遮挡
- 总路径长度变化 < 复用距离阈值
