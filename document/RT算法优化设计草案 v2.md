# RT算法优化设计草案 v2

## 0. 文档定位

本文档是 `RT算法优化开发文档 v2.md` 的配套详细设计草案。

其定位为：

1. 第二轮优化的详细设计字典；
2. 模块级、结构级、字段级、流程级讨论沉淀库；
3. 后续每次讨论时记录“已敲定项 / 候选方案 / 待定项 / 风险点”的正式载体；
4. 开发主文档在需要进一步下钻时的唯一细节参考。

本文档不是主批次推进文档，不直接替代开发文档；
第二轮开发顺序、优先级和通过标准，仍以 `RT算法优化开发文档 v2.md` 为准。

## 1. 本文档的使用规则

后续每次讨论模块和批次细节时，建议遵循以下规则：

1. 先在开发文档 v2 中确认当前讨论属于哪个批次；
2. 再在本文档中补写该批次对应的详细设计；
3. 对每一项设计，尽量区分：
   - 已敲定；
   - 候选方案；
   - 暂缓决定；
   - 明确不采用；
4. 若某项设计会影响接口、导出格式、验证标准，必须同步回写开发文档 v2；
5. 不允许只在对话中口头约定、不沉淀到本文档。

#### 13.1.18 BatchX资产分类与迁移设计定位

从设计视角，A0 第4节要解决的问题不是简单回答“删不删 BatchX 文件”，而是要冻结：

1. 它们当前属于哪类资产；
2. 它们当前承担什么职责；
3. 它们是否属于正式工程文件；
4. 后续由什么正式文件或正式体系接管；
5. 在什么条件下允许退场。

统一结论为：

> `BatchX` 文件不是永久正式命名，也不是可立即丢弃的临时文件，而是必须被规范接管的过渡闭环资产。

#### 13.1.19 BatchX资产分类（设计视角）

从设计视角，`BatchX` 资产统一分为四类：

1. **A类 过渡 Builder 类**：如 `SceneBatch2/3/4Builder`，后续由正式 scene / preprocess / cache pipeline 接管；
2. **B类 过渡 Reporter / Self-check 类**：如 `SceneBatchXReporter`、`Batch5~9Reporter`，后续由正式 validation / regression / report 体系接管；
3. **C类 README / 批次说明类**：如 `test/README_batch3~9.md`，后续由正式测试说明与验证说明体系接管；
4. **D类 批次脚本 / 辅助验证资产类**：如 `check_batch2_binding.py`，后续迁移到正式 tools/validation/visualization，或转归档保留。

#### 13.1.20 BatchX迁移设计原则（细化）

第二轮对 `BatchX` 迁移设计，统一采用以下原则：

1. 先分类，再接管，不先删除；
2. 按模块接管，不按批次继续膨胀；
3. 先建立正式接管者，再迁移职责；
4. 接管时必须同步迁移业务职责、验证职责与历史说明资产；
5. 允许归档保留，不强求物理彻底删除。

#### 13.1.21 BatchX按模块接管关系（设计视角）

1. **模块2**：最适合最早接管，未来指向正式 `SceneBuildPipeline`、`SceneDiagnostics`、`SceneValidation`、`SceneCacheValidation`；
2. **模块4**：中期重点接管，未来指向正式 `SearchValidation`、`SearchRegression`、`MixedPathScenarioCheck`；
3. **模块5**：与模块4联动接管，未来指向正式 `EMValidation`、`EMProfileValidation`、`AggregateResultValidation`；

##### 13.5.11 三类interaction 当前不足（设计视角）

1. **reflection**：更像“有一个反射影响”，对材料语义正式消费不足，对极化 / 相位变化解释力不足；

3. **diffraction**：更像“给绕射路径一个作用”，仍是模块5最弱的 interaction 之一，但必须明显前进。

##### 13.5.12 路径级传播链当前不足（设计视角）

1. 路径级相位传播目前还偏“有值”，不够“可解释”；
2. 极化传播链还未成为正式强约束主线；
3. 复振幅传播与机制响应关系还不够强。

1. aggregate 结果当前“能出”，但可信度上限由 path-level 求值决定；
2. 若 path-level 求值不升级，aggregate 再花哨也只是“更好看的粗结果”；

##### 13.8.2 A7当前问题定义

从设计视角，A7 当前主要面对以下问题：

1. validation 从存在性检查升级为质量检查；

3. 建立跨模块质量视角；
