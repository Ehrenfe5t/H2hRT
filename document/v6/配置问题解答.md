# RT 仿真配置 — 全量问题解答

> 对应 `configs/app/Q.txt` 38 项疑问。每项包含代码依据 + 分析 + 建议（不做修改，仅讨论）。

### Q1: production vs debug 模式区别？

**代码依据**: `app/main.cpp:23` — 默认配置路径在 debug 时可能不同；`AppConfigValidator.cpp` 中校验强度无模式区分。

**当前实际**: 两种模式仅在日志输出和自检行为上有差异：
- `debug`: 运行 `Module1SelfCheck`（负样例配置校验），输出 TRACE/DEBUG 日志
- `production`: 跳过自检，最低日志级别为 INFO

**建议**: 保留。`debug` 用于开发调试（跑完整自检链），`production` 用于批量仿真（跳过无效开销）。语义清晰。

### Q2: log_level 详细级别？

**代码依据**: `Logger.h` 中定义 6 级:
```
TRACE < DEBUG < INFO < WARN < ERROR < FATAL
```
- `TRACE`: 每步 BVH 遍历、每射线命中（海量输出）
- `DEBUG`: 函数进入/退出、中间变量
- `INFO`: 批次完成、路径数、SBR 统计
- `WARN`: 路径 EM 求解失败
- `ERROR`: 模块初始化失败
- `FATAL`: 核心闭环断裂（如 LOS 未建立）

**建议**: 生产环境 `WARN` 或 `INFO`。大场景 SBR 建议 `WARN`（避免每条射线 INFO 刷屏）。

### Q3: enable_console_logging 取消是否合理？

**分析**: 控制台输出是唯一的实时仿真进度反馈渠道。若取消此参数（常开），需提供替代进度指示（如 SBR 的百分比进度条）。

**建议**: 当前代码 SBR 无进度回调。暂保留此开关，v6 可改为 `console_log_level`（控制台独有级别）提供更细粒度控制。

### Q4: enable_file_logging — 为什么大场景有 IO 瓶颈？

**物理原因**: SBR 每射线每步都可能产生日志行。200K 射线 × 深度 6 = 1.2M 次交互，每次 `fprintf` + `fflush` → 数百 MB 日志文件 + 数秒 IO 等待。

**代码依据**: `Logger.cpp` 中每次 `Log()` 调用同步写文件。

**建议**: 常开但默认 `log_level=WARN` 可避免 IO 瓶颈。或改为异步日志（后台线程写）。

**分析**: 楔边构建耗时 < 0.1s（meeting.obj 400 面），但若配置 `enable_diffraction=false`（precise 和 SBR 都不做绕射），楔边构建是纯浪费。

### Q9: enable_scene_cache 常开是否合理？

**代码依据**: `SceneCache.cpp` 将预处理结果序列化到磁盘。大场景（100K 面）首次运行后，二次运行可跳过 OBJ 解析+材质绑定+BVH 构建（节省数秒至数十秒）。

**问题**: 当前缓存格式依赖 `reinterpret_cast` 写 `std::size_t`，32/64 位不兼容。

**建议**: v6 修复序列化可移植性后可常开。

### Q10: enable_bvh_bruteforce_validation 必要性？

**建议**: **删除此字段**。真正的 BVH 验证应该是一个独立的 debug 工具（硬编码调用），不需要从 JSON 配置控制。

### Q11: bvh_leaf_size 自适应？

**理论依据**: 
- Wald (2007): SAH 最优叶节点大小与场景面数 N 相关，通常 √N ~ N/100
- 实践经验: leaf_size=4~8 对小场景最优，16~32 对大场景最优

**建议**: 实现自适应: `auto_leaf = max(4, min(32, total_faces / 200))`。同时保留手动覆盖能力。

---

### Q12: wedge_min/max_angle_deg 是否有理论建议值？

**理论依据**: 
- UTD 在楔角→0°（共面）和→180°（平面）时退化
- Balanis Ch.12: UTD 有效范围是楔角在 10°~170°
- IEEE Std: 绕射系数在 α<5° 或 α>175° 时数值不稳定

**建议**: 取消这两个参数，硬编码为 10° 和 170°（与当前默认值一致）。这两个值来自 UTD 理论而非经验调参。

---

### Q13: filter_non_manifold_wedge_sources 取消？

**UTD 理论**: 非流形边（≥3 面共享）的绕射场是多个二面角绕射的叠加，UTD 标准公式不适用。

**建议**: 取消此开关，常开过滤。非流形边绕射需专门算法（v7+），不是简单的开关问题。

---

### Q14: skip_coplanar_edges_for_wedge 作用？

**定义**: 两个相邻面的二面角 ≈ 0°（共面）时的边。这种边没有"楔"的几何特征，不产生绕射。

**建议**: 取消此开关，常开。共面边在任何情况下都不应产生楔边。

### Q16: frequency_query_mode — 频率插值是否合理？（重点）

**你的观点完全正确。** 当前 `MaterialDatabase.h` 在相邻频点间做**线性插值**：

```
ε_r(f) = ε_r(f1) + (f-f1)/(f2-f1) * (ε_r(f2) - ε_r(f1))
```

**问题**: ITU-R P.2040 的 ε_r 和 σ 随频率的变化规律不是线性的。实际模型是：
```
ε_r(f) = a × f^b    (幂律)
σ(f)   = c × f^d    (幂律)
```

**代码依据**: Sionna RT 的 `itu.py` 使用精确的幂律拟合参数（a, b, c, d），而非线性插值。

**严格方案（你提出的）**: 
1. 检查 `frequency_hz` 是否在 ITU CSV 的频点列表中
2. 若精确匹配 → 用该频点的值
3. 若不匹配 → **报错退出**，要求用户使用 ITU 定义的标准频点（1G, 2G, 4G...）或扩展 CSV

**折中方案**: 实现 ITU 幂律插值 `ε_r = a*f^b` 替代线性插值。比线性插值准确，比严格报错灵活。

**建议**: 实现幂律插值（对标 Sionna），同时保留严格模式作为可选校验。

### Q27: ray_power_threshold 改为 dB 值？

**当前**: `ray_power_threshold_linear: 0.0001`（衰减 40dB 后停止）

**改为**: `ray_power_threshold_dB: -40`（含义：相对于 Tx 功率 0dB，衰减 40dB 后停止射线追踪）

### Q31: store_paths 详细讨论

**作用**: 每条 SBR 射线命中 Rx 时是否存储完整路径节点（`GeometricPath`）。

**存储开销**: 每条路径 ~200B × 平均 100 paths/Rx × 1M Rx = **20GB**（不可行）。

**必要性**: 
- 功率覆盖热力图：不需要（只需 `power_dBm` 标量）
- 可视化 SBR 路径轨迹：需要（但仅用于小规模调试）

**建议**: 常关（false）。若需可视化，单独写二进制格式（如 Protobuf flatbuffers）做序列化，而非 JSON。

### Q32: wedge_max_distance_m 作用

**你的理解正确**: 射线命中点周围此距离内的楔边才被考虑为绕射候选。

**全场景查找**: 可行但不是"遍历所有楔边"。应使用**楔边空间加速结构**（Wedge BVH 或网格哈希），O(log N) 而非 O(N)。v6 可设计。

**建议**: 保留此参数，但降低默认值（3.0m → 2.0m），室内绕射很少需要 >2m 的楔边距离。

### Q33: wedge_sample_count 作用

### Q34: frequency_hz 多频点支持？

**当前**: `em_solver.frequency_hz` 是标量。precise 和 coverage 共同使用同一频率。

**多频点**: 需要循环运行多次仿真（每次改频率），无法在单次运行中完成（因为不同频率的 Fresnel/UTD 系数不同，路径选择可能不同）。

**建议**: v6 支持 `frequency_sweep_hz: [2.4e9, 3.5e9, 5.0e9]`，内部循环运行。

### Q35: enable_polarization 作用？

**代码依据**: `ApplyReflectionInteraction.cpp` 始终使用 Jones 矢量。`enable_polarization=false` 的逻辑不存在——代码始终做 TE/TM 投影。

**precise 模式**: 极化应**永远开启**（Jones 矢量级）。这是 precise 的核心精度优势。

**coverage 模式**: SBR 用标量功率（Fresnel |Γ|² 平均），不跟踪极化方向。可以无极化开关。

**建议**: 删除 `enable_polarization`。precise 常开，SBR 用标量功率（在 SBR 代码中独立处理）。

### Q38: 数值容差是否有理论建议值？

**分析**: 6 个容差值的当前默认值已经合理，来自 IEEE 754 double 精度和几何计算的工程实践：

| 参数 | 当前值 | 理论依据 |
|------|--------|---------|
| `eps_length` | 1e-6 m | 1μm — double 精度在米级坐标下可分辨 |
| `eps_intersection` | 1e-7 m | 0.1μm — 射线-三角面交点的数值精度 |
| `eps_deduplicate` | 1e-5 m | 10μm — 两条路径的几何不可区分阈值 |
| `self_hit_ignore` | 1e-5 m | 同上 — 射线起点偏移防止自交 |
| `visibility_origin_offset` | 1e-5 m | 可见性射线起点偏移 |
| `visibility_target_shrink` | 1e-5 m | 可见性终点收缩 |

**自适应**: 这些值与场景绝对尺寸和坐标精度相关。对于 OBJ 格式（通常 4-6 位有效数字），当前值最优。不需要自适应。

**建议**: 保留当前默认值。删除 `eps_angle`, `eps_power`, `eps_normal` 三个未使用字段。
