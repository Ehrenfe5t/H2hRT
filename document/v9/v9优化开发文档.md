# v9 优化开发文档

## 1. 分析范围与结论

本文档基于 `RT.sln` 主工程进行全量全流程分析，覆盖 `app`、`preprocess`、`core/common`、`core/query`、`core/search`、`core/path`、`core/em`、`core/antenna`、`core/result` 等自实现代码。按要求，`算法\` 下参考实现暂不纳入本轮分析，`tools\SDK` 仅作为第三方依赖背景，不作为主实现评估对象。

当前工程已经形成较完整的射线追踪仿真链路：配置加载、OBJ 场景导入、材质绑定、拓扑恢复、BVH/OptiX 查询、几何路径搜索、SBR 覆盖粗扫、逐路径 EM 求解、CIR/PDP/APS/覆盖/ISAC 聚合与导出。整体架构清晰，模块边界基本合理，v8 已加入 PVS、Angular Grid、Coarse SBR、约束搜索、GPU wavefront/megakernel 等加速思路。

但当前实现仍更接近“工程可跑通原型”，尚未达到高精度 RT/EM 仿真器要求。核心风险集中在：透射几何与 Snell 闭合不一致、状态/路径签名过粗导致误去重、CPU/GPU 查询语义不一致、SBR 透射/绕射物理模型简化、配置/材质/OBJ 解析鲁棒性不足、若干硬编码剪枝会漏路径、测试闭环和参考对照不足。

v9 的主目标应是先修正“会产生明显错误物理结果或漏解”的问题，再补足可重复验证闭环，最后做性能和工程鲁棒性优化。

## 2. 现有全流程功能

### 2.1 入口与主管线

入口位于 `app/main.cpp`，默认读取 `configs/app/meeting_412.json`，或使用命令行传入配置文件。主管线位于 `app/RtPipeline.cpp::RtPipeline::Run`，按批次串联：

1. 配置加载、日志初始化、配置校验、配置快照、自检。
2. 场景导入与材质语义恢复。
3. Edge/Wedge 拓扑恢复、场景诊断、Face BVH 与 Wedge 查询记录构建。
4. SceneQuery 查询门面、场景缓存、CPU 或 GPU 查询后端挂接。
5. 材质数据库加载，并将面元 surface 材质电参数预填到 face。
6. 可选 Stage 0 可见性预计算：PVS、edge adjacency、angular grid。
7. 可选 Stage 1 coarse SBR 粗扫与 Rx 种子采样。
8. 可选 Stage 2 约束搜索，否则执行全量 `SearchEngine` 搜索。
9. 可选 Stage 3 路径复用验证。
10. A1 真实生产链：对几何路径逐条 EM 求解，聚合并导出。
11. 可选 SBR 覆盖通道：对 Rx 网格做覆盖仿真，可保存 SBR 路径并执行 EM 后处理。

### 2.2 场景预处理

预处理主要位于 `preprocess`：

1. `OBJImporter.cpp` 支持 OBJ 的 `o`、`v`、`vn`、`f`，当前主要支持三角面 `v//n` 格式。
2. `MaterialRuleLoader.cpp` 读取场景材质映射，支持对象名、简单通配、surface/front/back 材质和反射/透射/绕射开关。
3. `ResolveFaceDualSideMaterial.cpp` 将对象级规则写入 face 的双侧材质、介质 id 和传播标志。
4. `EdgeBuilder.cpp` 从三角面恢复唯一边、相邻面、二面角、共面和非流形标志。
5. `WedgeBuilder.cpp` 从相邻非共面边构造可绕射 wedge。
6. `FaceBVHBuilder.cpp` 构建 face 查询记录和 BVH。
7. `WedgeAccelerationBuilder.cpp` 构建 wedge 查询记录，目前本质是扁平列表而非空间索引。
8. `SceneVisibilityBuilder.cpp` 构建 PVS、wedge adjacency、angular grid。
9. `SceneCache.cpp` 支持缓存 meta 和二进制场景内容。

### 2.3 查询与加速

查询门面为 `core/query/SceneQuery`，能力包括：最近面命中、全部命中、区间命中、遮挡、可见性、wedge 候选、批量 face hit、批量 occlusion、批量 Rx grid 查询。

CPU 查询使用 BVH、Moller-Trumbore 三角形求交、AABB 相交、部分 AVX2 四三角批处理、近远遍历。GPU 后端使用 `OptiXSceneAccelerator` 和 `SbrGpuPipeline`，提供 OptiX GAS、批量最近命中、批量遮挡、Rx grid 查询，以及 SBR megakernel/wavefront 支持。

### 2.4 几何路径搜索

确定性几何搜索主要位于 `core/search/SearchEngine.cpp`，核心数据结构位于 `core/path`：

1. `PathSearchContext` 聚合配置、场景、SceneQuery、材质库和 Tx/Rx。
2. `PathState` 表示搜索中间状态，记录当前点、方向、介质、剩余反射/透射/绕射预算、路径历史和签名。
3. `PathNode` 记录 Tx/Rx/反射/透射/绕射节点、交互面/楔边、介质切换、点位、入射/出射方向、法线、上一段长度。
4. `GeometricPath` 表示闭合后的几何路径。

`SearchEngine::Run` 实际采用 priority queue best-first 搜索，而非纯 DFS。每个状态先尝试闭合到 Rx，再按预算调用：

1. `ReflectionExpander`：镜像法生成反射候选。
2. `TransmissionExpander`：沿当前点到 Rx 查询透射面，解析介质切换并计算 Snell 折射方向。
3. `DiffractionExpander`：查询 wedge 候选，在边参数上用黄金分割搜索 Fermat 最短点，并检查 Keller cone。

### 2.5 SBR 搜索与覆盖

`core/search/SbrEngine.cpp` 提供另一套 SBR 路径/覆盖体系：

1. CPU legacy SBR 使用 Fibonacci sphere 生成射线，OpenMP 并行传播。
2. 使用 `RxHashGrid` 判断线段是否穿过 Rx 接收球。
3. 反射和透射按 Fresnel 功率随机二选一，不做能量分裂。
4. 绕射采用靠近三角边触发和少量方向采样的简化模型。
5. GPU megakernel 一线程一射线，适合无绕射场景。
6. GPU wavefront 按深度层批量追踪，支持简化绕射分支和批量 Rx 查询。
7. `RunCoarsePass` 为 v8 约束搜索提供活跃面集合。

### 2.6 EM 求解与聚合导出

A1 真实链位于 `app/RtRealChainRunner.cpp`：

1. `SolveSinglePathEM` 对单条几何路径构造 `EMSolverInput`，建立 Tx/Rx 天线模型。
2. `PreparePathForEM` 做路径 EM 预检查。
3. `InitializeTxField` 初始化频率、波长、功率、相位、极化和天线增益。
4. 对路径节点逐段调用 `ApplyFreeSpaceSegment`，再按节点类型调用反射、透射或绕射交互。
5. `FinalizeAtReceiver` 施加总路径 FSPL、介质损耗、Rx 增益，生成 `EMPathResult`。
6. `BuildCIR`、`BuildPDP`、`BuildAPS`、`BuildChannelStatistics`、`BuildCoverageResult`、`BuildISACFeatureSet` 聚合结果。
7. `core/result` 导出 paths、channel、coverage、ISAC、visualization、validation、regression。

## 3. 理论依据与现有实现对应

### 3.1 几何光学与镜像法反射

反射路径基于几何光学和镜像法：对 Rx 关于候选反射面做镜像，从当前点向镜像 Rx 发射射线，若命中候选面且当前点到反射点、反射点到 Rx 均可见，则生成反射状态。EM 层再使用 Fresnel TE/TM 反射系数和 Jones 极化投影更新场。

关键实现：`ReflectionExpander.cpp`、`ApplyReflectionInteraction.cpp`。

### 3.2 Snell 折射与双侧介质透射

透射路径试图基于 Snell 定律，先解析 face 的 front/back 介质切换，再用 `SnellRefract` 计算折射方向。EM 层使用复介电常数 `epsilon_c = epsilon_r - j sigma/(omega epsilon0)`、Fresnel TE/TM 透射系数、折射率和介质衰减更新场。

关键实现：`TransmissionExpander.cpp`、`ResolveMediumTransition.cpp`、`ApplyTransmissionInteraction.cpp`、`MaterialDatabase.h`。

### 3.3 UTD/Keller 绕射

确定性绕射路径基于 Fermat 最短路径和 Keller cone：在 wedge segment 上求 `current -> edge point -> Rx` 最短点，检查入射/出射相对边方向的 cone 条件。EM 层使用 UTD edge diffraction 系数、Fresnel transition function、soft/hard 极化分量和 Jones 重构。

关键实现：`DiffractionExpander.cpp`、`ApplyDiffractionInteraction.cpp`。

### 3.4 自由空间传播与信道统计

自由空间传播累计路径长度、时延、相位和介质衰减，最终按 Friis 公式施加 `lambda/(4*pi*d)` 幅度尺度。CIR/PDP/APS 等基于路径级复幅度、时延、AoA/AoD 和功率聚合。

关键实现：`ApplyFreeSpaceSegment.cpp`、`FinalizeAtReceiver.cpp`、`BuildCIR.cpp`、`BuildPDP.cpp`、`BuildAPS.cpp`。

### 3.5 加速理论

CPU 查询采用 BVH 空间层次结构、AABB 剪枝和三角形求交。GPU 查询采用 OptiX GAS 和批量 trace。v8 混合搜索用 PVS、Angular Grid、coarse SBR 活跃面集合缩小确定性搜索候选。

关键实现：`FaceBVHBuilder.cpp`、`SceneQuery.cpp`、`OptiXSceneAccelerator.cpp`、`SbrGpuPipeline.cu`、`SceneVisibilityBuilder.cpp`。

## 4. 主要漏洞与风险

### 4.1 高优先级准确性与物理性问题

1. 透射路径闭合不强制 Snell 方向经过 Rx。

`TransmissionExpander` 先沿 `current -> Rx` 找透射面，再计算折射方向；但 `TryBuildLosPath` 闭合到 Rx 时只检查 `current_point -> Rx` 可见性，不检查当前 `state.current_direction` 是否指向 Rx。这会接受违反 Snell 定律的透射路径，属于高风险物理错误。

涉及文件：`TransmissionExpander.cpp`、`SearchEngine.cpp`、`Vec3.h::SnellRefract`。

2. Snell 法线和介质方向处理不够鲁棒。

`SnellRefract` 假设传入法线方向已经满足公式要求，没有自动翻转法线，也没有 clamp `cosI`。复杂法向、背面入射、多介质出入时可能折射方向错误。

涉及文件：`core/common/math/Vec3.h`、`ResolveMediumTransition.cpp`。

3. 初始介质固定为 0。

`BuildInitialState` 将 `current_medium_id` 固定为 0，默认 Tx 在空气/默认介质。若 Tx 位于房间内部、材料内部或嵌套介质中，透射语义会错。

涉及文件：`SearchEngine.cpp`、`ResolveMediumTransition.cpp`。

4. Fresnel 反射/透射未完整使用入射侧介质。

EM 交互多以空气到材料半空间近似为主，未系统使用当前 `field.current_refractive_index` 或入射侧复介电常数。材料到材料、材料到空气、多层透射和内部反射不严格。

涉及文件：`ApplyReflectionInteraction.cpp`、`ApplyTransmissionInteraction.cpp`。

5. SBR 透射全内反射处理错误风险。

CPU SBR `RefractDir` 全内反射时返回反射方向，但调用处仍可能按 Transmission 分支记录并乘以透射功率。会产生机制标记和能量错误。

涉及文件：`SbrEngine.cpp`。

6. SBR 绕射模型与精确 EM 绕射不一致。

SBR 绕射采用边附近触发、4 个方向采样和简化标量系数；确定性 EM 使用 UTD/Jones。两套绕射物理模型差异大，结果不可直接混用。

涉及文件：`SbrEngine.cpp`、`ApplyDiffractionInteraction.cpp`。

### 4.2 漏路径与误去重问题

1. 状态签名过粗。

`BuildStateSignature` 只包含最近交互类型、face/wedge id、介质、深度和预算，不含完整历史、当前点、方向和长度。不同几何路径可能被误判重复。

涉及文件：`StateSignatureBuilder.cpp`。

2. 路径签名过粗。

`BuildPathSignature` 主要基于节点类型和 face/wedge id，不含交互点坐标、路径长度、方向、object id。同一面元序列但交互点不同的路径可能被误合并。

涉及文件：`PathSignatureBuilder.cpp`。

3. 反射和状态候选剪枝过强。

反射候选使用 Tx/Rx AABB 过滤，每个扩展器仅保留 8 个候选，每状态保留 16 个候选。复杂室内、多反射、强非最短路径容易漏解。

涉及文件：`ReflectionExpander.cpp`、`SearchEngine.cpp`。

4. 约束搜索要求候选列表有序但接口未保证。

约束版反射/透射/绕射用 `std::binary_search` 判断候选 id，但传入 vector 未保证排序。无序候选会随机漏解。

涉及文件：`ReflectionExpander.cpp`、`TransmissionExpander.cpp`、`DiffractionExpander.cpp`。

5. Wedge 候选硬编码距离和数量限制。

`SceneQuery::QueryCandidateWedges` 使用 50 m 和 64 个候选硬编码，未使用配置中的 wedge 距离和数量参数。大场景会漏绕射。

涉及文件：`SceneQuery.cpp`、`AppConfig.h`。

### 4.3 CPU/GPU 查询一致性问题

1. OptiX `QueryAllFaceHits` 和 `QueryFaceHitsInRange` 只返回最近命中，不等价 CPU all hits。透射需要多层交点时 GPU 会漏。

2. OptiX batch occlusion 使用 `tMax=1e30`，没有使用线段长度，会把目标点之后的面误判为遮挡。

3. GPU 命中结果未完整回填 `object_id`，忽略 object/filter/material 语义也不完整。

4. GPU 使用 float 几何，CPU 使用 double，边界命中和近距离相位会有一致性差异。

5. CPU fallback 的 batch Rx 查询返回空，导致无 GPU 时相关 wavefront 功能不完整。

涉及文件：`OptiXSceneAccelerator.cpp`、`SceneQuery.cpp`、`ISceneAccelerator.h`。

### 4.4 数值精度问题

1. `FinalizeAtReceiver` 对总路径长度未做零长度保护，`lambda/(4*pi*d)` 有除零风险。

2. `Complex::operator/` 若无除零保护，Fresnel 分母极端情况下可能数值爆炸。

3. `ApplyDiffractionInteraction` 的 Fresnel transition 数值积分有限截断，`x > 10` 用简单渐近 `1 + j/(2x)`，在过渡区可能误差较大。

4. `FaceBVHBuilder` 用长度容差判断面积退化，量纲不一致，应使用面积容差或 `eps_length^2`。

5. BVH SAH cost 使用 AABB 体积而非表面积，薄片场景可能退化。

6. CPU double 与 GPU float 混用，需要给出误差预算和一致性测试。

涉及文件：`FinalizeAtReceiver.cpp`、`Complex.h`、`ApplyDiffractionInteraction.cpp`、`FaceBVHBuilder.cpp`。

### 4.5 配置、解析、材质与工程鲁棒性问题

1. 配置 JSON codec 是手写字符串解析，不是严格 JSON parser，对转义、嵌套、数组、重复 key、非法类型和尾逗号鲁棒性有限。

2. 配置快照 encoder 存在尾逗号风险，且未完整输出 `rx_list`。

3. `coordinate_transform` 在配置结构中存在，但 codec 读取不完整，可能导致坐标变换配置失效。

4. `pipeline.enable_stage4_precise_em` 等字段存在但读取不完整，配置项可能无效。

5. `ConfigSelfCheck` 负例自检被跳过，配置校验退化不能被捕获。

6. `MaterialDatabase::LoadFromCsv` 返回值在主流程未检查，加载失败仍继续。

7. 材质缺失默认真空，仅 stderr warning，容易掩盖材质绑定错误。

8. OBJ 导入仅支持三角 `v//n`，不支持常见 `v/t/n`、负索引、四边形、多边形，也未实现 `rebuild_normals`。

9. `run_id` 直接拼接输出路径，路径穿越和非法字符未过滤。

涉及文件：`AppConfigJsonCodec.cpp`、`ConfigSelfCheck.cpp`、`RtPipeline.cpp`、`MaterialDatabase.h`、`OBJImporter.cpp`、`AppConfigSnapshotWriter.cpp`。

### 4.6 效率问题

1. `SceneQuery::QueryClosestFaceHit` 通过 all hits 再找最近，反射扩展等热路径仍可能调用慢路径。

2. `IsOccluded` 通过 range all hits 判断遮挡，没有早停。

3. `CpuFaceBvhAccelerator` 每次查询临时构造 `SceneQuery`。

4. Wedge 查询仍是线性扫描，距离过滤也硬编码。

5. Stage 0 PVS 和 angular grid 的物理代表性有限，且开关没有逐项严格生效。

6. `RunWavefront` 中 Rx grid 使用 static 缓存，跨运行污染且线程不安全。

涉及文件：`SceneQuery.cpp`、`CpuFaceBvhAccelerator.cpp`、`SceneVisibilityBuilder.cpp`、`SbrEngine.cpp`。

## 5. v9 修改优化方向与具体方案

### 5.1 v9-P0：修正会产生错误物理结果的问题

#### 5.1.1 透射几何一致性重构

目标：生成的透射路径必须满足 Snell 几何，不能只满足可见性。

方案：

1. 在 `TryBuildLosPath` 中增加方向一致性检查：若当前状态不是自由方向闭合，则要求 `Normalize(rx - current_point)` 与 `state.current_direction` 夹角小于可配置容差。
2. 对 `TransmissionExpander` 区分两类模式：
   - 单界面直达 Rx：反求界面交点时必须满足 Snell 残差，可用面内二维牛顿/二分或采样优化。
   - 多界面后续搜索：折射后只能沿 `txDir` 继续追踪，不允许直接用 `current_point -> Rx` 闭合绕过方向。
3. 增加 `SnellResidual` 工具函数，输出入射角、出射角、`n1*sin(theta1)-n2*sin(theta2)` 残差。
4. 将 `PathNode` 中记录的 `incident_direction`、`direction`、`medium_in_id/out_id` 作为 EM 前强校验条件。

自测试：

1. 单平面空气到玻璃，Tx/Rx 对称布置，验证交点满足解析 Snell。
2. 法线翻转同一场景，验证路径和功率不随 OBJ 法线错误翻转而产生非物理跳变，或给出明确诊断。
3. 两层平行板，验证每个界面 Snell 残差均低于阈值。
4. 构造折射后不可能到达 Rx 的场景，要求搜索不产出透射路径。

#### 5.1.2 统一介质状态与 Fresnel 入射侧

目标：反射/透射 EM 系数使用真实入射侧和出射侧介质。

方案：

1. 增加介质状态初始化：根据 Tx 点与封闭介质/face 双侧语义确定 `initial_medium_id`，暂无法可靠判断时必须在配置中显式指定。
2. `ResolveMediumTransition` 校验 `state.current_medium_id` 是否与 face 的 front/back 一侧匹配，不匹配时返回 warning 或拒绝候选。
3. 在 EM 层建立 `GetMediumMaterialById` 和 `GetComplexPermittivityForMedium`，Fresnel 统一输入 `epsilon1, mu1, epsilon2, mu2, incidentDirection, normal`。
4. 反射使用当前介质和界面另一侧材料，不再默认空气到材料。
5. 透射更新 `field.current_refractive_index` 和 `current_attenuation_np_per_m` 时允许小于 1 的有效介质，但要由 validator 或 material policy 决定是否允许。

自测试：

1. PEC 近似材料反射系数接近 1。
2. 空气到无损介质 normal incidence 与解析 Fresnel 一致。
3. 介质到空气超过临界角时必须触发 TIR，不产生透射路径。
4. 材料 A 到材料 B 与空气到材料 B 结果不同，验证不再固定空气入射。

#### 5.1.3 修复 GPU/CPU 查询语义不一致

目标：同一查询接口在 CPU 和 GPU 下语义一致，或者明确声明能力差异并避免错误调用。

方案：

1. `ISceneAccelerator` 增加 capability flags：`supports_all_hits`、`supports_object_filter`、`supports_material_filter`、`uses_double_precision`。
2. 若 GPU 不支持 all hits，`SceneQuery::QueryFaceHitsInRange` 对透射和 all hits 场景强制回退 CPU。
3. 修复 `OptiXSceneAccelerator::IsOccludedBatch`，每条线段使用对应长度 tMax。
4. 完整回填 `FaceHit.object_id`，并补齐 ignored object/filter 语义。
5. 为 GPU float 精度定义边界容差，导出 CPU/GPU 差异统计。

自测试：

1. 随机射线 CPU/GPU 最近命中一致性测试。
2. 多层平行面 all hits/range hits 测试。
3. 目标点后方放置遮挡面，batch occlusion 必须不遮挡。
4. ignored face/object 测试。

### 5.2 v9-P1：降低漏路径和误去重

#### 5.2.1 重做状态与路径签名

目标：去重不能合并物理不同路径。

方案：

1. 状态签名加入完整交互序列 hash、量化当前点、量化当前方向、当前介质、累计长度 bucket。
2. 路径签名加入每个节点的量化点坐标、路径总长 bucket、入射侧/出射侧、object id。
3. 量化尺度来自 `NumericToleranceConfig`，例如坐标用 `eps_intersection * 10` 或配置项。
4. 导出 duplicate diagnostics，统计被合并路径数量和最大几何差异。

自测试：

1. 同一大平面不同反射点路径不能误合并。
2. 同一 face 序列但不同介质侧路径不能误合并。
3. 完全重复路径仍能去重。

#### 5.2.2 约束搜索候选集合安全化

目标：约束搜索不因 vector 无序而漏解。

方案：

1. 在 `ConstrainedSearchConfig` 构造或 `SearchEngine::Run` 入口排序并去重 `candidate_faces/wedges`。
2. 或将 expander 内 `binary_search` 改为 `unordered_set` membership。
3. 反射约束版在 BVH candidates 为空但 candidateFaces 非空时，回退遍历 candidateFaces。

自测试：

1. 候选 face 输入升序、降序、随机顺序，结果完全一致。
2. 禁用/缺失 BVH 时约束反射仍可从 candidateFaces 产出路径。

#### 5.2.3 剪枝策略配置化与可诊断

目标：启发式剪枝可控、可复现、可解释。

方案：

1. 将 `perExpanderKeepLimit`、`perStateKeepLimit`、wedge max distance、wedge max candidates、reflection AABB margin 全部配置化。
2. 增加 search diagnostics：候选总数、被 AABB 剪掉数、被角度剪掉数、被可见性剪掉数、被 top-K 剪掉数。
3. 增加 `exhaustive_debug_mode`，小场景可关闭 top-K 剪枝用于对照。

自测试：

1. 小型盒状房间 exhaustive 与剪枝搜索对比，确认剪枝只丢低贡献路径。
2. 增大 keepLimit 时路径数单调不下降。
3. 大场景 wedge 距离配置改变时候选数符合预期。

### 5.3 v9-P1：提升 EM 数值稳定与物理一致性

#### 5.3.1 复场链路统一

目标：减少幅相拆分导致的相位和极化误差。

方案：

1. 将 `FieldAccumulator` 的标量场改为主用 complex amplitude，`phase_rad` 作为派生或兼容输出。
2. 反射、透射、绕射均直接乘复系数并更新 Jones 复向量。
3. `FinalizeAtReceiver` 只在最终导出时计算幅度、相位、功率。
4. 对多个分量相位不能用功率加权 atan 简化，应保留复叠加。

自测试：

1. 两条等幅相反相路径 CIR 合成应接近抵消。
2. 单反射相位与解析 Fresnel 相位一致。
3. 连续两次反射的复系数乘积与逐步结果一致。

#### 5.3.2 UTD 绕射数值改进

目标：绕射过渡区稳定，soft/hard 极化和 wedge 角定义明确。

方案：

1. 引入可验证的 Fresnel transition 近似或自适应积分误差控制。
2. 明确 wedge 内外角、凸/凹边、face normal 方向和 UTD `n` 定义。
3. Boundary edge 是否生成绕射通过配置控制。
4. SBR 绕射与确定性 UTD 共享一套系数函数，至少系数定义一致。

自测试：

1. 半平面 knife-edge 绕射与标准解析/数值参考对照。
2. wedge 角扫描，验证系数连续且无异常尖峰。
3. shadow boundary 附近路径功率连续。

## 7. 自测试闭环方法

### 7.1 测试分层

1. 单元测试：数学函数、Fresnel、Snell、UTD transition、JSON codec、material interpolation、OBJ parser。
2. 几何测试：BVH vs brute force、visibility、reflection image method、transmission Snell residual、diffraction Fermat/Keller。
3. EM 测试：LOS Friis、单反射 Fresnel、单透射 Fresnel、knife-edge diffraction、双路径相干叠加。
4. 集成测试：小房间、多墙体、多 Rx grid、CPU/GPU backend diff、SBR vs deterministic coarse consistency。
5. 回归测试：固定配置输出 paths/channel/coverage/validation JSON，比较关键指标。

### 7.2 建议基线场景

1. `LOS_FreeSpace_1m_10m`：验证 Friis、时延、相位。
2. `SinglePECPlaneReflection`：验证镜像法和近似全反射。
3. `SingleDielectricPlaneTransmission`：验证 Snell 和 Fresnel。
4. `ParallelSlabTwoInterfaces`：验证多界面透射和介质衰减。
5. `KnifeEdgeDiffraction`：验证绕射过渡区。
6. `TwoRayGroundReflection`：验证相干叠加和深衰落。
7. `BoxRoomMultiReflection`：验证多反射路径数和去重。
8. `LargeWedgeCandidateScene`：验证 wedge 查询配置和性能。
9. `CpuGpuConsistencyScene`：验证 CPU/GPU 最近命中、遮挡、range hits。
10. `InvalidConfigAndMaterialCases`：验证负例自检。

### 7.3 关键验收指标

1. LOS Friis 功率误差小于 0.1 dB。
2. LOS 时延误差小于 1 ps 或由几何长度容差换算。
3. 单界面 Snell 残差小于 `1e-6` 或配置容差。
4. Fresnel normal incidence 幅度误差小于 `1e-6`。
5. CPU/GPU closest hit 距离差小于 `max(1e-5 m, eps_intersection*10)`。
6. CPU/GPU occlusion 分类一致率为 100%，边界样本单独统计。
7. 路径签名去重不得合并坐标差大于容差的路径。
8. 配置快照 round-trip 字段一致率 100%。
9. 无材质缺失 strict 模式通过率 100%，缺材质负例必须失败。
10. 关键集成场景输出稳定，重复运行结果 diff 在阈值内。

## 11. 对照开题报告的硕士论文算法缺口

### 11.1 开题报告对算法的真实要求

《西安电子科技大学硕士学位论文开题报告表 v1.23》的题目是“面向通感一体的多维信道建模与优化研究”。其算法要求不是单纯实现一个可运行 RT 程序，而是要形成可支撑硕士论文的“理论模型、算法实现、验证体系、通感应用分析”闭环。

从报告正文可抽取出以下核心目标：

1. 面向 6G ISAC/通感一体场景，构建高分辨率室内多径信道建模框架。
2. 以改进几何光学和一致性绕射理论为核心，生成包含时延、三维 AoA/AoD、路径损耗、复振幅、极化状态的全维度多径参数。
3. 支持复杂室内场景和关键散射体精细建模，包括办公室、L 形走廊、商业房间、家具、设备、复合材质表面等。
4. 建立天线“时延-极化-旋转”联合建模方法，把天线方向图、频率响应、Jones/PSM 极化、空间姿态旋转逐路径注入 RT 链路。
5. 建立多维验证体系，覆盖时延域、空间域、极化域、统计域，以及通信与感知双维度指标。
6. 与实测、传统 RT、统计模型或标准模型进行对照，解释误差来源并形成校准机制。
7. 面向典型室内场景分析通信性能和感知性能之间的耦合关系，为资源分配、波束设计、系统优化提供信道数据基础。

因此，当前 RT 工程距离论文要求的差距应从“物理模型是否可信、理论映射是否完整、功能输出是否覆盖论文指标、验证是否可支撑答辩”四个层面评估。

### 11.2 总体差距判断

当前工程已经具备论文软件原型基础：有 OBJ 场景导入、反射/透射/绕射路径搜索、SBR 覆盖、天线增益注入、路径级 EM 求解、CIR/PDP/APS/ISAC 输出和初步 GPU 加速。

但按硕士毕业论文算法要求判断，当前仍不能直接宣称满足“高分辨率通感一体多维信道建模”。主要原因是：

1. 物理层面仍存在会导致错误路径或错误功率的关键漏洞，尤其是透射 Snell 闭合、介质侧一致性、SBR TIR、CPU/GPU 查询语义等。
2. 理论层面尚未完整建立“天线时延-极化-旋转”联合模型，也没有把 Jones/PSM、频率色散、天线姿态和逐路径传播强耦合起来。
3. 功能层面缺少 STL/点云/精细散射体建模、实测数据导入、3GPP 标准对照、感知指标计算、通信-感知性能权衡分析等论文关键输出。
4. 验证层面缺少可复现的多维验证矩阵，现有验证更像工程 sanity check，不能支撑“模型准确性”结论。
5. 论文贡献层面需要明确算法创新点，目前工程更偏系统集成，需要把“改进 RT 搜索、天线联合建模、多维验证、通感指标映射”凝练为可论述、可实验对比的贡献。

### 11.3 物理模型缺失

#### 11.3.1 透射物理不满足严格几何与介质一致性

开题要求室内墙体、玻璃、家具、复合材质等介质透射能够准确影响多径时延、功率、相位和极化。当前实现存在：

1. 透射点由 `current -> Rx` 直线找面，再计算 Snell 折射方向，后续闭合未强制 Rx 位于折射方向上。
2. 初始介质固定为 0，无法可靠表达 Tx/Rx 位于房间内部、墙体内部或嵌套介质环境。
3. Fresnel 反射/透射未系统使用入射侧和出射侧复介电常数。
4. 材质缺失时可静默退化为真空，导致透射路径看似存在但物理参数错误。
5. 多层墙体、复合材料、有限厚度板材的多界面传播没有完整模型。

论文影响：若不修复，透射损耗 vs 频率、墙体穿透路径、NLOS 覆盖和 PDP 长尾都会缺乏可信度。

#### 11.3.2 绕射模型仍需从“可用实现”提升到“可证明可信”

当前确定性绕射使用 Fermat + Keller + UTD，形式上接近论文要求；但仍存在：

1. wedge 内外角、凸凹边、法向方向与 UTD 参数 `n` 的定义需要严格化。
2. Fresnel transition function 采用有限积分和简单渐近，缺少误差控制。
3. SBR 绕射与确定性 UTD 不是同一物理模型，覆盖结果和路径级 EM 结果不一致。
4. boundary edge、非流形边、复杂家具薄边是否参与绕射没有论文级建模规则。
5. 多次绕射、高阶绕射和 R/T/D 混合后绕射的物理适用范围未定义。

论文影响：绕射是室内 NLOS 与感知遮挡边界的重要机制。若仅展示“有绕射代码”，不足以支撑高保真结论。

#### 11.3.3 漫散射和粗糙表面缺失

开题报告多次强调复杂室内散射体、家具、粗糙表面和感知相关细节。当前工程主要覆盖镜面反射、透射、楔边绕射，缺少可用的漫散射模型。

缺失包括：

1. Lambertian 或 directive scattering 模型。
2. 粗糙度参数、相关长度、散射系数与材质数据库绑定。
3. 漫散射路径采样、能量归一化、相位随机化或相干/非相干处理策略。
4. 对家具、设备、人体等非理想平滑表面的散射贡献建模。
5. 漫散射对 APS、PDP 和感知目标回波的影响分析。

论文影响：如果论文声称复杂室内高分辨率或感知细节，完全缺少漫散射会被认为物理机制不完整。若不实现，需要在论文范围中明确限定为镜面/楔边主导场景。

#### 11.3.4 天线物理还停留在方向图增益注入

当前天线模型已有 ideal/pattern、Tx/Rx 增益注入和基础极化字段，但距离开题的“时延-极化-旋转联合建模”有明显差距。

缺失包括：

1. 天线频率响应和群时延模型。
2. 天线端口阻抗、效率、相位中心随频率变化。
3. 复方向图，即每个方向的幅度和相位，而不仅是 dBi 增益。
4. Jones 矢量方向图，区分 co-pol/cross-pol。
5. Polarization Scattering Matrix 或天线极化散射矩阵接口。
6. 欧拉角/四元数姿态旋转对方向图、极化基和 AoA/AoD 的影响。
7. 手机手持、倾斜安装、阵列姿态变化等实际终端安装状态。

论文影响：这是开题报告中最明确的算法创新点之一。若不实现，论文题目中的“天线多维特性/时延-极化-旋转”将缺少核心支撑。

#### 11.3.5 近场、阵列和波束物理缺失

ISAC 场景常涉及阵列天线、波束扫描、目标角度估计和距离分辨率。当前工程多以点 Tx/Rx 和单天线模型为核心。

缺失包括：

1. 阵列几何、阵元方向图、阵元相位中心。
2. 波束赋形权重、扫描角、旁瓣、波束宽度。
3. 近场球面波模型和远近场判定。
4. MIMO 信道矩阵 `H(f,t)` 或 `H(delay, angle)` 输出。
5. 阵列观测下的角度分辨率和角度估计误差模型。

论文影响：如果后续章节要讨论通信波束设计、感知角度估计或资源分配，仅有单链路路径结果不够。

#### 11.3.6 频率选择性和宽带信道不足

开题报告强调高分辨率时延和频率响应。当前 EM 主要按单频计算，再导出 CIR/PDP。

缺失包括：

1. 多频点扫频和宽带 transfer function `H(f)`。
2. 材料参数随频率的系统化建模和验证。
3. 天线频率响应对每条路径幅相和群时延的影响。
4. OFDM 子载波级信道输出。
5. 时延分辨率、带宽和路径可分辨性之间的关系建模。

论文影响：若论文要讨论高分辨率时延域、PDP 精度或 ISAC 距离分辨率，需要宽带模型支撑。

### 11.4 理论模型缺失

#### 11.4.1 “可观测基带多径”映射不完整

开题报告要求输出接近真实前端可观测的信道数据。当前路径结果包含 delay、phase、amplitude、power、AoA/AoD、polarization，但还没有形成严格的基带观测模型。

缺失包括：

1. 从路径级电场到复基带 CIR 的统一公式定义。
2. Tx/Rx RF chain、天线方向图、极化投影、阵列响应和系统带宽的联合映射。
3. 采样率、脉冲成形、窗函数对 CIR/PDP 的影响。
4. 多路径相干叠加和非相干统计的边界条件。
5. 输出结果与真实矢量网络分析仪或信道探测仪数据格式的对应关系。

论文影响：没有该映射，仿真结果只能称为几何/电磁路径参数，难以称为“真实前端可观测信道”。

#### 11.4.2 通感双功能指标映射不足

当前已有 `BuildISACFeatureSet`，但更像初步特征导出，尚未形成开题要求的通感性能关系分析。

缺失包括：

1. 通信指标：路径损耗、RMS delay spread、角度扩展、K 因子、容量、频谱效率、BER/SINR。
2. 感知指标：距离分辨率、角度估计精度、目标检测概率、虚警概率、感知 SINR、CRLB 或 Fisher 信息。
3. 同一信道数据下通信指标和感知指标的联合计算。
4. 不同天线姿态、波束、频率、带宽、场景材质对双指标的敏感性分析。
5. 资源分配或波束优化的输入输出接口。

论文影响：如果没有这些指标，最后一章“通信与感知性能相关性/资源优化”只能停留在概念层面。

#### 11.4.3 3GPP/标准模型对照不足

开题报告明确引用并扩展 3GPP TR 38.901/ISAC 相关标准。当前工程没有内建 3GPP InH-Office 路损、时延扩展、角度扩展或 cluster 统计对照。

缺失包括：

1. 3GPP TR 38.901 InH LOS/NLOS path loss 公式。
2. DS/AS/K 因子等大尺度参数统计对照。
3. 频段 0.5-100 GHz 的适用边界说明。
4. RT 结果向 3GPP cluster/参数的映射。
5. 与 Sionna RT 或参考实现的同场景对比。

论文影响：缺少标准模型对照会削弱论文“符合行业标准/扩展标准模型”的说服力。

#### 11.4.4 误差来源和校准理论缺失

开题报告要求通过实测或高可信对照校准模型。当前工程有 validation/report 结构，但缺少误差分解模型。

缺失包括：

1. 几何误差、材质误差、天线误差、数值误差、采样误差的分项定义。
2. 面向路径损耗、PDP、APS、极化的误差指标。
3. 材质参数反演或校准方法。
4. 天线方向图/姿态误差敏感性分析。
5. SBR 射线数、接收球半径、路径阶次对收敛性的误差模型。

论文影响：没有误差理论，仿真与参考/实测不一致时无法解释，也难以提出“优化研究”的实质内容。

### 11.5 功能缺失

#### 11.5.1 场景输入和精细散射体库不足

开题要求支持 `.Obj`、`.Stl` 等场景格式，建立典型室内场景库和关键散射体参数库。当前仅 OBJ 支持有限，STL 未见主链支持。

缺失包括：

1. STL 导入。
2. 点云/扫描模型预处理接口。
3. 家具、设备、人体、复合材料散射体库。
4. 表面粗糙度、厚度、层状材料参数。
5. 场景尺度、坐标系、单位、法向一致性的自动诊断。

#### 11.5.2 输出指标不够覆盖论文图表

当前能导出路径、channel、coverage、ISAC、visualization，但论文需要的数据产品更多。

缺失包括：

1. 路径损耗 vs 距离曲线。
2. RMS delay spread CDF/分布。
3. APS 相似度和角度扩展统计。
4. 极化判别度、XPR、co/cross polarization power ratio。
5. 不同天线姿态下的 PDP/APS/容量变化。
6. 不同频率、带宽、材质参数下的敏感性曲线。
7. SBR 收敛性曲线：ray count vs power/path loss/error。
8. 计算效率曲线：CPU/GPU、BVH/OptiX、剪枝前后。
9. 通信容量/频谱效率与感知精度之间的 trade-off 图。

#### 11.5.3 实测/参考数据闭环缺失

开题报告规划了高精度矢量网络分析仪、可控机械扫描平台和典型室内场景测量。当前工程没有实测数据导入和比对链路。

缺失包括：

1. 实测 CIR/PDP/APS 数据格式定义和导入。
2. VNA 或 channel sounder 数据预处理。
3. 实测与仿真坐标配准。
4. 路径级匹配算法：delay/angle/power matching。
5. 可视化误差热力图和统计报告。

如果短期无法实测，至少需要三层替代闭环：解析公式、Sionna/参考 RT、3GPP 统计模型。

#### 11.5.4 动态与多普勒能力未实现

开题正文提到移动人员、动态姿态、实际终端使用场景等。当前工程是静态信道。

缺失包括：

1. Tx/Rx/目标/散射体轨迹。
2. 多普勒频移。
3. 时间序列 CIR。
4. 相干时间和时变 PDP/APS。
5. 移动人体或动态遮挡。

处理建议：若毕业论文时间有限，可将完整多普勒/动态信道明确降级为未来工作，但天线姿态旋转和静态多姿态扫描仍应实现，否则与开题目标偏差较大。

### 11.7 为达到硕士论文要求的最低可行补齐范围

若目标是在毕业论文中可信地支撑“面向通感一体的多维信道建模与优化”，建议至少完成以下最低闭环。

P0 论文可信底线：

1. 修复透射 Snell 几何闭合、介质侧一致性、TIR 和材质 strict policy。
2. 修复 CPU/GPU 查询一致性中会改变路径存在性的错误。
3. 完成复场/Jones 极化链路，输出复极化状态和 co/cross polarization 指标。
4. 建立 LOS、单反射、单透射、knife-edge、two-ray 的解析验证。
5. 增加 3GPP InH path loss/DS/AS/K 因子基础对照。

P1 论文核心贡献：

1. 实现天线姿态旋转：forward/up 或 yaw/pitch/roll 影响方向图查询、极化基、AoA/AoD。
2. 实现复方向图或 Jones pattern 导入，至少支持 `theta, phi, Etheta_re, Etheta_im, Ephi_re, Ephi_im` CSV。
3. 实现天线频率响应/群时延的简化模型，支持多频点 sweep。
4. 输出宽带 CIR/PDP/APS 和极化统计，形成论文图表。
5. 建立 Sionna RT 或参考实现同场景对照。

P2 论文增强项：

1. 实现简化漫散射模型，重点覆盖家具/粗糙墙面。
2. 实现 STL 导入和典型室内散射体库。
3. 增加通信容量、频谱效率、感知距离/角度精度或 CRLB 指标。
4. 做通信-感知 trade-off 实验：频率、带宽、波束、天线姿态、场景材质变化。
5. 接入实测数据或构造可复现实测替代数据集。

### 11.8 建议论文实验闭环

论文中建议至少形成四组实验，而不是只展示程序输出。

实验 1：解析物理正确性验证。

1. LOS Friis。
2. 单平面反射 Fresnel TE/TM。
3. 单界面透射 Snell + Fresnel。
4. Knife-edge/wedge 绕射。
5. Two-ray 相干叠加。

实验 2：典型室内场景多维信道输出。

1. 办公室、L 形走廊、会议室或商业房间。
2. 输出路径三维可视化、PDP、APS、XPR、RMS DS、AS、K 因子。
3. 展示反射、透射、绕射路径占比和功率贡献。

实验 3：天线联合建模影响。

1. 对比 ideal antenna、标量方向图、Jones 方向图。
2. 改变 yaw/pitch/roll，观察 PDP/APS/容量/感知角度误差变化。
3. 多频点扫频，观察群时延和频率选择性。

实验 4：通感性能与对照验证。

1. 与 3GPP InH path loss 和统计参数对照。
2. 与 Sionna RT 或参考实现对照路径损耗/PDP/APS。
3. 输出通信容量/SINR 与感知距离或角度指标的 trade-off。
4. 若有实测数据，增加实测 CIR/PDP/APS 对照；若没有，明确采用三级替代验证。

### 11.9 与开题报告表述的风险边界

后续论文写作时需要避免超过当前实现能力的表述。建议边界如下：

1. 若未实现动态轨迹和多普勒，不应宣称支持完整时变信道，只能说支持静态多姿态/多位置信道快照。
2. 若未实现漫散射，不应宣称完整复杂室内散射，只能说覆盖镜面反射、介质透射和楔边绕射主导机制。
3. 若未接入实测，不应宣称实测验证完成，只能说通过解析公式、标准模型和第三方/参考实现进行交叉验证。
4. 若天线只实现标量方向图，不应宣称完成“时延-极化-旋转联合建模”。该项至少需要复方向图、Jones 极化和姿态旋转闭环。
5. 若 SBR 使用随机二选一反射/透射，不应将单次 SBR 覆盖结果作为严格相干电磁解，只能作为大规模覆盖近似并给出收敛性分析。

### 11.10 后续讨论建议

浏览本文档后，建议优先讨论以下决策点：

1. 论文范围是否保留“天线时延-极化-旋转联合建模”为核心创新。如果保留，需要将其提升为 v9/v10 主线。
2. 是否实现漫散射。如果不实现，需要明确论文场景和物理机制边界。
3. 实测数据是否可获得。如果不可获得，需要确定 Sionna/参考实现/3GPP 三层替代验证方案。
4. 通感性能指标选哪些。建议最小集合为通信侧 `path loss/RMS DS/AS/K factor/capacity`，感知侧 `range resolution/angle spread or angle estimation proxy/sensing SINR`。
5. 毕业论文更偏“算法物理正确性”还是“系统效率优化”。这会决定优先做精确 EM/天线模型，还是优先做 GPU/SBR/大规模覆盖。

## 12. v9 后续开发主线讨论提纲

本章用于后续逐轮讨论和同步。每完成一次主线讨论，应在对应小节下补充最终决策、开发范围、实现方案、影响评估和自测试闭环。

### 12.1 v9 总体定位需要先讨论

核心问题：v9 是“论文可信度修复版”，还是“工程效率优化版”，还是“两者兼顾但分阶段交付”。

建议初始定位：v9 优先成为论文可信度修复版，即先保证物理正确、理论闭环、验证可信，再选择性做性能优化。原因是当前最影响硕士论文的不是速度，而是透射、介质、天线、验证和通感指标能否支撑高保真结论。

需要讨论的具体点：

1. v9 的最小可交付目标是什么。
2. 哪些内容必须进入 v9，哪些可以放到 v10 或论文未来工作。
3. v9 是否以确定性 SearchEngine 为主，SBR/GPU 为辅。
4. v9 是否要保留“天线时延-极化-旋转联合建模”为核心创新。
5. v9 是否要实现漫散射，还是先声明镜面反射/透射/楔边绕射为论文边界。
6. v9 的验收标准是解析验证通过、参考实现对比通过、还是能生成论文图表。

系统级影响：

1. 若聚焦物理正确性，短期速度可能下降，但路径、功率、相位、极化更可信。
2. 若聚焦效率，覆盖仿真规模会提升，但论文中高保真结论仍可能被质疑。
3. 若聚焦天线联合建模，论文创新点更清晰，但开发复杂度明显增加。

参考依据：

1. Wireless InSite、WinProp、Remcom 等商业 RT 软件通常先确保 GO/UTD、材质、天线方向图和验证闭环，再做大规模加速。
2. Sionna RT 提供可对照的路径级 RT、材质、天线阵列和可微仿真思路。
3. 3GPP TR 38.901 可作为统计基线，不替代 RT 物理模型，但适合论文对照。

### 12.2 主线 A：precise 几何路径状态机、双侧介质一致性与表面语义定版

本节为 v9 主线 A 的最终定版方案。主线 A 不再只是“修透射”，而是建立 precise 模式下几何寻径和电场计算之间的清晰契约：几何模块返回尽可能全量、物理成立、介质语义完整的几何路径；电场模块只消费这些路径节点语义，计算幅度、相位、极化和损耗，不反向决定路径是否存在。

#### 12.2.1 定版原则

1. precise 几何寻径模块和电场计算模块必须尽可能解耦。
2. precise 几何寻径模块负责按配置弹射数寻找尽可能多的准确物理几何路径。
3. 电场计算模块负责消费几何路径，计算 Fresnel 系数、传播相位、介质衰减、天线响应和极化演化。
4. 几何模块可以使用折射率、Snell 定律、Keller cone 等几何光学约束判断路径是否成立，但不计算最终路径功率。
5. 电场模块不重新猜测路径的介质切换，不用默认真空掩盖路径语义缺失。
6. precise 模式下的所有剪枝、拒绝、残差和介质切换都必须可配置、可诊断、可导出。

#### 12.2.2 场景建模约定

v9 主线 A 采用以下场景建模约定：所有可观测到的表面都显式建模。有厚度的物体不能用一个零厚度面元代表，而应建为两个或多个真实外表面。

以一片极薄木板为例，仍使用两个面元表示：

1. 第一个外表面法向指向一侧空气，`front=air`，`back=wood`，`surface=wood`。
2. 第二个外表面法向指向另一侧空气，`front=air`，`back=wood`，`surface=wood`。
3. 射线从空气进入木板第一面后，当前介质变为 `wood`。
4. 射线在木板内部传播一段距离后，从第二面 `back` 侧入射，透射到 `front=air`。

该约定消除了“一个面元表达整块有厚度物体”的主要歧义，也使 front/back 双侧介质可以自然表达传播介质状态。

#### 12.2.3 面元三类材质语义

现有 `Face` 已包含三类材质字段，v9 定版语义如下：

1. `front_material_name/front_medium_id`：法向正侧传播介质，是介质状态机的权威来源之一。
2. `back_material_name/back_medium_id`：法向负侧传播介质，是介质状态机的权威来源之一。
3. `surface_material_name/surface_material_id`：该可观测表面的实体材料或表面属性标签，通常等于固体侧材料，例如 `wood`、`concrete`、`glass`、`metal`。

v9 默认规则：

1. 透射和介质状态切换只由 `front/back` 决定。
2. Snell 折射只由入射侧介质和出射侧介质决定。
3. 反射后的传播介质保持为入射侧介质。
4. 反射 Fresnel 默认由入射侧介质与面元另一侧介质共同决定。
5. `surface_material_name` 不覆盖 `front/back` 的物理介质语义，主要作为表面实体材料、粗糙度、散射、涂层或后续经验修正入口。

这样处理后，外侧空气打到木板表面时，反射界面自然是 `air -> wood`；如果射线已经在木板内部并打到出射面，反射界面自然是 `wood -> air`。这避免了“反射永远默认空气打 surface”的错误。

#### 12.2.4 几何路径状态机

每个 `PathState` 必须表达当前几何状态：

1. `current_point`：当前射线所在点。
2. `current_direction`：当前出射方向。
3. `current_medium_id`：当前点之后这一段传播所在介质。
4. 剩余反射、透射、绕射和总扩展预算。
5. 已遍历节点列表和当前状态签名。

每次射线与面元交互时，按以下状态机执行：

反射：

1. 根据 `Dot(direction, face.normal)` 判断入射侧。
2. 校验当前 `current_medium_id` 与入射侧 medium 一致。
3. 按镜面反射公式计算出射方向。
4. 反射后 `current_medium_id` 不变。
5. 记录 face、入射侧、介质、入射方向、出射方向和反射几何残差。

透射：

1. 根据 `Dot(direction, face.normal)` 判断从 front 侧还是 back 侧入射。
2. 入射侧 medium 必须等于 `current_medium_id`。
3. 出射侧 medium 为 face 另一侧 medium。
4. 根据 `n_in/n_out` 和 Snell 定律计算折射方向。
5. 若发生全内反射，不能生成 Transmission 状态；可由反射扩展处理反射路径。
6. 透射后 `current_medium_id = medium_out_id`。
7. 记录 Snell residual、入射角、出射角、TIR 标志。

绕射：

1. 绕射默认不改变介质，`current_medium_id` 保持不变。
2. 绕射点满足 Fermat 路径极值和 Keller cone 条件。
3. 记录 wedge、绕射点、入射方向、出射方向、Keller residual。
4. 若后续扩展支持跨介质边缘绕射，再单独增加介质切换规则。

LOS/Rx 闭合：

1. 当前点到 Rx 必须可见。
2. 若当前状态已有明确 `current_direction`，则 `current_direction` 必须与 `current_point -> Rx` 方向一致，夹角小于配置容差。
3. 对透射后的闭合尤其必须执行方向一致性检查，避免折射方向未指向 Rx 仍被接受。

#### 12.2.5 PathNode 语义契约

几何模块输出的 `PathNode` 必须成为 EM 模块的唯一语义来源。每个交互节点至少应明确：

1. `interaction_type`：Reflection / Transmission / Diffraction / Tx / Rx。
2. `face_id` 或 `wedge_id`。
3. `point`：交互点。
4. `incident_direction`：进入该节点的方向。
5. `direction`：离开该节点后的方向。
6. `surface_normal`：命中面法向。
7. `medium_in_id`：节点前一段介质。
8. `medium_out_id`：节点后一段介质。
9. `entered_from_front_side`：是否从 face front 侧入射。
10. `segment_length_from_previous`：上一节点到当前节点的几何长度。
11. 几何残差诊断：reflection residual、Snell residual、Keller residual。

介质连续性要求：前一个交互节点的 `medium_out_id` 必须等于下一个交互节点的 `medium_in_id`。若不满足，路径不应进入 EM precise 求解。

#### 12.2.6 具体修改优化方案

任务 A-1：补全几何诊断结构。

1. 新增或扩展 `PathNode` 中的几何诊断字段。
2. 建议引入 `InteractionGeometryDiagnostics`，记录 Snell、Keller、反射残差、入射角、出射角、TIR 标志。
3. 导出路径时同步输出这些诊断字段。

任务 A-2：严格化 `ResolveMediumTransition`。

1. 当前逻辑只根据入射侧决定 `medium_out_id`，但未校验 `state.current_medium_id` 是否等于入射侧 medium。
2. v9 应改为：`expected_in = entered_from_front ? face.front_medium_id : face.back_medium_id`。
3. `expected_out = entered_from_front ? face.back_medium_id : face.front_medium_id`。
4. 若 `state.current_medium_id != expected_in`，precise strict 模式拒绝该候选，并记录 medium mismatch。

任务 A-3：升级 `SnellRefract`。

1. 不再只返回 `Vec3`，改为结构化 `SnellResult`。
2. 输出 `valid`、`total_internal_reflection`、`direction`、`cos_i`、`cos_t`、`theta_i_rad`、`theta_t_rad`、`residual`。
3. 内部自动处理法线朝向、`cos_i` clamp、TIR 判断。
4. residual 使用 `abs(n1*sin(theta_i) - n2*sin(theta_t))`。

任务 A-4：修复透射闭合逻辑。

1. 透射后 `nextState.current_direction = snell.direction`。
2. 若下一步尝试直接闭合 Rx，必须检查 `Angle(nextState.current_direction, Normalize(rx - hit.position)) < direction_closure_tol`。
3. 不满足闭合方向的透射候选不能直接生成 Rx 路径。
4. 后续若要进一步精确，应将“每步朝 Rx 找透射面”的简化策略升级为沿 `current_direction` 传播的状态机搜索。

任务 A-5：统一 EM 对节点介质语义的消费。

1. `ApplyTransmissionInteraction` 根据 `node.medium_in_id` 和 `node.medium_out_id` 查询材料，不再默认真空。
2. `ApplyReflectionInteraction` 使用 `node.medium_in_id` 和面元另一侧 medium 计算反射 Fresnel。
3. 若材料或介质语义缺失，precise EM 失败并输出诊断，不静默降级。
4. EM 层只消费 `GeometricPath` 节点，不重新推导介质切换方向。

任务 A-6：材质 strict policy。

1. 主流程必须检查 `MaterialDatabase::LoadFromCsv` 返回值。
2. 增加 `material.missing_material_policy`，默认建议为 `strict`。
3. 可选值：`strict`、`warn_default_vacuum`、`allow_missing_for_debug`。
4. 论文实验必须使用 strict，debug 才允许真空 fallback。

任务 A-7：支持真实双表面薄板。

1. `self_hit_ignore_distance` 必须小于最薄板厚度，或为透射后下一面命中使用特殊容差策略。
2. 透射后 origin offset 不能越过极薄板另一侧面元。
3. 增加薄板场景诊断，检查双面法向是否分别指向外侧介质。

任务 A-8：precise 模式全量性与剪枝诊断。

1. `perExpanderKeepLimit`、`perStateKeepLimit`、wedge max distance、wedge max candidates 全部配置化。
2. `precise.exhaustive_mode=true` 时禁用 top-K 硬剪枝。
3. 状态签名和路径签名加入量化交互点、方向、介质和历史序列，降低误去重。
4. 输出 search diagnostics：候选数、拒绝原因、剪枝原因、残差分布。

#### 12.2.7 对系统级仿真的影响

1. 错误透射路径会被剔除，路径数量可能下降，但物理可信度提升。
2. 穿墙、薄板、玻璃、木板、混凝土等场景的 NLOS 功率、PDP 长尾和相位会更合理。
3. 多次透射后，路径介质状态连续，后续自由传播相位和介质衰减更可信。
4. 反射不再默认“空气打 surface”，材料内部反射和出射面反射可正确表达。
5. precise 模式速度可能下降，因为硬剪枝减少，但结果更适合论文验证。
6. fast/SBR 模式可继续保留近似，但必须与 precise 模式在文档和报告中区分。

#### 12.2.8 闭环自测方案

单元测试：

1. `Snell_AirToGlass_30deg`：空气到玻璃，入射角 30 度，出射角与解析值一致。
2. `Snell_GlassToAir_TIR`：玻璃到空气超过临界角，返回 TIR，不产生透射路径。
3. `Snell_NormalFlipInvariant`：法线反向输入时，结果一致或给出明确 side 诊断。
4. `Fresnel_NormalIncidence`：normal incidence 下 TE/TM 系数与解析值一致。
5. `Fresnel_BrewsterAngle`：无损介质 TM 反射在 Brewster angle 附近接近 0。

几何集成测试：

1. `SingleInterfaceTransmission`：单空气-玻璃界面，路径 Snell residual 小于阈值。
2. `ThinSlabTransmission`：双表面薄木板，路径介质序列应为 `air -> wood -> air`。
3. `ThinSlabSelfHitTolerance`：薄板厚度接近容差时，检查不会错误忽略第二面。
4. `MediumMismatchReject`：故意设置当前介质与入射侧不一致，precise strict 必须拒绝。
5. `TransmissionClosureDirectionReject`：折射方向不指向 Rx 时，不得生成直达 Rx 透射路径。

EM 集成测试：

1. `Reflection_AirWood_OuterFace`：空气侧打木板外表面，反射使用 `air -> wood` 界面。
2. `Reflection_WoodAir_InnerSide`：木板内部打出射面，反射使用 `wood -> air` 界面。
3. `Transmission_AirWoodAir_SlabLoss`：双界面木板透射功率、相位、时延随板厚变化合理。
4. `MissingMaterialStrictFail`：缺材质时 precise EM 必须失败并报告。
5. `PathMediumContinuity`：所有进入 EM 的路径满足节点介质连续性。

回归与报告：

1. `geometry_validation.json` 输出每条路径的 medium sequence、Snell residual、Keller residual、direction closure residual。
2. `em_validation.json` 输出每个交互的 Fresnel 输入介质、TE/TM 系数、介质衰减。
3. `search_diagnostics.json` 输出候选数、拒绝原因、剪枝原因。
4. 旧场景路径数变化必须有诊断解释。

#### 12.2.9 理论依据与成熟参考

理论依据：

1. 几何光学 GO：反射满足镜面反射定律，透射满足 Snell 定律。
2. 分片均匀介质模型：场景由介质边界面分割，射线状态携带当前传播介质。
3. Fresnel 方程：TE/TM 反射和透射系数由入射侧和出射侧复介电常数共同决定。
4. UTD/Keller：绕射路径需满足 Fermat 极值和 Keller cone 条件。
5. 复传播常数：介质中的相位和衰减由复介电常数和频率决定。

成熟软件与开源参考：

1. Wireless InSite：典型商业 RT 软件，采用场景几何、材料库、GO/UTD 相互作用和路径级输出分层思想。
2. WinProp/ProMan：支持室内外射线追踪、材料参数、反射/透射/绕射和覆盖仿真，区分精确模式与快速覆盖模式。
3. Remcom 相关 RT/FDTD 产品：材料和几何边界明确绑定，电磁交互由材料和入射角决定。
4. Sionna RT：开源 RT 信道仿真框架，可参考其材质、路径交互、天线和信道系数组织方式。
5. 3GPP TR 38.901：不替代 RT 物理路径，但可作为 path loss、DS、AS、K factor 等统计对照基线。

#### 12.2.10 v9-A 最终交付标准

1. precise 几何路径与 EM 求解之间形成明确数据契约。
2. 所有透射路径具有完整 medium in/out 和 Snell residual。
3. 所有反射路径具有明确入射侧 medium 和另一侧 medium。
4. 双表面薄板可稳定得到 `air -> material -> air` 路径。
5. 材质缺失在 strict 模式下不再静默降级。
6. precise 模式可关闭硬剪枝并输出完整诊断。
7. 解析测试、几何集成测试、EM 集成测试和回归报告全部通过。
8. 文档中明确区分 precise 模式和 fast/SBR 近似模式。

### 12.3 主线 B：precise 复矢量电场传播与天线 Jones/姿态联合建模定版

本节为 v9 主线 B 的最终定版方案。主线 B 是 precise 模式和论文创新点的核心之一：将现有电场计算从“标量幅度 + 简化极化”升级为“复矢量电场沿几何路径传播”，并接入天线复方向图、极化匹配和姿态旋转。

#### 12.3.1 定版目标

1. 建立 precise 模式下的复矢量电场传播链路。
2. 让自由空间、反射、透射、绕射都对同一个复电场状态进行更新。
3. 让反射/透射的 TE/TM 分解和绕射的 soft/hard 分解都成为复矢量线性变换。
4. 让 Tx 天线发射和 Rx 天线接收支持方向相关的复 Jones 响应。
5. 让天线姿态旋转影响方向图查询、极化基变换、AoA/AoD 和接收功率。
6. 让多径 CIR 支持复相干叠加，而不是只做功率相加。
7. 输出论文需要的极化、co/cross、XPR、极化失配、姿态敏感性等指标。

主线 B 不是推翻现有 EM 流程，而是在保留 `InitializeTxField -> ApplyFreeSpaceSegment -> ApplyReflection/Transmission/Diffraction -> FinalizeAtReceiver` 流水线的基础上，升级其内部主状态表达和天线接口。

#### 12.3.2 当前不足

1. `FieldAccumulator` 当前主要由 `amplitude_real/amplitude_imag`、`phase_rad`、`power_linear`、`polarization_vector/polarization_imag` 共同表达场状态，容易退化为“复振幅 + 归一化极化”的近似模型。
2. 反射、透射、绕射中存在将复分量压回幅度模值、再用 `phase_rad` 单独累加的情况，TE/TM 或 soft/hard 分量间的相对相位可能被削弱或丢失。
3. 多次交互后的极化椭圆、交叉极化和相干相消难以严格保证。
4. 天线方向图目前更接近标量增益模型，不能完整表达 `Etheta/Ephi` 的复幅相。
5. Rx 端没有严格通过复 Jones 响应做极化匹配内积。
6. 天线姿态尚未成为统一坐标变换，方向图查询和极化基变换对旋转的响应不够完整。
7. `BuildCIR/PDP/Coverage` 等聚合需要区分复相干叠加和非相干功率统计。

#### 12.3.3 复矢量场主状态

v9-B 引入复矢量电场作为 precise EM 的主状态：

```cpp
struct ComplexVec3 {
    Complex x;
    Complex y;
    Complex z;
};
```

`FieldAccumulator` 增加：

```cpp
ComplexVec3 electric_field_world;
bool vector_field_valid = false;
```

旧字段暂时保留：

1. `amplitude_real/amplitude_imag`。
2. `phase_rad`。
3. `power_linear`。
4. `polarization_vector/polarization_imag`。

但旧字段逐步降级为从 `electric_field_world` 派生的兼容输出，避免一次性打断现有导出和报告。

派生关系建议：

1. `power_linear = |Ex|^2 + |Ey|^2 + |Ez|^2`。
2. `amplitude = sqrt(power_linear)`。
3. `polarization_vector/polarization_imag` 由归一化 `electric_field_world` 得到。
4. `phase_rad` 由参考极化分量或接收端复标量结果得到，不再作为内部唯一相位源。

#### 12.3.4 统一复矢量交互模型

反射、透射、绕射统一为复矢量线性变换：

1. 输入世界坐标复电场 `E_in`。
2. 根据入射方向、出射方向和交互几何构造局部正交极化基。
3. 将 `E_in` 投影到局部基上，得到两个复分量。
4. 对两个复分量分别乘以对应复系数。
5. 在出射方向对应的局部基上重构 `E_out`。

反射/透射：

```text
[E_TE', E_TM']^T = diag(r_TE, r_TM) [E_TE, E_TM]^T
[E_TE', E_TM']^T = diag(t_TE, t_TM) [E_TE, E_TM]^T
```

绕射：

```text
[E_soft', E_hard']^T = diag(D_soft, D_hard) [E_soft, E_hard]^T
```

建议新增工具函数：

1. `ComplexDot(const ComplexVec3&, const Vec3&)`。
2. `ScaleComplexVec3(const Vec3&, const Complex&)`。
3. `ProjectToPolarizationBasis(E, basis1, basis2)`。
4. `ReconstructFromPolarizationBasis(c1, basis1, c2, basis2)`。
5. `BuildReflectionBasis(incident, reflected, normal)`。
6. `BuildTransmissionBasis(incident, transmitted, normal)`。
7. `BuildDiffractionBasis(kOut, edgeDirection)`。

#### 12.3.5 自由空间与介质传播

自由空间段不改变极化基，只对复矢量电场整体施加传播相位、介质相位和介质衰减：

```text
E <- E * exp(-j k0 n d) * exp(-alpha d)
```

其中：

1. `k0 = 2*pi/lambda0`。
2. `n` 来自当前传播介质折射率。
3. `alpha` 来自当前传播介质衰减常数。
4. `d` 为当前几何段长度。

最终路径级 Friis/球面扩散因子仍可在 `FinalizeAtReceiver` 统一处理；后续宽带主线 C 可讨论是否改成逐段几何扩散或频域路径响应形式。

#### 12.3.6 天线姿态与坐标系

v9-B 增加天线姿态 frame：

```cpp
struct AntennaPose {
    Point3 position;
    Vec3 forward;
    Vec3 up;
    Vec3 right;
    bool valid;
};
```

规则：

1. `forward/up/right` 构成天线本地坐标系。
2. 输入配置可继续使用 `forward/up`，内部正交化得到 `right`。
3. 世界方向先变换到天线本地坐标，再计算 `theta/phi`。
4. Tx 使用 departure direction 查询本地复方向图。
5. Rx 使用 arrival direction 查询本地复方向图。
6. 世界电场进入 Rx 后，需要投影到 Rx 本地极化基，再与 Rx Jones 响应做复内积。

必须明确方向约定：

1. Tx departure direction：从 Tx 指向路径下一节点。
2. Rx arrival direction：从路径上一节点指向 Rx 的射线传播方向；查询接收天线时可根据方向图定义转换为“入射来向”。
3. `incident_direction` 永远表示射线前进方向，不表示从当前点回看上一点。

#### 12.3.7 Jones 方向图数据格式

v9-B 支持基础 Jones pattern CSV，不要求一次性支持 HFSS/CST 原生复杂格式。

建议格式：

```text
theta_deg,phi_deg,Etheta_re,Etheta_im,Ephi_re,Ephi_im
```

可选扩展字段：

```text
gain_dbi,phase_deg,frequency_hz,port_id
```

解释：

1. `Etheta/Ephi` 是天线本地球坐标极化基上的复方向图。
2. Tx 用该复方向图生成初始 `electric_field_world`。
3. Rx 用该复方向图作为矢量有效长度或接收 Jones 响应的简化模型。
4. 若只有旧标量 dBi pattern，则自动降级为单极化或配置指定极化方向的 Jones pattern。

#### 12.3.8 Tx/Rx 天线处理

Tx 初始化：

1. 根据路径第一段方向得到 Tx 本地 departure `theta/phi`。
2. 查询 Tx Jones pattern，得到 `Etheta/Ephi`。
3. 将本地 `theta/phi` 极化基旋转到世界坐标。
4. 初始化 `field.electric_field_world`。
5. 发射功率、天线效率和归一化策略由配置控制。

Rx 接收：

1. 根据路径最后一段方向得到 Rx 本地 arrival `theta/phi`。
2. 查询 Rx Jones response。
3. 将世界入射复电场投影到 Rx 本地极化基。
4. 做复内积：

```text
v_rx = h_rx^H * E_incident
```

5. `v_rx` 作为该路径最终接收复幅度，用于 CIR、PDP、覆盖和 ISAC 指标。

#### 12.3.9 多径复相干叠加

v9-B 后，`EMPathResult` 应保留：

1. 路径级接收复幅度 `received_real/received_imag`。
2. 路径级世界复电场或接收前复电场摘要。
3. 路径极化状态、co-pol/cross-pol 功率、XPR。
4. AoA/AoD 在世界坐标和天线本地坐标下的值。

`BuildCIR` 应区分两种模式：

1. 相干 CIR：同一 delay bin 内复数相加，再取功率。
2. 非相干 PDP：路径功率直接累加，用于统计功率延迟谱。

建议输出：

1. `cir_complex.json/csv`。
2. `pdp_incoherent.json/csv`。
3. `polarization_stats.json`。
4. `antenna_pose_response.json`。

### 12.4 主线 C：宽带频域信道、理想 CIR 与可观测基带 CIR 定版

本节为 v9 主线 C 的最终定版方案。主线 C 的目标是把 precise 几何路径和主线 B 的路径级复响应，升级成可用于论文、通信系统和实测对照的宽带信道输出，包括 `H(f)`、理想 delta CIR、有限带宽可观测 CIR、相干/非相干 PDP 以及时延分辨率相关指标。

#### 12.4.1 定版原则

1. 相同场景、相同几何结构、相对窄带频率范围内，默认使用同一套 precise 几何寻径结果。
2. 几何路径只在参考频率或中心频率下生成一次，频率 sweep 内复用该路径集合。
3. 每个频点只重评估频率相关的 EM 响应，包括传播相位、材料电参数、Fresnel/UTD 系数、介质衰减和天线响应。
4. RT 层必须输出理想多径参数和理想 delta CIR。
5. 为与实测设备对齐，系统还必须提供有限带宽、有限频点、带窗函数的测量等效 CIR。
6. CIR 输出复相干叠加结果；PDP 同时输出相干和非相干两类功率结果，供论文对比。
7. CFR 可以由理想 CIR 解析生成，也可以由逐频点 EM 重评估生成；两者都保留并明确适用范围。

#### 12.4.2 单频到宽带的仿真方式

当前单频仿真示例：

```text
frequency = 3 GHz
```

v9-C 扩展后支持：

```text
center_frequency = 3 GHz
bandwidth = 200 MHz
frequency range = 2.9 GHz ~ 3.1 GHz
frequency points = N
```

推荐流程：

1. 在中心频率或参考频率下运行 precise 几何寻径，得到 `GeometricPath_l`。
2. 对每条路径 `l`，在每个频点 `f_i` 上重新计算 EM 响应。
3. 得到路径级宽带复响应 `alpha_l(f_i)`。
4. 对所有路径求和得到总频域信道 `H(f_i)`。
5. 根据 `H(f_i)` 和设备频率采样设置生成可观测 CIR。

默认不对每个频点重新几何寻径。原因是：

1. 3 GHz 中心、200 MHz 带宽下，几何路径通常不会随频率显著变化。
2. 每频点重新寻径计算量会按频点数成倍增加。
3. 不同频点路径集合可能排序和数量不同，路径匹配复杂。
4. 频率主要影响电磁响应，而不是几何路径拓扑。

保留高级配置：

```text
retrace_per_frequency = false
```

仅当频率跨度极大、粗糙散射机制强烈频变或研究频率依赖路径可见性时，才考虑打开。

#### 12.4.3 两种 CFR 生成模式

模式 C1：固定路径增益的相位展开。

```text
H(f_i) = Σ_l alpha_l(f0) exp(-j 2π f_i tau_l)
```

特点：

1. 几何和 EM 都只在参考频率计算一次。
2. 宽带变化只体现不同频率下的传播相位。
3. 速度快，适合窄带近似和结果对比。
4. 若材料、天线和绕射系数在带宽内变化很小，该模式足够准确。

模式 C2：逐频点 EM 重评估。

```text
H(f_i) = Σ_l alpha_l(f_i)
```

其中 `alpha_l(f_i)` 由同一条几何路径在频点 `f_i` 上重新计算得到，包含该频点下的材料、天线、Fresnel/UTD、介质衰减和传播相位。

特点：

1. 几何路径固定，但 EM 响应随频率变化。
2. 能体现材料频散、天线频响和频率相关绕射/透射损耗。
3. 更适合论文高精度宽带信道结果。
4. 计算量约随频点数线性增加。

v9-C 默认支持两种模式，并建议论文实验中同时输出对比。如果二者差异很小，可说明窄带近似成立；如果差异明显，可分析材料频散或天线频响的影响。

#### 12.4.4 材料电参数频率模型

宽带仿真中材料电参数必须随频率查询：

```text
epsilon_c(f) = epsilon_r(f) - j sigma(f)/(omega epsilon0)
```

v9-C 定义三类材料频率模型：

1. `constant`：材料参数在频带内固定，适合调试或窄带近似。
2. `table_interpolation`：材料数据库给出多个频点的 `epsilon_r/sigma/mu_r`，仿真时插值。
3. `itu_formula`：根据 ITU 或文献模型使用公式计算材料参数，v9 可先预留接口，优先实现常用材料。

推荐优先级：

1. v9 先完整实现 `constant` 和 `table_interpolation`。
2. `itu_formula` 先设计接口和配置字段；若论文需要，可补常见材料公式。
3. 超出材料有效频率范围时，strict 模式失败，debug 模式 warning。

表格插值建议：

1. `epsilon_r` 可使用频率线性插值或 log-frequency 插值。
2. `sigma` 建议支持 log-log 插值，避免电导率跨数量级时失真。
3. 插值方法写入配置，导出报告中记录实际使用方法。

论文表述建议：材料介电常数和电导率随频率变化，由材料数据库插值或 ITU 建议书模型获得。

#### 12.4.5 理想 CIR 与可观测 CIR

RT 得到的理想 CIR 是：

```text
h_ideal(tau) = Σ_l alpha_l δ(tau - tau_l)
```

这里的 `δ` 是无限窄冲激，代表理想多径真值。真实 VNA 或 channel sounder 由于带宽有限，测到的是：

```text
h_observed(tau) = h_ideal(tau) * p(tau)
```

其中 `p(tau)` 是测量系统的等效脉冲响应。带宽越小，`p(tau)` 越宽，多个相近路径越容易被合并。

v9-C 必须同时输出：

1. `ideal_paths.json`：路径级理想多径参数。
2. `ideal_delta_cir.json`：理想 delta CIR。
3. `sampled_cfr.json`：按设备频点采样得到的 `H(f)`。
4. `observed_cir_ifft.json`：通过窗函数和 IFFT 得到的测量等效 CIR。
5. `observed_pdp.json`：测量等效 PDP。

这样既保留 RT 真值，也能与有限带宽实测对齐。

#### 12.4.6 可测 CIR 的生成方法

优先实现频域采样 + IFFT 方法，流程与 VNA 测量一致：

1. 给定设备频点：`f_start`、`f_stop`、`N_points`。
2. 生成或重评估 `H(f_i)`。
3. 乘窗函数：`H_windowed(f_i) = H(f_i) W(f_i)`。
4. IFFT 得到 `h_observed(tau_k)`。
5. 输出时延轴、复 CIR、功率 PDP 和窗函数参数。

同时保留时域卷积解释：

```text
h_observed(tau) = Σ_l alpha_l p(tau - tau_l)
```

两者本质等价。频域 IFFT 适合对齐 VNA/channel sounder；时域卷积适合解释带宽对理想冲激的展宽。

必须记录的测量等效参数：

1. `f_start/f_stop`。
2. `bandwidth`。
3. `point_count`。
4. `frequency_spacing`。
5. `window_type`。
6. `ifft_convention`。
7. `time_zero_reference`。
8. `delay_resolution`。
9. `max_unambiguous_delay`。

#### 12.4.7 CIR、PDP 和相干/非相干输出

v9-C 输出三类时延域结果：

复 CIR，相干叠加：

```text
h[k] = Σ_{l in bin k} alpha_l
```

相干 PDP：

```text
P_coh[k] = |Σ_{l in bin k} alpha_l|^2
```

非相干 PDP：

```text
P_incoh[k] = Σ_{l in bin k} |alpha_l|^2
```

论文解释：

1. 复 CIR 是通信基带信道表达。
2. 相干 PDP 体现同一时延 bin 内多径相干增强或抵消。
3. 非相干 PDP 体现能量到达分布，适合统计 delay spread。
4. 相干和非相干 PDP 的差异可以作为多径相干性分析内容。

#### 12.4.8 CFR 与 CIR 的关系

数学上，理想 CIR 的 Fourier transform 就是 CFR：

```text
H(f) = ∫ h(tau) exp(-j 2π f tau) d tau
H(f) = Σ_l alpha_l exp(-j 2π f tau_l)
```

因此，若 `alpha_l` 在频带内近似不变，由理想 CIR 解析生成 CFR 是准确的。若材料、天线、透射、绕射或介质衰减随频率变化明显，则必须使用逐频点 EM 重评估：

```text
H(f_i) = Σ_l alpha_l(f_i)
```

结论：

1. `FFT/解析变换 ideal CIR` 适合固定路径增益的快速宽带近似。
2. `frequency_sweep_EM` 适合高精度宽带仿真。
3. v9-C 同时提供两种结果，论文中可比较二者差异。

#### 12.4.9 时延分辨率与带宽

有限带宽决定可观测时延分辨率：

```text
delay_resolution ≈ 1 / B
```

单站雷达距离分辨率近似：

```text
range_resolution ≈ c / (2B)
```

通信链路路径长度差分辨能力可用：

```text
path_length_resolution ≈ c / B
```

v9-C 将这些作为后处理和报告指标输出。RT 内部仍输出理想多径真值，不因某个设备带宽直接丢弃或合并路径。不同带宽的实测对齐由 measurement-equivalent CIR 层完成。

### 12.5 主线 D：precise/SBR 绕射精细化与散射接口预留定版

本节为 v9 主线 D 的最终定版方案。v9-D 的大基调是“绕射更准 + 散射接口预留”，不全面实现漫散射。主线 D 聚焦现有 precise 模式和 SBR-coverage 模式中的绕射机制，使其几何语义更明确、UTD 参数更可诊断、覆盖近似更稳定，同时为未来散射模型保留接口。

#### 12.5.1 定版原则

1. v9-D 不把漫散射作为主贡献，不在论文主实验中使用未充分验证的散射结果。
2. v9-D 优先增强 precise 绕射的几何可信度、UTD 参数透明度和自测闭环。
3. v9-D 同步增强 SBR-coverage 绕射，使其更稳定、更可解释，但明确其仍是覆盖近似。
4. 散射只做配置、材质、交互类型、导出和扩展点预留，默认关闭。
5. 论文表述中明确：v9 precise 主链覆盖镜面反射、介质透射和 UTD 楔边绕射；粗糙表面漫散射作为未来扩展。

#### 12.5.2 当前 precise 绕射不足

1. wedge 几何语义需要更严格：`positive_face`、`negative_face`、edge direction、wedge angle、0-face、凸/凹关系需要明确。
2. UTD 中 `n=(2π-α)/π` 的 `α` 必须与代码中的 `wedge_angle_deg` 定义一致。
3. Keller cone 当前可检查，但缺少可导出的 residual、容差和失败原因。
4. Fermat 绕射点求解缺少 `edge parameter t`、端点 warning、`s1/s2` 等诊断。
5. UTD EM 计算中的 `phi`、`phip`、`beta0`、`L`、`Dsoft/Dhard` 等关键参数未形成完整验证报告。
6. SBR 绕射与 precise 绕射模型差异较大，当前更像覆盖近似，需要明确边界并尽量共享 wedge/UTD 语义。

#### 12.5.3 precise 绕射增强目标

1. 每条 precise 绕射路径都有明确 `wedge_id`。
2. 每条绕射路径都有 Fermat 最优点、`s1/s2`、边参数 `t`。

4. 不全面实现散射，避免引入缺少校准的不可信经验模型。
5. 散射接口预留保证未来可以扩展粗糙表面、家具、人体和目标散射。

#### 12.5.11 闭环自测方案

precise 绕射测试：

1. `KnifeEdge_Basic`：单个理想边缘应产生绕射路径。
2. `KellerCone_Residual`：绕射路径 Keller residual 小于阈值。
3. `FermatPoint_Interior`：最优点落在边内部，输出正常 `t`。
4. `FermatPoint_EndpointWarning`：最优点落在端点附近，输出 warning。
5. `WedgeAngleSweep`：wedge angle 从 30 到 150 度，UTD 系数无 NaN/Inf 和异常跳变。
6. `DiffractionVisibility`：当前点到绕射点或绕射点到 Rx 被遮挡时拒绝候选。
7. `DiffractionPolarization`：soft/hard 分量输出稳定，极化状态连续。
8. `DiffractionDiagnosticsExport`：路径或 validation report 中包含 wedge angle、Keller residual、s1/s2、Dsoft/Dhard。

SBR 绕射测试：

1. `SbrDiffraction_EnabledDisabled`：开关绕射后，NLOS 阴影区覆盖有合理差异。
2. `SbrDiffraction_RayCountConvergence`：射线数增加时绕射覆盖功率趋于稳定。
3. `SbrDiffraction_WedgeStats`：输出命中 wedge id 和绕射贡献占比。
4. `SbrVsPrecise_DiffractionSanity`：简单单边缘场景中，SBR 绕射覆盖趋势与 precise 绕射结果量级相近。

散射接口测试：

1. `ScatteringDisabledByDefault`：默认不生成 scattering path。
2. `ScatteringConfigParsed`：配置字段可读取、快照和导出。
3. `ScatteringNodeStrictFail`：手工构造 scattering node 时，precise strict 明确失败。
4. `ScatteringExportCompatibility`：导出格式支持 Scattering 类型但默认为空。

#### 12.5.12 理论依据与成熟参考

1. Keller 几何绕射理论。
2. Uniform Theory of Diffraction, UTD。
3. Fermat 原理：绕射点满足光程驻值或最短路径条件。
4. Kouyoumjian and Pathak UTD wedge diffraction。
5. Balanis 天线与传播教材中的 wedge diffraction 公式和 soft/hard 极化分解。

2. Ray launching 中基于 UTD 的边缘绕射贡献。
3. Wireless InSite/WinProp 等商业 RT 软件常用 GO + UTD 组合。
4. 覆盖模式下绕射通常是近似贡献，需要收敛性验证。

散射参考：

2. Directive scattering model。
3. Rough surface scattering。
4. Sionna RT 中 scattering coefficient / scattering pattern 的接口思想。
5. 商业 RT 中 diffuse scattering 通常依赖经验参数和实测校准。

2. wedge 几何语义在代码和文档中固定。
3. 绕射容差和退化保护配置化。

### 12.6 主线 E：precise/SBR 双模式寻径合理性、路径验证与候选控制定版

本节为 v9 主线 E 的最终定版方案。v9-E 不对现阶段 precise 和 SBR 的主要寻径架构做大规模重构，而是在现有架构基础上，进一步优化两种模式，使 precise 能找到更多、更准、更物理的路径，使 SBR 能更稳定地服务大规模覆盖统计。

#### 12.6.1 定版原则

1. 用户可见仿真模式只保留两类：`precise` 和 `sbr`。
2. `precise` 面向少量 Tx-Rx，量级约 `1e2~1e3`，目标是尽可能多、尽可能准地找到配置阶数内的物理路径。
3. `sbr` 面向大规模覆盖和统计仿真，例如极大规模 Rx 网格、功率覆盖图和活跃区域统计。
4. v9-E 不追求数学绝对全量路径枚举，而追求在配置预算、候选策略和容差下更少漏解、更少误删、更严格物理验证。
5. v9-E 不实现完整 mixed R/T/D 全局非线性求解器，只预留 `PathRefinementEngine` 作为未来增强。
6. precise 和 SBR 的职责边界必须清楚：SBR 可辅助发现候选和做 sanity check，但不替代 precise 路径真值。

#### 12.6.2 precise 模式现有机制理解

当前 precise 搜索本质是状态扩展式 deterministic candidate search：

1. 从 Tx 初始状态开始。
2. 每个状态尝试闭合到 Rx。
3. 若未闭合，则按预算调用反射、透射、绕射 expander。
4. 每个 expander 生成下一批 `PathState`。
5. 搜索队列继续扩展，直到达到深度、预算或找到可闭合路径。
6. 最终输出 `GeometricPathSet` 给 EM 模块。

这种框架可以自然组合 R/T/D 混合路径，例如 `Tx-R-T-D-R-R-Rx`，其路径由连续状态扩展形成。但它不是严格的全局混合路径边值求解器。镜像法适合纯反射或局部反射候选构造；透射由 Snell 状态机处理；绕射由 Fermat/Keller/UTD 候选处理；复杂混合路径依赖状态扩展、候选生成和最终物理验证共同保证。

#### 12.6.3 当前主要风险

1. 反射/透射/绕射候选存在硬编码 top-K 或数量限制，可能漏掉物理路径。
2. 状态签名和路径签名过粗，可能误合并不同路径。
3. 约束搜索中候选集合若未排序却使用 `binary_search`，可能随机漏候选。
4. PVS、AngularGrid、coarse SBR 若作为硬约束，可能把 precise 路径挡在候选集外。
5. 复杂混合路径虽然可以由状态扩展形成，但局部候选生成偏向尽快连 Rx，可能漏掉需要多次交互后才闭合的路径。
6. 缺少统一 path validation，候选路径进入 EM 前的物理残差检查不够集中。
7. SBR 模式缺少系统的 ray count 收敛性和统计稳定性报告。

#### 12.6.4 precise 模式优化目标

1. 保持现有 `SearchEngine + R/T/D Expander` 架构。
2. 修复严重漏解、误去重和候选集合实现风险。
3. 让所有剪枝、拒绝、去重和路径闭合都有 diagnostics。
4. 让反射、透射、绕射分别调用主线 A/D 的物理残差检查。
5. 建立统一 `ValidateGeometricPathPhysics`，所有路径进入 EM 前必须通过。
6. 对纯反射序列强化 generalized image method 验证或求解能力。
7. 对复杂 mixed path 保持状态扩展搜索，预留后续 path refinement。

#### 12.6.5 SBR 模式优化目标

1. 保持 SBR 作为独立 ray launching / shooting and bouncing rays 覆盖仿真模式。
2. 明确 SBR 不追求 precise path completeness，而追求大规模统计稳定和覆盖趋势。
3. 增强 SBR 的反射、透射、绕射物理一致性，尤其复用主线 A/D 的介质和绕射改进。
4. 输出 ray count、命中数、Rx 功率、机制贡献、活跃面元/楔边和运行时间。
5. 增加 ray count 收敛性测试，证明覆盖结果随射线数增加趋于稳定。
6. 小场景下可用 precise 结果对 SBR 的主要强路径和功率趋势做 sanity check，但不要求路径一一对应。

#### 12.6.6 precise 候选控制与剪枝策略

v9-E 将所有硬编码候选限制配置化：

```json
"path_search": {
  "candidate_control": {
    "enable_topk": false,
    "per_expander_keep_limit": 0,
    "per_state_keep_limit": 0,
    "wedge_max_distance_m": 0,
    "wedge_max_candidates": 0
  }
}
```

约定：

1. `0` 表示 unlimited。
2. precise 默认关闭硬 top-K。
3. 若用户打开 top-K，必须输出被剪枝候选数量和剪枝分数。
4. SBR 或 fast coverage 可使用 top-K/距离限制，但必须在报告中注明。

#### 12.6.7 状态签名与路径签名增强

状态签名应至少包含：

1. 完整交互类型序列 hash。
2. face/wedge id 序列 hash。
3. 当前点量化坐标。
4. 当前方向量化。
5. 当前介质 id。
6. 当前深度和剩余预算。
7. 累计长度 bucket。

路径签名应至少包含：

1. 交互类型序列。
2. 每个节点 face/wedge id。
3. 每个交互点量化坐标。
4. 每段介质 id。
5. 总长度 bucket。
6. Tx/Rx 坐标或 id bucket。

新增配置建议：

```json
"path_signature": {
  "position_quantization_m": 1.0e-5,
  "direction_quantization_rad": 1.0e-6,
  "length_quantization_m": 1.0e-5
}
```

目标是只去掉真正重复路径，不误删几何点、方向或介质状态不同的路径。

#### 12.6.8 约束搜索候选集合安全化

v9-E 必须修复候选集合实现风险：

1. 所有 `candidate_faces/candidate_wedges` 在使用前 sort + unique。
2. 或者统一转为 `unordered_set` 做 membership。
3. 禁止对未声明有序的 vector 直接 `binary_search`。
4. 若 BVH candidates 为空但 constrained candidates 非空，不应误跳过所有候选。
5. 输出 constrained search 的候选命中率、候选外拒绝数量和空候选原因。

#### 12.6.9 统一路径物理验证

新增或强化：

```cpp
ValidateGeometricPathPhysics(path, context)
```

进入 EM 前统一检查：

1. 每段可见性。
2. 每个节点是否落在对应 face/edge 内。
3. 反射残差。
4. Snell residual。
5. Keller residual。
6. medium continuity。
7. direction closure residual。
8. segment length 和 total length 一致性。
9. 自命中和重复交互是否合法。

输出：

1. `valid`。
2. failure reason。
3. per-node residual。
4. per-segment visibility status。

#### 12.6.10 混合路径与成熟算法参考

v9-E 对复杂混合路径采用状态扩展式搜索，不实现完整全局非线性求解器。成熟理论和算法参考如下：

1. Image method：用于单次或多次纯反射路径求解和验证。
2. Fermat principle：反射、折射、绕射都可视作光程驻值条件的不同边界形式。
3. Snell law：透射路径几何约束。
4. Keller cone / UTD：绕射路径几何和电磁约束。
5. Ray shooting + correction：先通过射线或状态扩展生成候选，再做路径物理验证或局部校正。
6. Beam tracing / ray tube：商业 RT 常用的降低漏检方法，v9 不全面实现，可作为后续增强。
7. Visibility graph / interaction graph：可将 face/wedge 视作图节点，后续可用于候选序列增强。

未来预留：

```text
PathRefinementEngine
```

用于给定 R/T/D interaction sequence 后求解或优化交互点，使整条路径满足光程驻值和物理残差最小。v9 只预留，不实现完整 nonlinear solver。

#### 12.6.11 SBR 收敛性与统计验证

SBR 需要输出并验证：

1. `ray_count`。
2. `max_depth`。
3. Rx hit count。
4. coverage power map。
5. active face/wedge set。
6. LOS/R/T/D 贡献占比。
7. runtime。
8. 不同 ray count 下 coverage RMSE 或功率变化率。

推荐 ray count 扫描：

```text
1e4, 5e4, 1e5, 5e5, 1e6
```

对于论文大规模仿真，可以展示射线数增加时覆盖结果收敛。

### 12.7 主线 F：CPU/GPU 统一查询后端、硬件自适应降级与低风险性能升级定版

本节为 v9 主线 F 的最终定版方案。v9-F 的大基调是：加速不能改变仿真语义。CPU 和 GPU 可以速度不同，但同一查询语义下的结果应尽可能一致；没有独显、CUDA 或 OptiX 的平台必须能完整运行 CPU 仿真；v9 做必要的加速架构整理和低风险性能升级，不为了速度破坏 precise 物理正确性。

#### 12.7.1 定版原则

1. CPU 后端作为 correctness reference，GPU 后端作为 acceleration backend。
2. precise 和 SBR 共用统一几何查询接口，不冗余实现互相不一致的查询逻辑。
3. 加速结构只改变速度，不改变 closest hit、all hits、range hits、visibility、occlusion、Rx grid 等查询语义。
4. GPU 不支持的能力必须自动 fallback CPU，不能用 closest hit 假装 all hits。
5. precise 模式优先正确性；SBR 模式优先吞吐，但仍需避免系统性几何语义错误。
6. 默认 `backend=auto`，没有 GPU 或 GPU 初始化失败时自动回退 CPU，除非用户显式 `require_gpu=true`。
7. v9 涉及架构级加速整理，但不进行大规模物理算法重写。

#### 12.7.2 统一查询接口与能力声明

v9-F 应在 `ISceneAccelerator` 或等价接口中增加后端能力声明：

```cpp
struct SceneAcceleratorCapabilities {
    bool supports_closest_hit;
    bool supports_all_hits;
    bool supports_range_hits;
    bool supports_occlusion;
    bool supports_batch_closest_hit;
    bool supports_batch_occlusion;
    bool supports_rx_grid_query;
    bool supports_ignored_face;
    bool supports_ignored_object;
    bool supports_material_filter;
    bool supports_double_precision;
    std::string backend_name;
    std::string device_info;
};
```

统一查询类型包括：

1. `QueryClosestFaceHit`：最近命中。
2. `QueryAllFaceHits`：全部命中，按距离排序。
3. `QueryFaceHitsInRange`：区间命中，用于透射、多层面和可见性诊断。
4. `IsOccluded/IsVisible`：线段遮挡和可见性。
5. `QueryClosestFaceHitBatch`：批量最近命中。
6. `IsOccludedBatch`：批量线段遮挡。
7. `QueryRxHitsBatch`：SBR Rx 网格命中。
8. `QueryCandidateWedges`：wedge 候选查询，可先 CPU 后端实现，未来扩展 GPU。

#### 12.7.3 SceneQuery 调度规则

`SceneQuery` 不应盲目调用当前 accelerator，而应按请求能力调度：

```text
requested capability -> check active backend -> use backend or CPU fallback
```

典型规则：

1. GPU 支持 closest hit，则 closest hit 可用 GPU。
2. GPU 不支持 all hits/range hits，则 precise 透射相关查询自动走 CPU。
3. GPU 不支持 ignored object 或 material filter，则相关查询自动走 CPU。
4. GPU batch occlusion 只有在线段 `tMax`、ignored face/object 语义完整时才可用于 precise。
5. strict GPU 模式下，所需 capability 不满足时直接报错；普通 auto 模式下 fallback CPU。

#### 12.7.4 CPU/GPU 一致性目标

CPU 使用 double，GPU 通常使用 float，因此不要求 bitwise 完全一致，但要定义容差并统计差异。

建议容差：

```text
distance_tol = max(1e-5 m, 10 * eps_intersection)
position_tol = max(1e-5 m, 10 * eps_intersection)
normal_dot_tol = 1e-5
```

边界样本单独标记为 `boundary_ambiguous`：

1. 射线打在三角形边或顶点附近。
2. 射线几乎平行于面。
3. 命中距离接近 self-hit ignore。
4. 极薄双表面间距接近容差。

普通样本应高度一致，差异必须进入 `backend_consistency.json`。

#### 12.7.5 硬件自适应与 fallback

配置建议：

```json
"acceleration": {
  "backend": "auto",
  "require_gpu": false,
  "gpu_device_id": 0
}
```

后端选择规则：

1. `backend=CPU_FaceBVH`：强制 CPU。
2. `backend=GPU_OptiX` 且 `require_gpu=true`：GPU 不可用则失败。
3. `backend=GPU_OptiX` 且 `require_gpu=false`：GPU 不可用则 fallback CPU。
4. `backend=auto`：尝试 GPU capability probe，失败自动 CPU。

必须适配：

1. CPU-only 平台。
2. 无独显平台。
3. NVIDIA GPU 存在但无 CUDA runtime。
4. CUDA 版本不匹配。
5. OptiX 初始化失败。
6. GPU device capability 不满足。
7. 多 GPU 设备选择。

日志必须明确 fallback 原因，例如：

```text
GPU unavailable: OptiX initialization failed. Fallback to CPU_FaceBVH.
```

#### 12.7.6 precise 模式中的 GPU 使用边界

precise 模式优先正确性：

1. closest hit 可用 GPU 加速。
2. batch occlusion 可用 GPU，但必须保证每条线段 tMax 正确。
3. all hits/range hits 若 GPU 不完整，必须回退 CPU。
4. 透射、多层面、薄板路径所需的 range hits 默认以 CPU reference 为基准。
5. final path validation 可选 CPU 复查，确保 GPU 加速不改变 precise path set。
6. GPU 不应因为 capability 不足导致 precise 漏路径。

#### 12.7.7 SBR 模式中的 GPU 使用边界

SBR 模式适合 GPU 加速：

1. GPU megakernel 可用于无绕射或简化机制覆盖。
2. GPU wavefront 可用于分层传播和批量查询。
3. Rx grid query 可优先 GPU。
4. 没 GPU 时必须有 CPU SBR fallback。
5. CPU/GPU SBR 参数必须一致，包括 ray count、max depth、接收球半径、材料参数、随机种子策略。
6. SBR 输出必须记录 backend、device、kernel runtime、吞吐量和统计误差。

#### 12.7.8 v9 必做的加速架构整理

P0 必做：

1. `ISceneAccelerator` capability flags。

#### 12.7.12 闭环自测方案

后端一致性测试：

1. `BackendClosestHitConsistency`：随机射线 CPU/GPU 最近命中一致。
2. `BackendRangeHitsConsistency`：多层面/薄板 range hits 数量和顺序一致。
3. `BackendOcclusionSegmentTMax`：目标点后方遮挡面不能被算作遮挡。
4. `BackendIgnoredFaceObject`：ignored face/object 语义一致。
5. `BackendThinSlabAllHits`：双表面薄板 all/range hits 正确。
6. `BackendRxGridConsistency`：CPU/GPU Rx grid 命中一致或在容差内。
7. `BackendPrecisionBoundaryCases`：边界样本单独标记，不污染普通一致率。

硬件 fallback 测试：

1. `BackendNoGpuFallback`：无 GPU 时自动 CPU。
2. `BackendCudaInitFailFallback`：CUDA/OptiX 初始化失败时 CPU fallback。
3. `BackendRequireGpuFail`：`require_gpu=true` 且 GPU 不可用时明确失败。
4. `BackendCpuOnlyBuild`：CPU-only 构建可运行 precise 和 SBR CPU 模式。

模式级测试：

1. `PreciseCpuGpuPathSetCompare`：小场景 precise CPU 与 GPU 加速路径集一致或差异可解释。
2. `SbrCpuGpuCoverageCompare`：SBR CPU/GPU coverage 统计在容差内。
3. `BackendConsistencyReportExport`：导出 `backend_consistency.json`。

### 12.8 主线 G：规范建模流程下的轻量输入预检、材质绑定校验与 ITU-R P.2040 电参模型定版

本节为 v9 主线 G 的最终定版方案。v9-G 不大规模重构现有场景导入、材质绑定和配置方法论，而是在用户规范建模流程基础上做轻量预检、鲁棒性增强和材质电参数模型规范化，避免错误输入静默污染 precise/SBR 仿真结果。

#### 12.8.1 定版建模准则

1. 所有可观测到的表面都在 Blender 中显式建模。
2. 不使用一个面元表示有厚度的物体；任何有厚度物体都由两面或多面组成。
3. 例如水泥墙由两个面元层表示，两个面元法向分别由水泥指向空气。
4. 每个此类面元在材质绑定 JSON 中明确 `front/back/surface`：通常 `front=air`，`back=concrete`，`surface=concrete`。
5. 所有面元在 Blender 中已处理为三角面元。
6. 所有法向在 Blender 中已处理准确，法向正侧即 `front` 侧。
7. v9-G 不负责自动修复复杂坏模型，而负责检查建模约定是否满足。

#### 12.8.2 主线 G 的边界

v9-G 要做：

1. 保持现有 OBJ 导入和材质绑定主流程。
2. 检查 OBJ 是否满足三角面、法向、索引、尺度等基本要求。
3. 检查材质绑定 JSON 是否覆盖全部相关对象/面元。
4. 检查 `front/back/surface` 是否完整且符合 transmission/reflection 需求。
5. 检查所有材料名是否能映射到 ITU-R P.2040 材料电参模型或显式自定义模型。
6. 输出 scene/material/preflight report。
7. 缓存 meta 中加入法向策略、材质模型、绑定文件和频率模型 hash。

v9-G 不做：

1. 大规模 importer 重写。
2. 完整 CAD/STL/FBX/glTF/STEP 支持。
3. 点云重建或自动建模。
4. 自动三角化复杂模型。
5. 自动修复非流形拓扑。
6. 自动推断物体内外侧。
7. 自动识别或反演场景材料。

#### 12.8.3 三档职责划分

建模阶段人工保证：

1. 全部可见面建模。
2. 有厚度物体双面或多面表达。
3. 所有面均为三角面。
4. 法向准确。
5. 材质绑定规则按场景明确书写。

预处理阶段自动检查：

1. 三角面格式。
2. OBJ 索引越界。
3. normal 存在性和 normal/winding 一致性。
4. 材质绑定覆盖率。
5. transmission semantic complete。
6. 材料模型存在性和频率有效范围。
7. scene bounds、尺度、薄板间距与 self-hit 容差关系。

仿真阶段物理验证：

1. medium continuity。
2. Snell residual。
3. reflection residual。
4. Keller residual。
5. visibility。
6. self-hit ignore。

#### 12.8.4 材质电参数模型

v9-G 明确：建筑材料频率相关电参数不采用简单数学插值作为默认策略，而采用 ITU-R P.2040 建议的电参值和幂律公式，计算任意目标频点下材料的相对介电常数和电导率。

抽象形式：

```text
epsilon_r(f) = a * f^b
sigma(f)    = c * f^d
mu_r(f)     = 1
```

注意：`f` 的单位必须严格遵循 ITU-R P.2040 参数表定义，通常需明确 GHz/MHz/Hz 单位转换，不能在实现中混用。

默认材料模型：

1. `itu_p2040`：默认推荐，用 ITU-R P.2040 给出的建议电参值和幂律参数计算目标频点电参数。
2. `constant`：仅用于 debug、窄带特例或明确自定义材料。
3. `measured_table`：保留给未来实测材料数据，不作为默认建筑材料策略。
4. `fit_power_law`：有多个可靠实测频点时可选拟合，v9 仅预留。

建筑材料的相对磁导率默认：

```text
mu_r = 1
```

该简化适用于普通建筑材料，如混凝土、木材、玻璃、砖、石膏板、天花板、地板、塑料等。金属材料建议作为高电导率材料或 PEC/metal boundary 处理，不通过 `mu_r != 1` 表达。

#### 12.8.5 材料绑定和 strict 策略

材质绑定 JSON 必须明确每类对象或面元的：

1. `surface_material_name`。
2. `front_material_name/front_medium_id`。
3. `back_material_name/back_medium_id`。
4. reflection/transmission/diffraction flags。

strict 策略：

1. precise 论文实验默认 strict。
2. 场景引用未知材料，失败。
3. transmission enabled 但 front/back 不完整，失败。
4. material map 中对象未匹配，失败或按配置 warning。
5. ITU-P.2040 材料模型缺失或频率超出有效范围，按 policy 失败或 warning。
6. debug 模式可允许 `warn_default_vacuum`，但必须写入 report，不能静默。

#### 12.8.6 法向和 front/back 语义检查

由于主线 A 依赖 `front=法向正侧`，v9-G 必须轻量验证法向：

1. 使用三角形顶点 winding 计算 geometry normal。
2. 读取 OBJ normal。
3. 检查二者点积。
4. 若 dot < 0，记录 normal mismatch。
5. 默认相信 Blender 输出，不自动修改；`validate_normals=true` 默认开启。
6. `rebuild_normals` 作为可选项，默认关闭。

配置建议：

```json
"scene_preprocess": {
  "validate_normals": true,
  "rebuild_normals": false,
  "normal_mismatch_policy": "warn"
}
```

#### 12.8.7 OBJ 导入策略

v9-G 不追求广泛 OBJ 兼容，只保证用户规范导出的 OBJ 能被稳定读取。

要求：

1. 支持现有 `f v//n`。
2. 支持常见三角 `f v/t/n`。
3. 可支持 `f v v v`，无 normal 时要求 `rebuild_normals=true` 或报错。
4. 所有 face 必须是三角面。
5. 不支持的四边形、多边形、负索引等格式应明确报错，而不是静默读错。
6. 顶点索引和 normal 索引越界立即失败。

### 12.9 主线 H：验证体系、参考对照和论文图表

讨论目标：把算法从“能运行”转成“能证明”。主线 H 的主要任务是建立一套可复现、可量化、可解释的多层验证方法论，从解析公式、几何路径、电磁传播、宽带信道、后端一致性、参考模型和论文图表多个角度，系统证明算法结果的真实性、物理正确性、可复现性和论文可信度。

大白话说：主线 A-G 是把算法做好，主线 H 是证明算法真的做对了。主线 H 不以新增某个传播模型为核心，而以形成“验证体系 + 验证用例 + 误差指标 + 自动报告 + 论文图表”为核心交付。

#### 12.9.1 主线 H 的边界

1. 主线 H 不替代 A-G 的算法修正，而是为 A-G 提供持续验证闭环。
2. 主线 H 不追求用某一个外部软件或某一组数据证明全部正确，而是建立多层证据链。
3. 没有实测数据时，不声称完成实测验证；应采用解析公式、第三方/参考实现和 3GPP 统计模型构成替代验证链。
4. 验证报告必须能说明误差来自几何、材质、天线、数值精度、采样收敛还是模型假设，而不是只给一个 pass/fail。
5. 论文图表应尽量由验证输出自动生成，避免手工改数据导致不可复现。

#### 12.9.2 分层验证体系

第一层是解析公式验证，用最简单、最可控的 canonical case 验证基础物理模块：

1. LOS Friis 路径损耗和自由空间传播相位。
2. 单界面 Snell 折射、Fresnel TE/TM 反射和透射系数。
3. Brewster angle 和全内反射。
4. two-ray 相干增强和抵消。
5. knife-edge 或简化 wedge 绕射。
6. 单频到宽带相位斜率。

第二层是几何路径验证，用于证明 `SearchEngine` 输出路径在几何上成立：

1. 每段 visibility、interaction sequence 合法性。
2. 反射残差、Snell residual、Fermat/Keller residual。
3. medium continuity。
4. 交互点是否在 face 内或 wedge 上。
5. path length 和分段长度一致性。

第三层是电场与极化验证，用于证明复矢量 EM、Jones 天线和姿态建模：

1. 复矢量场传播相位、TE/TM 投影与重构。
2. Jones 极化保持和旋转、Rx 极化匹配。
3. co/cross power 和 XPR。
4. Rx 姿态旋转 90 度后的接收功率变化。
5. 圆极化同手性/反手性接收差异。
6. 多径复相干叠加和抵消。

第四层是宽带信道验证，用于证明 CFR、CIR、PDP 和可观测基带 CIR：

1. 单路径 CFR 相位斜率对应传播时延。
2. 两路径 CFR 波纹间隔对应路径时延差。
3. IFFT CIR 峰值位置和理论时延一致。
4. 带宽与时延分辨率关系正确。
5. RMS delay spread 计算正确。
6. coherent PDP、incoherent PDP、ideal delta CIR 和 observed CIR 的差异可解释。

第五层是后端一致性与性能验证，用于证明 CPU reference、GPU acceleration 和 fallback 语义一致：

1. CPU/GPU closest hit、range hits、occlusion 对照。
2. ignored face/object、thin slab all hits、Rx grid query 对照。
3. precise CPU/GPU path set compare。
4. SBR CPU/GPU coverage compare。
5. runtime 和 speedup 报告。

第六层是外部参考对照，用于论文说服力：

1. 3GPP TR 38.901 InH path loss、LOS/NLOS 趋势、RMS DS、AS、K factor 统计范围对照。
2. Sionna RT 或其他开源 RT 的同场景路径、path loss、CIR/PDP、coverage trend 对照。
3. 必要时使用 `算法\` 下参考实现作为独立算法对照，但不作为主工程代码依据。
4. 若后续具备实测数据，则加入 VNA/channel sounder 或功率扫描数据对照。

#### 12.9.3 无实测数据时的替代验证链

如果短期没有实测数据，论文不能写成“完成实测验证”。建议形成以下替代链：

1. 解析公式验证证明基础物理模块正确。
2. 第三方 RT 或参考实现对照证明同类算法输出趋势一致。
3. 3GPP 统计模型对照证明大尺度统计指标合理。
4. SBR ray count convergence 证明覆盖图不是随机射线数不足造成的偶然结果。
5. CPU/GPU backend consistency 证明加速后端没有改变仿真语义。

可采用的论文表述：由于实际测量条件受限，本文采用解析公式、第三方 RT/参考实现和 3GPP 标准统计模型构成多层验证链，对模型进行交叉验证。若后续增加实测数据，应把实测作为最高等级的场景级验证，而不是替代基础解析验证。

#### 12.9.4 验证报告与导出产物

建议新增或规范以下输出：

1. `analytic_validation.json`：解析公式验证结果。
2. `geometry_validation.json`：路径几何合法性、残差和介质连续性。
3. `vector_em_validation.json` 和 `antenna_pose_validation.json`：复矢量电场、TE/TM、Jones 极化和姿态验证。
4. `wideband_channel_validation.json`：CFR、CIR、PDP、RMS DS 验证。
5. `backend_consistency.json` 和 `performance_report.json`：CPU/GPU 一致性和性能。
6. `3gpp_comparison.json`：3GPP InH 等标准模型对照。
7. `sionna_comparison.json` 或 `reference_comparison.json`：第三方或参考实现对照。
8. `validation_summary.md`：面向论文和答辩的汇总报告。

每个验证项至少记录：输入 case、解析值或参考值、仿真值、绝对误差、相对误差、容差、pass/fail、误差解释和关联代码模块。

### 12.11 主线 J：SBR 覆盖、收敛性和大规模仿真

讨论目标：明确 SBR 在 v9 中的定位，并对现有 SBR 覆盖、大规模 Rx 网格和 coarse candidate 功能做稳健性、物理性和可验证性升级。v9 不把 SBR 改造成完整 beam tracing/ray tube 商业级覆盖引擎，也不让 SBR 替代 precise 模式的高可信逐路径求解。

#### 12.11.1 v9 中 SBR 的定位

v9 只保留两种用户可见仿真模式：`precise` 和 `sbr`。

`precise` 的定位是少量 Tx-Rx 下的高可信确定性路径求解，目标是尽可能多、尽可能准地找到配置阶数内物理成立的路径，并由 EM 链路精细计算复场、极化、CIR/PDP 等结果。

`sbr` 的定位是大规模覆盖、统计趋势、活跃区域探索和候选辅助，不承诺逐路径完备性，也不作为替代 precise 的精确链路模式。对于 `1e7` 级 Rx 覆盖图、粗略功率分布、热点区域识别和 ray count 收敛分析，SBR 是合理工具。

大白话说：precise 负责“少量点算准”，SBR 负责“大范围看趋势”。两者可以互相校验，但不能混成一个语义。

#### 12.11.2 v9-J 的优化边界

v9-J 应做：

1. 修复 SBR 中明显不物理或不稳定的问题，例如 TIR、介质透射、反射/透射概率、接收球命中和路径长度记录。
2. 增加 ray count convergence report，用于证明覆盖图随射线数增加趋于稳定。
3. 增加 SBR 与 precise 的小场景对照，用于说明 SBR 统计趋势是否合理。
4. 明确 SBR 输出中的路径是采样路径或覆盖贡献，不等价于 precise 的确定性完整路径集合。
5. 统一 SBR CPU/GPU 的输出语义，避免 GPU 加速改变覆盖结果解释。
6. 让 SBR coverage 输出包含足够 diagnostics：ray count、hit count、receiver hit radius、max depth、random seed、backend、runtime、convergence level。

v9-J 暂不做：

1. 完整 ray tube/beam tracing 重构。
2. 全量漫散射模型。
3. 完整多分支能量守恒 SBR 树展开，除非作为局部试验开关。
4. 复杂重要性采样体系。
5. 用 SBR 取代 precise 作为论文核心逐路径真值。

#### 12.11.3 接收球和覆盖判定

当前 SBR 使用接收球命中 Rx，这对覆盖图是常见工程近似，但必须在文档和输出中显式记录接收球半径，否则不同仿真不可比。

建议 v9 中：

1. `rx_hit_radius` 必须进入配置快照和 coverage report。
2. 推荐支持按波长、Rx 网格间距或用户显式值设置接收球半径。
3. coverage 图必须记录每个 Rx 的 hit count 或 effective sample count。
4. 对低 hit count 的 Rx 标记为 low-confidence，避免把采样不足误认为真实深衰落。
5. 对论文图，必须同步给出 ray count 和收敛曲线。

后续 v10 或更长期可研究 ray tube/beam tracing 以降低接收球半径对结果的影响。

#### 12.11.4 SBR 与 precise 的关系

v9 中两者关系应明确为：

1. precise 是高可信单链路或少量链路 reference。
2. SBR 是大规模覆盖和统计趋势工具。
3. 小场景下可用 precise 检查 SBR 的 LOS、强反射、主要 NLOS 趋势是否存在。
4. 大场景下不能要求 SBR 与 precise 路径集合一一对应。
5. SBR coarse pass 可以为 precise 约束搜索提供活跃面候选，但候选不能成为唯一真值来源，必须保留 fallback 或漏检诊断。

#### 12.11.5 SBR 收敛性验证

v9-J 必须把收敛性作为 SBR 可信度核心，而不是只输出一张覆盖图。

建议至少输出：

1. 不同 ray count 下的 coverage heatmap。
2. 每个 Rx 或区域平均功率随 ray count 的变化。
3. 全局 RMSE/MAE 或相邻 ray count 差异。
4. low-confidence Rx 比例。
5. runtime vs ray count。
6. CPU/GPU coverage 差异。

推荐报告文件：

```text
sbr_convergence_report.json
sbr_coverage_diagnostics.json
sbr_backend_consistency.json
```

#### 12.11.6 v9-J 最终交付标准

1. SBR 模式和 precise 模式的语义在配置、导出和文档中明确区分。
2. SBR coverage report 包含 ray count、hit count、rx_hit_radius、random seed、backend、runtime 和 confidence 标记。
3. SBR 的 TIR、介质透射和基础路径长度/相位记录不再产生明显物理错误。
4. 至少支持一组 ray count convergence 验证。
5. 至少支持一组 precise vs SBR 小场景趋势对照。
6. CPU/GPU SBR 后端在同一配置下有一致性或统计一致性报告。
7. 论文中对 SBR 的表述限定为覆盖/统计趋势/候选辅助，不写成逐路径完备真值。

需要讨论的内容：

1. SBR 是否作为论文覆盖仿真的主要方法。
2. SBR 与确定性 SearchEngine 的结果如何互相校验。
3. 接收球半径如何选择，是否与波长、距离、Rx 尺寸相关。
4. 射线数收敛性如何证明。
5. 随机二选一反射/透射是否改为能量分裂或多重要性采样。
6. SBR 输出路径是否进入精确 EM 求解。

增删改优化方向：

1. 增加 ray count convergence report。
2. 修复 TIR 和介质透射。
3. 研究 energy splitting 替代随机二选一。
4. 接收球命中改为相位/路径长度更准确的 hit point 模型。
5. 区分 SBR coverage mode 和 precise path mode。

系统级影响或提升：

1. 覆盖图和大规模 Rx 网格更稳定。
2. 可展示计算效率和误差收敛曲线。
3. 若能量分裂增加，路径数和计算量会提高。

理论依据和成熟参考：

1. Shooting and Bouncing Rays 是商业 RT 覆盖仿真的常用方法。
2. Ray tube/beam tracing 可改善接收判定和能量守恒。
3. Wireless InSite、WinProp 等通常区分 deterministic ray tracing 和 coverage ray launching。

## 14. v9 代码修复任务清单

本节把前文讨论过的不足收敛为可执行代码任务。若后续正式开发，应按本清单逐项拆分 issue 或 commit，避免一次性大改造成不可验证风险。

### 14.1 P0-A 几何路径、介质和 Snell/Fresnel 语义

现有不足：

1. 透射路径可能不满足严格 Snell 几何闭合。
2. `ResolveMediumTransition` 原先更偏向决定 out medium，未严格校验 `state.current_medium_id` 是否等于入射侧 medium。
3. `PathNode` 携带的介质、法向、方向和残差信息不足，EM 层容易重新猜测介质语义。
4. `SnellRefract` 返回信息过少，不利于区分正常折射、全内反射、数值退化和非法入射。

修复方案：

1. 在 `core/search/ResolveMediumTransition.cpp` 中严格校验入射侧介质和当前传播介质一致。
2. 将 `SnellRefract` 升级为结构化 `SnellResult`，包含 success、tir、eta、cos_i、cos_t、refracted_dir、residual、failure_reason。
3. 扩展 `PathNode`，记录 `medium_in_id`、`medium_out_id`、`entered_from_front_side`、入射/出射方向、交互法向和几何 residual。
4. 在路径进入 EM 前增加 medium continuity validation。
5. EM 反射/透射只消费几何路径携带的介质语义，不重新默认空气/真空。

自测试闭环：

1. 单界面空气-玻璃 Snell case，检查折射角误差。
2. 玻璃-空气大入射角 TIR case，检查 TIR 标记和路径拒绝。
3. 双面墙体 case，检查 air-concrete-air medium sequence。
4. 反向法向 case，检查 preflight 或 path validation 能定位错误。

验收标准：

1. 每条透射路径的 medium continuity 通过。
2. Snell residual 小于配置容差。
3. TIR 不再生成非法透射路径。
4. EM 层不再依赖硬编码空气介质。

### 14.2 P0-B 复矢量 EM、极化和天线姿态

现有不足：

1. EM 主状态仍更接近标量幅相，不足以表达完整复矢量电场。
2. TE/TM 分解、Fresnel 复系数、极化旋转和 Jones 天线响应缺少统一主状态。
3. 旧导出字段和新极化语义之间缺少清晰派生关系。

修复方案：

1. 新增或完善 `ComplexVec3`。
2. 在 `core/em/FieldAccumulator.h` 中增加 `electric_field_world` 和 `vector_field_valid`。
3. 反射、透射、绕射统一实现为复矢量线性变换。
4. Tx/Rx 天线使用 Jones pattern 和 `AntennaPose` 做坐标变换。
5. 旧 `amplitude_real/amplitude_imag`、`phase_rad`、`power_linear`、`polarization_vector` 保留为派生兼容字段。

自测试闭环：

1. 线极化 LOS case，检查 Rx 同极化和交叉极化接收功率。
2. Rx 姿态旋转 90 度 case，检查接收功率变化。
3. 圆极化同手性/反手性 case。
4. 两径相干叠加 case，检查复场叠加结果。

验收标准：

1. `electric_field_world` 是 EM 内部主状态。
2. 旧标量字段可由复矢量状态稳定派生。
3. Jones 极化匹配与解析预期一致。
4. 多径相干叠加结果可复现。

修复方案：

2. 新增 `analytic_validation.json`、`geometry_validation.json`、`vector_em_validation.json`、`wideband_channel_validation.json`。
3. 新增 `backend_consistency.json` 和 `performance_report.json`。

### 14.4 P1-C 宽带信道、CFR/CIR/PDP

现有不足：

1. 当前 CIR/PDP 更偏窄带路径聚合，宽带有限带宽观测 CIR 支撑不足。
2. ideal delta CIR、sampled CFR、observed CIR、coherent/incoherent PDP 的边界需要明确。
3. 几何路径固定和逐频点 EM 重评估之间缺少统一导出语义。

修复方案：

1. 对同一几何路径集合逐频点重评估 EM 响应。
2. 同时支持固定路径增益 CFR 和逐频点 EM CFR。
3. 导出 ideal CIR、sampled CFR、observed IFFT CIR、coherent PDP、incoherent PDP。
4. 明确窗函数、频点间隔、带宽、IFFT 长度和时延轴定义。

自测试闭环：

1. 单路径 CFR 相位斜率验证传播时延。
2. 两路径 CFR 波纹间隔验证时延差。
3. IFFT CIR 峰值位置验证有限带宽观测结果。
4. RMS delay spread 手算 case。

验收标准：

1. 单路径相位斜率误差在容差内。
2. IFFT CIR 峰值时延与理论值一致。
3. ideal CIR 和 observed CIR 同时导出且语义清晰。

### 14.5 P1-E 搜索完备性、去重和候选控制

现有不足：

1. `StateSignatureBuilder` 和 `PathSignatureBuilder` 过粗，可能误去重。
2. hard-coded top-K 和 wedge 候选限制可能漏路径。
3. constrained search 中未排序 vector 使用 `binary_search` 有漏候选风险。
4. 搜索结果缺少足够 path validation diagnostics。

修复方案：

1. 增强 state/path signature，纳入 interaction sequence、face/wedge id、medium sequence、side info 和量化几何信息。
2. 将 top-K、wedge candidate radius、max candidate count 配置化。
3. 修复所有 binary_search 前置排序假设。
4. 增加 path validation pass，输出 rejected reason 和 residual。

自测试闭环：

1. 对称双反射场景，检查不误去重。
2. 多候选反射场景，检查 top-K 增大后路径单调不减少。
3. constrained search 与 full search 小场景对照。
4. path signature collision regression case。

验收标准：

1. 去重不合并物理不同路径。
2. top-K 配置化且导出实际剪枝统计。
3. constrained search 不因排序错误漏候选。

### 14.6 P1-F CPU/GPU 查询一致性

现有不足：

1. GPU batch occlusion 可能未使用线段实际 `tMax`。
2. GPU all/range hits、hit metadata 和 ignored face/object 语义可能与 CPU 不一致。
3. capability 不足时缺少清晰 CPU fallback 语义。

修复方案：

1. 在 `ISceneAccelerator` 增加 capability flags。
2. `SceneQuery` 按 capability 调度，不满足能力时自动 CPU fallback。
3. 修复 GPU batch occlusion 的 segment `tMax`。
4. 统一 CPU/GPU hit metadata、face id、distance、barycentric、normal 和 ignored entity 语义。
5. 增加 boundary ambiguous 分类和容差。

自测试闭环：

1. closest hit CPU/GPU 对照。
2. occlusion segment tMax 对照。
3. range hits/all hits thin slab 对照。
4. ignored face/object 对照。
5. precise path set CPU/GPU 对照。

验收标准：

1. CPU-only 必须完整可运行。
2. GPU capability 不足时结果语义不变，只影响性能。
3. CPU/GPU 差异在容差内或被标记为 boundary ambiguous。

### 14.7 P1-D 绕射精细化和散射接口预留

现有不足：

1. 绕射候选和 UTD 计算 diagnostics 不足。
2. Fermat/Keller residual 缺少统一导出。
3. 散射接口尚未预留清晰 interaction type 和材质入口。

修复方案：

1. 在 precise 绕射中导出 Fermat residual、Keller cone residual、wedge metadata。
2. UTD 计算失败时输出 structured failure reason。
3. SBR 绕射增加 hit statistics 和采样参数导出。
4. 预留 scatter interaction type、材质粗糙度入口和导出字段，但默认关闭。

自测试闭环：

1. canonical wedge/knife-edge case。
2. 不同 wedge angle 绕射损耗趋势。
3. Keller cone 边界 case。
4. 绕射关闭/开启对路径集合影响对照。

验收标准：

1. 绕射路径均有 Fermat/Keller diagnostics。
2. 失败路径有可解释 reason。
3. 散射接口存在但默认不影响现有结果。

### 14.8 P2-G 输入预检、材质和缓存

现有不足：

1. OBJ、法向、front/back/surface 绑定和材质频变模型缺少统一 preflight 报告。
2. 配置快照和 cache meta 对材质模型、法向策略和预处理参数变化识别不足。
3. 材料频变电参不能依赖简单数学插值作为默认策略。

修复方案：

1. 增加 scene preflight report。
2. strict 检查所有可传播 face 的 front/back/surface 绑定。
3. ITU-R P.2040 幂律模型计算目标频点 `epsilon_r/sigma`，频率单位严格匹配参数表。
4. 法向验证默认开启，`rebuild_normals` 默认关闭。
5. cache meta 纳入材质模型版本、法向策略、配置 hash 和预处理参数。

自测试闭环：

1. 缺失材质绑定 case。
2. 法向反转 case。
3. ITU-R P.2040 频点计算 case。
4. 配置快照 round-trip case。
5. cache invalidation case。

验收标准：

1. 错误输入能在仿真前明确报出。
2. 材料频变电参可复现。
3. cache 不复用过期材质或预处理结果。

### 14.9 P2-J SBR 覆盖和收敛性

现有不足：

1. SBR coverage 缺少 ray count convergence 证明。
2. 接收球半径、hit count、random seed 和 low-confidence 区域记录不足。
3. SBR CPU/GPU 输出语义需要对齐。

修复方案：

1. 导出 `sbr_coverage_diagnostics.json`。
2. 导出 `sbr_convergence_report.json`。
3. coverage report 记录 `rx_hit_radius`、ray count、hit count、effective sample count、random seed、backend、runtime。
4. 对 low hit count Rx 标记 low-confidence。
5. 小场景执行 precise vs SBR trend compare。

自测试闭环：

1. LOS coverage convergence case。
2. simple reflector coverage convergence case。
3. CPU/GPU SBR coverage consistency case。
4. 不同 ray count 下 RMSE/MAE 收敛曲线。

验收标准：

1. SBR 覆盖图必须附带采样和置信度信息。
2. ray count 增加时覆盖统计指标趋于稳定。
3. SBR 论文表述限定为覆盖、统计趋势和候选辅助。

## 15. v9 自测试闭环与验证方法

v9 的测试策略应分为四类：单元级解析测试、场景级几何/EM 测试、后端一致性测试和论文输出验证。每类测试都必须能说明它验证了哪个物理假设或工程语义。

### 15.1 最小必跑验证集

每次完成 P0/P1 主线修改后，至少运行以下验证：

1. Friis LOS 功率和相位。
2. Fresnel TE/TM 反射和透射。
3. Snell 折射和 TIR。
4. two-ray 相干叠加。
5. 单 wedge 或 knife-edge 绕射。
6. 单路径 CFR 相位斜率和 IFFT CIR 峰值。
7. medium continuity path validation。
8. CPU closest hit、occlusion、range hits。
9. precise 小场景 path set regression。
10. SBR ray count convergence smoke test。

### 15.2 报告格式

每个 validation report 应至少包含：

1. case name。
2. git 或源码版本标识。
3. 配置文件路径和配置 hash。
4. 场景文件路径和场景 hash。
5. 频率、带宽、Tx/Rx、天线和材质参数。
6. reference value。
7. simulated value。
8. absolute error。
9. relative error。
10. tolerance。
11. pass/fail。
12. failure reason。
13. related module。

### 15.3 验证命令建议

正式命令可在实现时确定，但文档层面建议保留以下入口：

```text
validation/run_all.py --suite smoke --backend cpu
validation/run_all.py --suite analytic --backend cpu
validation/run_all.py --suite geometry --backend cpu
validation/run_all.py --suite em --backend cpu
validation/run_all.py --suite wideband --backend cpu
validation/run_all.py --suite backend --backend cpu,gpu
validation/run_all.py --suite sbr --backend cpu,gpu
```

如果短期不引入 Python runner，也应在 C++ 测试或现有可执行程序中提供等价 suite 入口。

### 15.4 论文验证方法

论文中建议按以下证据链组织验证章节：

1. 解析公式验证：证明基础物理正确。
2. 几何路径验证：证明路径满足反射、透射、绕射和介质连续性。
3. 电磁极化验证：证明复场、TE/TM、Jones 和姿态处理正确。
4. 宽带信道验证：证明 CFR/CIR/PDP 和有限带宽观测关系正确。
5. 后端一致性验证：证明 GPU 加速不改变 CPU reference 语义。
6. 标准/参考对照：证明统计趋势或第三方结果一致。
7. SBR 收敛性验证：证明覆盖图不是采样不足造成的偶然结果。

## 16. v9 验收标准

v9 完成时，应满足以下系统级验收标准。

### 16.1 功能验收

1. 用户可见模式仅保留 `precise` 和 `sbr`。
2. `precise` 可输出经过 path validation 的几何路径、复场结果、CIR/PDP/APS 和 diagnostics。
3. `sbr` 可输出 coverage、采样统计、收敛性和置信度信息。
4. CPU-only 环境可以完成完整仿真和验证基线。
5. GPU 可用时作为可选加速，capability 不足时自动 fallback。
6. 宽带信道输出同时包含理想和可观测结果。
7. 输入预检能发现材质绑定、法向、配置和缓存风险。

### 16.2 物理正确性验收

1. LOS Friis、自由空间相位、Fresnel、Snell、TIR、two-ray、knife-edge/wedge 基础 case 通过。
2. 反射、透射、绕射路径均能导出几何 residual。
3. 介质序列连续，EM 不重新猜测传播介质。
4. 复矢量电场和 Jones 极化匹配结果符合解析预期。
5. 宽带相位斜率、CIR 峰值和 RMS delay spread 可验证。

### 16.3 工程可靠性验收

1. 配置快照无尾逗号等格式风险，能 round-trip。
2. cache meta 能识别场景、材质、配置和预处理策略变化。
3. CPU/GPU 查询差异有一致性报告和容差分类。
4. 所有关键导出包含足够 metadata，结果可复现。
5. validation failure 能定位到模块和原因。

### 16.4 论文支撑验收

1. 能生成论文所需的路径、PDP、CIR、CFR、APS、coverage、Fresnel、Snell、SBR 收敛和 CPU/GPU speedup 图。
2. 每张核心图都有固定输入、配置快照和生成脚本。
3. 文档明确 v9 已验证范围和未验证范围。
4. 对无实测数据的情况，论文采用解析公式、第三方/参考实现和 3GPP 统计模型构成替代验证链。

## 17. v10 后续工作边界

以下内容不纳入 v9 实施范围，统一作为 v10 或后续研究方向：

1. 完整 ISAC 指标体系，包括通信 capacity、sensing SINR、ISAC score 和 Pareto front。
2. 通信-感知联合参数扫描和资源优化。
3. 显式目标 RCS、目标回波路径标记和目标检测链路。
4. 距离-多普勒处理、检测概率、虚警概率、CRLB 和定位误差下界。
5. 完整漫散射模型和粗糙面散射校准。
6. 完整 ray tube/beam tracing 或 beam launching 重构。
7. 完整 mixed R/T/D global nonlinear solver。
8. 更复杂的 GPU wavefront path tracing 和多分支能量守恒 SBR 树。
9. 与商业 RT 软件或实测 campaign 的大规模系统校准。

v10 的前提是 v9 先完成可信 RT 底座。若 v9 的几何、EM、宽带、输入和验证闭环未完成，不建议提前推进 v10 范围。

## 18. 总结

v9 开发文档到此闭环。本文档已经从现有代码分析、理论依据、问题定位、主线设计、代码修复方向、自测试闭环、验证方法、验收标准和 v10 边界多个层面完成规划。

最终结论如下：

1. v9 的核心不是增加更多新概念，而是把现有 RT 功能做正确、做稳定、做可验证。
2. precise 和 sbr 是 v9 唯二用户可见仿真模式。
3. precise 负责少量链路高可信路径和 EM 结果。
4. sbr 负责大规模覆盖、统计趋势和候选辅助。
5. CPU 是 correctness reference，GPU 是 optional acceleration backend。
6. 几何和 EM 必须解耦，几何负责路径物理成立，EM 消费路径语义并计算复矢量电场。
7. 宽带信道必须同时支持 ideal CIR 和有限带宽 observed CIR。
8. 材质频变电参采用 ITU-R P.2040 幂律模型，输入场景依赖 strict preflight。
9. 验证体系是 v9 的核心交付之一，必须形成解析、几何、EM、宽带、后端、SBR 和参考对照的多层证据链。
10. ISAC 完整指标和系统级联合优化后移到 v10，v9 只提供可信 RT 信道底座和文档边界说明。

后续进入代码开发时，应严格按 P0/P1/P2/P3 优先级推进，并坚持“先定义验证，再修改代码，最后用报告闭环”的开发规则。
