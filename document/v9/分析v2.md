# Precise模式几何寻径重构：SBR拓扑探测 + 参数化精确求解

> 文档版本：v2.0  
> 用途：可直接提供给编码工具的完整实施规格  
> 核心思想：将SBR从"路径检测器"降级为"交互拓扑探测器"，消除接收球依赖，用BFGS参数化优化实现精确几何求解

---

## 一、总体架构与数据流

### 1.1 设计原则

整个方案围绕一个核心分工：

- **SBR阶段**：只负责发现"哪些面元/棱边序列在几何上可能构成传播路径"，完全不涉及Rx位置，输出的是与Rx无关的交互拓扑候选池
- **精确求解阶段**：给定面元序列，用参数化光程优化（Fermat原理）求出每个交互点的精确位置，Rx只在这个阶段介入

这一分工带来三个关键优势：
1. 接收球半径问题被彻底消除（SBR阶段不检测Rx）
2. 候选拓扑池对所有Rx共享，摊薄了SBR的计算开销
3. 精确度完全由优化求解器保证，与SBR采样密度解耦

### 1.2 五阶段流水线

```
阶段0：离线预计算（场景级，仅执行一次）
  输入：场景几何 + 搜索配置
  输出：合法交互类型序列集合（~500-800条）+ 场景交互图

阶段1：SBR拓扑探测（per Tx，对所有Rx共享）
  输入：Tx位置 + 场景 + 类型配额配置
  输出：CoarseCandidatePool（面元/棱边交互拓扑候选池）

阶段2：Rx相关性过滤（per Rx，轻量级）
  输入：候选池 + Rx位置 + BVH
  输出：对当前Rx可能相关的面元序列子集

阶段3：参数化精确求解（per Rx per 候选序列）
  输入：面元序列 + Tx + Rx + 材质数据库
  输出：精确交互点坐标序列（满足Fermat极值条件）

阶段4：几何验证与去重（per Rx）
  输入：精确求解结果
  输出：通过验证的最终路径集合
```

### 1.3 计算量分配（对1e2-1e3个Rx的场景）

| 阶段 | 执行频率 | 预期耗时 | 跨Rx复用 |
|------|---------|---------|---------|
| 阶段0 类型枚举 | 1次/场景配置 | <1ms | ✅ 完全共享 |
| 阶段1 SBR探测 | 1次/Tx | 500ms-2s | ✅ 所有Rx共享 |
| 阶段2 Rx过滤 | 1次/Rx | 5-20ms | ❌ per Rx |
| 阶段3 BFGS求解 | N次/Rx（N=候选序列数） | 0.5-2ms/条 | ❌ per Rx |
| 阶段4 验证 | 1次/Rx | 10-30ms | ❌ per Rx |

## 二、阶段0：离线预计算

### 2.1 类型序列枚举器

**职责**：在不涉及任何几何信息的情况下，穷举所有满足配额约束的交互类型排列。

**输入**：
- `max_R`：最大反射次数（如4）
- `max_T`：最大透射次数（如2）
- `max_D`：最大绕射次数（如1）
- `max_total`：最大总交互次数（如8）

**算法**：深度优先递归枚举，每一步选择追加R、T或D，检查约束后递归。

**约束规则**（可配置，影响完备性与效率的权衡）：
- 追加某类型前检查该类型剩余配额 > 0
- 追加后总深度不超过`max_total`
- 可选规则：禁止连续两次绕射（D→D）
- 可选规则：禁止在绕射后立即透射（D→T，物理上罕见）

**输出**：`TypeSequence`列表，每条记录包含：
- 交互类型的有序数组，如`[R, T, R, R, T, R]`
- 各类型计数`(n_R, n_T, n_D)`
- 总深度`k`
- 序列的唯一哈希标识

**规模**：配置`{max_R=4, max_T=2, max_D=1, max_total=8}`下约产生500-800条序列。

### 2.2 场景交互图构建

**职责**：将已有的`FacePVS`和`EdgeAdjacency`数据，按交互类型重组为三张分类邻接表，供后续阶段快速查询。

**三张邻接表**：
- **R邻接表**：面元→该面元PVS可见且`reflection_enabled=true`的面元集合
- **T邻接表**：面元→该面元PVS可见且`transmission_enabled=true`的面元集合  
- **D邻接表**：面元/棱边→邻近楔边，以及楔边→其绕射可见的面元集合

**额外预计算**：
- 按交互类型预分组的全局候选集：`all_R_faces`、`all_T_faces`、`all_D_edges`（用于阶段1初始可见性检测的快速候选迭代）
- 每个面元的包围球半径（用于射线锥相交判断）

## 三、阶段1：SBR拓扑探测

这是整个方案的核心创新模块。

### 3.1 射线锥（Ray Cone）机制

标准SBR发射的是几何意义上的射线（1D），与面元的相交是精确的点。本方案将射线升级为射线锥：锥轴是传播方向，锥有一个半角`α`，锥与场景的相交结果不是单个面元，而是一个**面元候选集合**（所有与锥体相交的面元）。

**射线锥的两个参数**：
- `cone_half_angle`：锥半角，决定每跳的候选面元范围
- `cone_axis`：锥轴射线，决定传播的中心方向

**锥半角的演化规则**：
- 初始半角由初始射线密度决定：`α₀ = sqrt(4π / N_rays)`，保证相邻锥无间隙
- 每次反射后：`α += cone_spread_per_bounce`（小增量，约0.5-1°）
- 每次透射后：`α += refraction_uncertainty + cone_spread_per_bounce`（折射引入额外方向不确定性）
- 每次绕射后：`α = diffraction_base_spread`（重置为较大值，如5-10°，因Keller锥覆盖范围广）

**与标准SBR的关键区别**：在每个交互点，不是只记录锥轴命中的单个面元，而是记录所有与锥体相交的面元，每个都作为独立候选产生新的子射线锥。

### 3.2 初始射线生成策略

**基础层（均匀覆盖）**：
- 使用Fibonacci球面分布生成`N_base`个均匀方向（推荐8000-15000）
- 仅保留能命中场景的方向（BVH预检查，过滤空方向）
- 每个方向的初始锥半角 = `sqrt(4π / N_base)`

**加密层（可选，对特殊几何加密）**：
- 对Tx正对的大面元方向加密采样（大面元漏检风险低，可跳过）
- 对Tx附近的楔边方向加密采样（绕射起始方向需要覆盖）

**实践参数建议**：
- 室内小场景（<5k面元）：`N_base = 5000`
- 室内中等场景（5k-20k面元）：`N_base = 10000`
- 室内复杂场景（>20k面元）：`N_base = 20000`

### 3.3 锥与场景的相交逻辑

对每个射线锥状态，与场景求交的流程：

**Step 1 - 锥轴精确求交**：对锥轴射线做标准BVH最近命中查询，得到命中面元`f_axis`和命中距离`d`。

**Step 2 - 候选面元扩展**：以命中点为中心，以`d × tan(α) × expansion_factor`为半径（`expansion_factor`约1.5），查询该球内的所有面元。这里使用空间索引（BVH球查询或AngularGrid方向索引）。

**Step 3 - 类型过滤**：根据当前射线锥的剩余配额和下一步允许的交互类型，过滤候选面元：
- 如果`remaining_R > 0`：保留`reflection_enabled=true`的面元
- 如果`remaining_T > 0`：保留`transmission_enabled=true`的面元
- 如果`remaining_D > 0`：保留附近楔边

**Step 4 - 结果**：返回`(face_id, interaction_type, hit_approximate_point)`的列表，每个条目产生一条子射线锥。

### 3.4 弹射展开与候选池构建

**展开方式**：使用显式队列（BFS宽度优先）控制展开，避免深度优先的栈溢出和计算不均匀。

**每次出队处理**：
1. 取出当前`RayConeState`（包含锥轴、锥半角、已走过的面元序列、剩余配额）
2. 调用锥与场景相交逻辑，得到候选命中列表
3. 对每个候选命中：
   - 将`(当前序列 + 新面元ID, 对应交互类型)`哈希后存入候选池
   - 如果剩余配额允许继续展开，构造新的`RayConeState`入队
4. 新`RayConeState`的构造规则：
   - 反射：用反射定律计算新锥轴方向（以命中面元法线反射），更新锥半角
   - 透射：用Snell定律计算折射方向（以面元中心材质参数估算），更新锥半角
   - 绕射：在楔边上取若干采样点，按Keller锥条件计算绕射方向，每个方向产生一条子射线锥

**候选池的存储结构**：
- Key：面元/棱边ID序列的哈希值（与Tx/Rx位置无关）
- Value：`FaceInteractionSequence`，包含面元ID序列、对应交互类型序列
- 去重：相同面元序列（即使由不同射线锥发现）只保留一份
- 规模控制：设置候选池最大容量上限（如50000条序列），超出时按路径长度排序丢弃最长的

**绕射处理的特殊性**：
- 绕射不发生在面元上，而发生在楔边上
- 当射线锥命中面元附近的楔边时，记录`edge_id`而非`face_id`
- 绕射后的子射线需要按Keller锥条件采样多个方向（推荐8-16个方向均匀覆盖Keller锥）

### 3.5 配额约束的执行

每个`RayConeState`携带`(remaining_R, remaining_T, remaining_D, remaining_total)`四元组。产生子射线时：
- 反射子射线：`remaining_R--`，`remaining_total--`
- 透射子射线：`remaining_T--`，`remaining_total--`
- 绕射子射线：`remaining_D--`，`remaining_total--`
- 任意值降为0时，对应类型的展开停止
- `remaining_total`降为0时，该射线锥停止所有展开

## 四、阶段2：Rx相关性过滤

### 4.1 过滤逻辑

对候选池中的每条面元序列，判断其是否可能到达当前Rx。

**过滤条件（从宽松到严格，按序执行）**：

**条件1 - 最后节点对Rx的可见性**：对面元序列的最后一个节点（面元或棱边），检查从该节点中心到Rx是否有遮挡（一次BVH射线查询）。不通过则直接丢弃。

**条件2 - 最后节点朝向检查**：对于反射/透射面元，检查Rx是否在面元的正确一侧（法线方向判断）。不通过则丢弃。

**条件3 - 粗略路径长度检查（可选）**：估算路径总长度下界是否在合理范围内，用于极端情况剪枝。

**过滤结果**：通常10%-40%的候选序列对特定Rx相关，这是候选池大小的进一步压缩。

### 4.2 Rx可见面元预计算（优化）

对每个Rx，在进入过滤阶段前，先做一次全场景可见性预查询：从Rx向所有`all_R_faces ∪ all_T_faces ∪ all_D_edges`发射射线，建立`visible_from_rx`集合。

过滤阶段的条件1变为：检查面元序列最后节点是否在`visible_from_rx`集合中（O(1)哈希查询，而非BVH查询）。

这个预计算开销约10-20ms，但能将过滤阶段的BVH查询次数从`|候选池| × BVH查询`降为`|候选池| × O(1)`。

## 五、阶段3：参数化精确求解

### 5.1 核心数学框架

**Fermat原理的统一表述**：对给定的面元序列`[f₁, f₂, ..., fₖ]`，在每个面元上存在一个交互点`xᵢ`，使得光程长度泛函取极值：

```
L(x₁, x₂, ..., xₖ) = n₀·‖x₁-Tx‖ + n₁·‖x₂-x₁‖ + ... + nₖ·‖Rx-xₖ‖
```

其中`nᵢ`是第`i`段的等效折射率（反射和绕射段折射率不变，透射段折射率按材质切换）。

**参数化**：每个交互点被参数化为所在面元上的二维坐标。面元是三角形，可用两个参数`(u, v)`的仿射组合表示面元内任意一点：`xᵢ = A·[u,v]ᵀ + b`，其中A是面元两条边构成的2×3基矩阵，b是面元某顶点坐标。楔边上的交互点只需一个参数`t`表示在棱边上的位置。

**优化问题**：对所有交互点的参数向量`T = [u₁,v₁, u₂,v₂, ..., uₖ,vₖ]`，最小化`L(T)`。

驻点条件`∇L = 0`等价于：
- 反射点：入射角等于出射角（反射定律）
- 透射点：Snell折射定律自动满足
- 绕射点：Keller锥条件满足

因此BFGS的最优解就是满足所有物理约束的精确交互点，无需额外约束方程。

### 5.2 纯反射子链：级联镜像法（解析解）

对面元序列中的连续纯反射段`[f_a, f_{a+1}, ..., f_b]`（中间无T或D），使用级联镜像法给出O(k)解析解，不需要BFGS迭代。

**算法**：
1. 从target点出发，逆序对每个反射面做镜像，得到最终镜像点`target_final`
2. 从source点向`target_final`连线，与`f_a`求交，得到第一个反射点`P_a`
3. 从`P_a`向逐步解除镜像的中间target点连线，依次求交，得到所有反射点

**使用时机**：
- 路径全为反射（纯R序列）：整条路径用级联镜像法
- 混合路径中的反射段：将连续反射段识别为子问题，用级联镜像法求解该子段，其source和target由BFGS的当前参数估算

### 5.3 混合路径：BFGS参数化优化

**BFGS的输入**：
- 面元序列和对应交互类型
- Tx和Rx位置
- 各面元的几何数据（顶点、法线）
- 材质数据库（透射面的折射率）
- 初始参数猜测（见5.4）

**梯度计算**：解析梯度（不用数值差分）。对第`i`个交互点的参数`(u,v)`：

```
∂L/∂u = n_{i-1} · dot(dir_in, ∂x/∂u) + n_i · dot(-dir_out, ∂x/∂u)
```

其中`dir_in = (xᵢ - x_{i-1}) / ‖xᵢ - x_{i-1}‖`，`dir_out = (x_{i+1} - xᵢ) / ‖x_{i+1} - xᵢ‖`，`∂x/∂u`是面元基矩阵A的第一列。

**Hessian逆矩阵更新**：使用标准BFGS的Sherman-Morrison更新公式，存储稠密逆Hessian矩阵（对k≤8，总参数数≤16，矩阵规模16×16，完全可接受）。

**线搜索**：使用Wolfe条件或固定点迭代线搜索（推荐固定点迭代，在RT路径求解中收敛更稳定）。

**收敛判断**：梯度范数 `‖∇L‖ < tol`（推荐`tol = 1e-8`）。

**处理参数越界**：若BFGS步骤导致某交互点参数越出面元边界（超出三角形），使用投影梯度方法：将参数投影回最近的三角形内点，继续迭代。若投影后梯度不变或连续投影，说明该面元序列无有效路径，标记为失败。

### 5.4 初始值策略

初始参数猜测质量直接影响BFGS是否能找到正确极值点。

**方案A - 面元重心**：将每个交互点初始化为对应面元的重心。简单但对某些几何形状可能收敛到错误极值。

**方案B - 直线投影**：将Tx→Rx直线（或Tx→面元重心→Rx折线）投影到每个面元上，以投影点作为初始值。对多数路径效果优于方案A。

**方案C - SBR近似点**：利用阶段1中射线锥的近似命中点作为初始值。这是三种方案中最接近真实解的初始猜测。

**推荐策略**：优先使用方案C（SBR近似点），若该点不在面元三角形内（因为射线锥命中是近似的），退化到方案B；方案B失败则退化到方案A。

### 5.5 求解器的路由逻辑

对每条候选面元序列，按以下规则选择求解路径：

```
if 序列全为R:
    → 级联镜像法（解析解，最快）
    
elif 序列包含连续R段 + 少量T/D:
    → 识别连续R子段 → 子段用级联镜像法
    → T/D节点 + R子段结果 → 组合进BFGS（降低参数维度）
    
else:
    → 全参数BFGS优化
```

## 六、阶段4：几何验证

### 6.1 验证项目

对BFGS输出的每组交互点，按以下顺序逐项验证：

**验证1 - 点在面元内**：每个交互点必须在对应面元/棱边的几何范围内（重心坐标法，判断是否在三角形内）。失败则丢弃该路径。

**验证2 - 各段无遮挡**：路径的每个线段（Tx→P₁, P₁→P₂, ..., Pₖ→Rx）必须在场景中无遮挡（BVH射线查询，忽略端点所在面元）。任一段有遮挡则丢弃。

**验证3 - 物理约束残差**：
- 反射点：入射角与出射角之差 < `angle_tol`（推荐0.1°）
- 透射点：Snell残差`|n₁sinθ₁ - n₂sinθ₂| < snell_tol`（推荐1e-6）
- 绕射点：Keller条件残差 < `keller_tol`

**验证4 - 路径唯一性去重**：对通过前三项验证的路径，计算面元序列哈希+交互点坐标哈希，去除重复路径（不同初始值收敛到同一解的情况）。

### 6.2 验证顺序的重要性

按上述顺序执行验证，因为：
- 验证1最快（纯数学计算，无BVH查询）
- 验证2较慢（BVH查询），但在验证1之后大量路径已被过滤
- 验证3用于检查BFGS是否真正收敛（部分情况BFGS收敛到鞍点或局部极值）

## 七、数据结构规格

### 7.1 核心数据结构一览

**`TypeSequence`**（阶段0输出）：
- `steps: vector<InteractionType>`：类型序列
- `n_R, n_T, n_D: uint8_t`：各类型计数
- `depth: uint8_t`：总深度
- `hash: uint64_t`：序列哈希

**`RayConeState`**（阶段1内部）：
- `axis_origin, axis_direction: Vec3`：锥轴射线
- `cone_half_angle: float`：当前锥半角
- `face_ids: vector<int>`：已经过的面元/棱边ID
- `itypes: vector<InteractionType>`：对应交互类型
- `approx_hit_points: vector<Vec3>`：SBR近似命中点（供阶段3初始化用）
- `remaining_R, remaining_T, remaining_D, remaining_total: uint8_t`：剩余配额

**`FaceInteractionSequence`**（阶段1输出/阶段2输入）：
- `face_ids: vector<int>`：面元/棱边ID序列
- `itypes: vector<InteractionType>`：对应交互类型
- `approx_points: vector<Vec3>`：SBR阶段的近似交互点（用于BFGS初始化）
- `type_seq_hash: uint64_t`：对应的类型序列哈希

**`CoarseCandidatePool`**（阶段1输出）：
- `sequences: unordered_map<uint64_t, FaceInteractionSequence>`：去重候选池
- 统计字段：`total_rays_shot`, `unique_sequences_found`等

**`ParameterizedInteraction`**（阶段3内部）：
- `face_or_edge_id: int`
- `itype: InteractionType`
- `A: double[6]`：面元基矩阵（3×2，列主序），棱边仅用前3个元素
- `b: double[3]`：参考点（面元某顶点坐标）
- `t: double[2]`：当前参数值（优化变量），棱边仅用`t[0]`
- `n_before, n_after: double`：入射侧和出射侧折射率

**`SolvedPath`**（阶段3输出/阶段4输入）：
- `interaction_points: vector<Vec3>`：精确交互点坐标
- `face_ids: vector<int>`：对应面元/棱边
- `itypes: vector<InteractionType>`
- `optical_path_length: double`：收敛后的光程长度
- `converged: bool`：BFGS是否收敛

## 八、与现有代码的集成方式

### 8.1 新增模块清单

| 模块名 | 对应阶段 | 复用现有组件 |
|--------|---------|------------|
| `TypeSeqEnumerator` | 阶段0 | 无（纯逻辑） |
| `SceneInteractionGraphBuilder` | 阶段0 | `FacePVS`、`EdgeAdjacency` |

| `GeometricPathValidator` | 阶段4 | `BVH/SceneQuery` |
| `PreciseSearchEngineV2` | 总调度 | 以上所有模块 |

### 8.2 与现有`SearchEngine`的并存方式

新引擎`PreciseSearchEngineV2`与现有`SearchEngine`并行存在，通过配置项`enable_precise_v2 = true/false`切换。两者接受相同的`PathSearchContext`输入，输出相同格式的`SearchEngineResult`。

`RtPipeline`中添加一个`if/else`分支：若启用V2则调用新引擎，否则调用原引擎。其余下游（EM计算、路径导出、验证）完全不变。

### 8.3 现有组件的复用说明

- **`FacePVS`（152k entries）**：在阶段0构建场景交互图时直接读取，不修改
- **`EdgeAdjacency`（645k pairs）**：同上，用于构建D邻接表
- **`AngularGrid`**：在阶段1射线锥相交的"候选面元扩展"步骤中，用方向索引快速查找近轴面元
- **`MaterialDatabase`**：在阶段1的透射Snell计算和阶段3的折射率查询中使用
- **`MirrorPointAcrossPlane`**：在阶段3的级联镜像法中直接调用
- **`SnellRefractV2`**：在阶段1的透射子射线方向计算中使用
- **`GoldenSectionSearch`**：在阶段3中可作为单绕射路径的1D优化备选（对比BFGS）

| 参数名 | 推荐值 | 影响 |

### 9.2 完备性与速度的权衡点

```
完备性优先（论文验证模式）：
  sbr_base_ray_count = 20000
  cone_spread_per_bounce = 1.0°
  diffraction_keller_samples = 16
  → 预期耗时：阶段1约3-5s per Tx，但对所有Rx共享

  sbr_base_ray_count = 5000

  diffraction_keller_samples = 8
  → 预期耗时：阶段1约0.5-1s per Tx

推荐在开发阶段用完备性优先模式，与现有SearchEngine输出做对比验证，
确认无明显漏检后再调整到速度优先模式。
```

## 十、实施路线（分迭代交付）

### Iter 1 — 级联镜像法（纯反射精确求解验证）
**目标**：验证精确求解模块可以正确工作。  
**内容**：实现`CascadedMirrorSolver`，支持任意深度的纯反射路径求解。  
**验证**：用已知几何解析解的简单场景（如两面平行墙的反射路径）做数值验证，要求交互点坐标误差 < 1mm。  
**与现有代码对比**：在412个测试场景上，与现有`ExpandReflection`输出对比，统计路径条数差异。

### Iter 2 — BFGS参数化求解器（混合路径精确求解）
**目标**：实现通用的参数化优化求解器。  
**内容**：实现`ParameterizedBfgsSolver`，支持R/T/D任意组合序列，包含解析梯度计算和固定点线搜索。  
**验证**：用手动构造的混合路径场景（已知精确解）做收敛性测试，统计收敛率和收敛精度。

### Iter 3 — SBR拓扑探测（射线锥粗筛核心）
**目标**：实现候选池生成模块。  
**内容**：实现`SbrTopologyProber`，包含射线锥生成、锥与场景相交、弹射展开、候选池去重。  
**验证**：检查已知存在的路径（来自现有SearchEngine的输出）是否都在候选池中被发现，统计召回率。

### Iter 4 — Rx过滤 + 完整流水线集成
**目标**：端到端流水线可运行。  
**内容**：实现`RxCandidateFilter`，接入`GeometricPathValidator`，在`PreciseSearchEngineV2`中串联所有模块。  
**验证**：在完整的412场景测试集上与现有引擎对比，统计路径条数、路径长度分布、新增混合路径数量。

### Iter 5 — 性能优化与参数调优
**目标**：达到性能指标（< 200ms per Rx，候选池生成 < 2s per Tx）。  
**内容**：候选池的多线程并行化，BFGS的批量化，Rx过滤的向量化。  
**验证**：性能剖析，确认瓶颈位置，达到目标吞吐量。

## 十一、完备性保证边界说明

本方案的完备性上界由两个因素共同决定，需要在文档中明确说明（特别是论文撰写时）：

**因素1 - SBR采样密度**：若某条面元序列所经过的所有第一跳面元，对Tx所张的立体角均小于`1/N_rays`（即极小面元或极远面元），则存在漏检风险。通过增大`N_base`可以系统性降低此风险，但无法绝对消除。

**因素2 - FacePVS质量**：若FacePVS中缺少某对面元的可见性关系（因BVH精度或预计算截断），则经过这对面元的路径无法被发现。这是由已有基础设施决定的，本方案不引入新的风险。

**实践中的完备性**：基于上述两点，在`N_base = 10000`的配置下，对典型室内场景（10k面元，路径深度≤6），漏检率预期 < 2%（可通过与穷举搜索对比来量化验证）。这对工程应用和论文中的仿真对比是可接受的，并且优于现有SBR方案（现有方案因接收球依赖，漏检率更难界定和控制）。
