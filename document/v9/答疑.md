## 问题 1：相比 v9 修复前有了哪些变化？

### 大白话版

修之前，这个程序像个"跑得通但算不准"的原型。透射路径可能违反物理定律（光线在玻璃里拐弯后，程序说它到了，但实际上根本没对准接收器），GPU 和 CPU 算出来的遮挡关系不一样，材质丢了自己静悄悄用空气代替，配置写错也不告诉你。

修之后，程序真正开始"算得准"了。每一束光走到哪、打在什么材料上、折射角度对不对、能量对不对，全部有据可查。CPU 和 GPU 不再各说各话，配置文件有错马上报，每条仿真结果都能溯源。

### 严谨版

**56 个文件修改，2425 行新增，1115 行删除**。核心变化：

| 维度 | 修复前 | 修复后 |
|------|--------|--------|
| **Snell 折射** | 旧版 `SnellRefract`：不 clamp cosI，不自动翻转法线，TIR 仅返回零向量 | `SnellRefractV2`：自动法线翻转 + cosI clamp + 结构化 `SnellResult`（含 residual/TIR/入射角/出射角） |
| **透射几何** | `TryBuildLosPath` 只检查可见性，不检查出射方向是否指向 Rx | 方向闭合检查：`Dot(direction, toRx) >= cos(tol)`，违反即拒绝 |
| **GPU 遮挡** | `IsOccludedBatch` 硬编码 `tMax=1e30`，目标点后方物体被误判遮挡 | Per-ray tMax：每条射线使用实际段长度 |
| **材质安全** | `LoadFromCsv` 返回值不检查，缺失静默变真空 | Strict policy：`missing_material_policy=strict` 时中止并报错 |
| **配置解析** | 手写字符串 parser，含尾逗号 bug，不支持 round-trip | nlohmann/json 标准解析，合法 JSON 输出 |
| **配置自检** | 负例全部跳过（"v6: skipped"） | 7 项内联负例自检 |
| **路径去重** | 状态签名仅含 id+深度+预算 | 加入量化坐标/方向/累计长度/节点数 |
| **剪枝控制** | `perExpanderKeepLimit=8, perStateKeepLimit=16` 硬编码 | 全部配置化，`<=0` 表示不截断，`exhaustive_debug_mode` 关闭全部剪枝 |
| **绕射诊断** | PathNode 无绕射诊断字段 | `DiffractionDiagnostics`：keller_residual, s1, s2, edge_t, fermat_warning, visibility |
| **复矢量 EM** | 标量幅度+相位，TE/TM 分量压扁丢失相位 | `ComplexVec3` 全程追踪，反射/透射/绕射均 TE/TM→复系数→重构 |
| **Rx 接收** | Tx 极化矢量作 Rx 响应，无共轭匹配，Rx gain 计算后未用 | Rx 独立极化 + 共轭匹配 `E·conj(h_rx)` + `vr *= sqrt(rxGain)` |
| **宽带信道** | 仅单频点 CIR/PDP | 宽带 CFR H(f) + IFFT 可观测 CIR + 理想 δ CIR + 相干/非相干 PDP |
| **OBJ 导入** | 仅支持 `v//n` 三角面 | v/t/n 三种格式 + 负索引 + 四边形 fan triangulation + 无法线时自动计算 |
| **Wedge 查询** | O(W) 线性扫描 + 硬编码 50m/64 候选 | 均匀网格空间索引 + 配置化 |
| **JSON Parser** | 手写 600+ 行 | nlohmann/json 280 行 |
| **ITU 材料** | 启发式指数 b=-0.1, d=0.5 | ITU-R P.2040-1 参数表（12 种建筑材料） |
| **Scene Preflight** | 无 | `scene_preflight_report.json` 含材质/介质诊断 |

## 问题 2：这些优化对系统级仿真有哪些改进？

### 大白话版

以前你跑一个会议室的仿真，看起来有 200 多条路径，但其中几条"透射穿墙"的路径其实是算错了——光线折射后根本没对准接收器，程序也让它通过了。现在这种错误路径会被直接拒绝，剩下的都是物理上真正能到达的路径。

以前如果你把接收天线转了 90 度（比如手机横过来），程序算出来的信号强度基本不变，因为极化信息在计算过程中被"压扁"了，TE 和 TM 分量之间的微妙相位关系丢了。现在每一束光的 TE 和 TM 分量都独立追踪，到了接收端才做复矢量内积——所以你能真正看到"天线转了 90 度信号掉了 30dB"这种物理上正确的现象。

### 严谨版

| 仿真环节 | 修复前 | 修复后 | 系统级影响 |
|---------|--------|--------|----------|
| **透射路径** | 违反 Snell 仍可闭合 | 方向一致性检查拦截 | NLOS 穿墙路径更可信，PDP 长尾更准 |
| **反射 EM** | 固定空气→材料 | 入射侧+出射侧双介质 | 材料内部反射（如木板内部打到出射面）物理正确 |
| **绕射** | 无诊断 | 完整 Keller/Fermat 诊断 | 可验证绕射路径是否满足 UTD 理论 |
| **极化追踪** | 每步重归一化，分量间相位丢失 | 全程复矢量，TE/TM 复关系保留 | 深衰落、极化失配、交叉极化隔离度可量化 |
| **GPU 遮挡** | 目标点后方误判 | per-ray tMax | GPU 加速不会引入错误遮挡 |
| **配置可靠性** | 错误配置静默退化 | strict 中止 + 自检 | 实验可复现，不会因配置错误得到虚假"好结果" |
| **宽带输出** | 无 | H(f) + IFFT CIR + 相干/非相干 PDP | 可直接对标 VNA/channel sounder 测量数据 |
| **场景诊断** | 无 | preflight + 材质检查 | 场景建模错误在仿真前被拦截 |

### 3.2 关键文件解读

#### `precise_paths.json` — 最重要文件

每条路径包含：
- **path_id**：路径编号
- **delay_s**：传播时延（秒）。除以光速得到等效距离。
- **phase_rad**：到达相位（弧度）。多条路径间的相位差决定相干叠加结果。
- **power_linear**：接收线性功率。**相干 PDP 用 `|Σα|²`，非相干 PDP 用 `Σ|α|²`**。
- **amplitude_real / amplitude_imag**：接收复幅度。这是 `FinalizeAtReceiver` 经过共轭匹配 + Rx gain 后的最终值。
- **polarization_vector / polarization_imag**：到达极化 Jones 矢量。
- **aod_theta_deg / aod_phi_deg**：离开角（发射方向）。
- **aoa_theta_deg / aoa_phi_deg**：到达角。
- **is_los / contains_transmission**：路径类型标记。
- **geometry_nodes**：路径上每个交互节点的详细信息：
  - `interaction_type`：1=Tx, 2=Rx, 4=Reflection, 5=Transmission, 6=Diffraction
  - `medium_in_id / medium_out_id`：交互前后介质 ID
  - `snell_residual / snell_theta_i_rad / snell_theta_t_rad / snell_tir`：Snell 诊断
  - `diffraction_diag`：绕射诊断（keller_residual, s1, s2, edge_t, fermat_endpoint_warning）

#### `search_diagnostics.json` — 搜索行为诊断

```json
{
  "failure_reason_counts": {
    "2": 10,    // NoHit (10次)
    "7": 3,     // TotalInternalReflection (3次透射候选因TIR被拒绝)
    "8": 2      // SnellResidualExceeded (2次残差超限)
  },
  "generated_state_count": 45,
  "deduplicated_state_count": 12
}
```

- `failure_reason_counts` 中的数字对应 `GeometryValidityReason` 枚举值
- 7=TotalInternalReflection, 8=SnellResidualExceeded

#### `cfr_sampled.json` — 频域信道响应

```
freq_hz, H_real, H_imag, magnitude, phase_rad, power_dB
```

- H(f) = Σ α_l · exp(-j2πf·τ_l)
- 可画 |H(f)| vs f 来观察频率选择性衰落

#### `cir_ideal_delta.json` — 理想 δ CIR

- 每个 tap 对应一个时延 bin
- `amplitude_real/imag` 是 bin 内所有路径的 **相干叠加** 结果
- power = |相干和|²

#### `pdp_coherent.json` vs `pdp_incoherent.json`

- **相干 PDP**：P_coh[k] = |Σ_{l in bin k} α_l|² → 体现干涉效果
- **非相干 PDP**：P_incoh[k] = Σ_{l in bin k} |α_l|² → 体现能量到达分布
- 两者差异大 → 说明该 bin 内多径相干性强
- 论文中 RMS delay spread 应该用 **非相干 PDP** 计算

## 问题 4：天线联合建模是否完备？极化信息是否准确？矢量电场相干是否准确？

### 大白话版

**不完备，但基础已经打好。** 现在的情况是：

- **发射端**：天线方向图（增益随角度变化）已经生效。极化矢量（电场振荡方向）从 Tx 天线模型读取，初始化到 `electric_field_world`（世界坐标下的三维复电场矢量）。
- **传播过程**：每一段自由空间传播、每一次反射、每一次透射、每一次绕射，都用复矢量计算。反射时把电场投影到 TE（垂直极化）和 TM（平行极化）两个基上，分别乘 Fresnel 反射系数，再重构回世界坐标。TE 和 TM 之间的复相位关系**全程保留**。
- **接收端**：到达 Rx 的复电场 `E_inc` 与 Rx 天线的复有效长度 `h_rx` 做**共轭内积**：`vr = E_inc · conj(h_rx)`，然后乘以 `sqrt(rxGainLin)`。这就是该路径的最终接收复幅度。

**"完备"还缺什么？** 目前 Rx 天线极化主要从天线模型的 `polarization_vector`（一个固定方向）读取。真正的 Jones 方向图（每个方向的复响应 `E_theta(f,θ,φ), E_phi(f,θ,φ)`）的 CSV 导入接口已预留（`AntennaPattern::QueryPolarization`），但实际使用时通常会 fallback 到标量增益+固定极化。如果你有 HFSS/CST 导出的 Jones pattern CSV，可以填进去。

### 严谨版

**当前状态：矢量 RT 底座完成，Jones 天线方向图导入预留。**

| 环节 | 状态 | 说明 |
|------|------|------|
| Tx 场初始化 | ✅ | 从天线模型读取极化矢量 → 构建 `ComplexVec3 electric_field_world` |
| 自由空间传播 | ✅ | `E ← E · exp(-j·k₀·n·d) · exp(-α·d)` |
| 反射 TE/TM | ✅ | 投影→Fresnel Γ_TE,Γ_TM→出射基重构，复相位保留 |
| 透射 TE/TM | ✅ | 投影→Fresnel T_TE,T_TM→出射基重构 |
| 绕射 soft/hard | ✅ | 投影→D_soft,D_hard→出射基重构 |
| Rx 共轭匹配 | ✅ | `vr = ComplexDot(E_inc, conj(h_rx))` |
| Rx gain 应用 | ✅ | `vr *= sqrt(rxGainLin)` |
| Rx Jones pattern CSV | ⚠️ 预留 | `QueryPolarization` 存在，需要实际 Jones CSV 文件 |
| 天线频率响应 | ❌ 未实现 | 需要天线频响模型或 CSV 多频点数据 |
| 天线群时延 | ❌ 未实现 | |
| AntennaPose 姿态旋转 | ⚠️ 结构体存在 | `AntennaPose{forward,up,right}` 已定义，`BuildRxAntennaModel` 支持 |

**多径相干叠加**：`BuildCIR` 在 `delay_bin_s > 0` 时按 bin 做复数值相加——即 `h[k] = Σ α_l` for paths in bin k。这保证了同一 bin 内的路径能够正确相干增强或抵消。功率则是 `P[k] = |h[k]|²`。对于非相干 PDP，功率是 `P_incoh[k] = Σ |α_l|²`。

## 问题 5：两种主要模式的寻径机制是否完备准确？

### 大白话版

程序有两种找路径的方式，对应两种使用场景：

**Precise 模式（确定性路径搜索）**：像一个细心的人，从发射天线出发，一步一步地找可能的路径——先看有没有镜面反射、有没有透射面、有没有可以绕射的楔边。每走一步都要检查物理上是否成立（反射方向对不对、折射符不符合 Snell 定律、绕射是否满足 Keller 锥条件）。最后把所有可能的路径收集起来，去重，按优先级排序。这种方式**慢但准**，适合做高精度信道分析。

**SBR 模式（射线弹射覆盖）**：像往四面八方发射很多小球，每个小球遇到墙壁就弹射（反射），遇到玻璃就穿过去（透射），能量降到一定程度就放弃。最后看哪些小球经过接收天线附近。这种方式**快但粗**，适合做大范围覆盖仿真，让你知道整个房间里哪里信号强哪里信号弱。

### 严谨版

#### Precise 模式（SearchEngine）

**机制**：
1. Best-first 优先队列搜索
2. 从 Tx 初始状态出发
3. 每步尝试：反射镜像法、透射 Snell 折射、绕射 Fermat+Keller
4. 闭合条件：当前点与 Rx 可见，且（若已有约束方向）方向对齐
5. 状态/路径签名去重
6. 候选按复合评分排序后截断（可配置）

**完备性**：
- 反射：✅ 镜像法 + BVH 空间过滤
- 透射：✅ SnellRefractV2 + 方向闭合检查 + TIR 拒绝 + Snell 残差检查
- 绕射：✅ Fermat 黄金分割搜索 + Keller 锥验证 + UTD 系数
- LOS：✅ 可见性检查

**不完整处**：
- 透射面选择策略是沿 `current→Rx` 找面再算折射，而非纯沿 `current_direction` 传播——单界面场景影响小，多层复杂场景可能漏非直达透射路径
- 绕射路径在当前 test 场景中难以触发（Keller 锥条件严格），但生产场景（如 meeting_412）中大量生成
- 漫散射未实现

#### SBR 模式（SbrEngine）

**机制**：
1. Fibonacci 球面生成射线
2. 沿射线方向追踪：反射（镜像法）、透射（Fresnel 功率概率二选一，不分裂）
3. RxHashGrid 判断射线是否穿过接收球
4. CPU legacy 模式 + OpenMP 并行

**完备性**：
- 反射：✅
- 透射：⚠️ Monte Carlo 近似（概率选择反射/透射，不分裂能量）
- 绕射：⚠️ 简化模型（边附近触发 + 4 方向采样），与 precise UTD 模型不同
- TIR：✅ v9 已修复（TIR 退化为反射）

**适用边界**：
- SBR 结果适合做**覆盖统计**（Rx 处功率、覆盖热力图），不适合做**逐路径高保真分析**
- SBR 绕射功率与 precise UTD 绕射功率不可直接比较
- 论文中应明确区分两种模式的适用范围
