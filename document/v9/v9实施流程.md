# v9 实施流程

本文档用于指导使用 `claude code cli + deepseekv4pro[1m]` 单 agent 按顺序完成 v9 全量全流程闭环优化开发。

v9 的目标不是继续扩展新研究方向，而是在现有 RT 主链路基础上完成代码层面的真实修复、真实接入、真实验证，使系统达到论文实验可用标准。

## 1. 总原则

### 1.1 v9 目标

v9 只聚焦现阶段 RT 已有功能的全面优化升级：

1. precise 几何路径、介质语义和 Snell/Fresnel 物理闭环。
2. 复矢量 EM、Jones 极化、天线姿态和接收端极化匹配。
3. 宽带 CFR/CIR/PDP 和有限带宽可观测 CIR。
4. 搜索候选控制、路径去重、path validation 和 diagnostics。
5. CPU/GPU 查询语义一致、capability fallback 和 backend consistency。
6. 绕射 Fermat/Keller/UTD diagnostics。
7. 输入 preflight、材质绑定、ITU-R P.2040 电参模型、配置快照和缓存安全。
8. SBR coverage diagnostics、ray count convergence 和大规模覆盖可信度。
9. v9 validation suite、validation report、论文图表数据和验收闭环。

### 1.2 v9 禁止范围

v9 不做以下内容：

1. 完整 ISAC 指标体系。
2. 通信-感知联合优化。
3. 完整雷达目标检测器。
4. RCS 目标散射系统。
5. 距离-多普勒处理。
6. CRLB 或定位误差下界。
7. 完整漫散射模型。
8. 完整 ray tube/beam tracing 重构。
9. 完整 mixed R/T/D global nonlinear solver。
10. 新增 `precise` 和 `sbr` 之外的用户可见仿真模式。

### 1.3 单 agent 执行规则

因为计划使用单 agent 顺序完成，因此必须遵守：

1. 每次只推进一个阶段，不并行混改多个主线。
2. 每个阶段先审计现状，再修改代码，最后运行自测。
3. 不允许只新增未接入的代码。
4. 不允许只写 TODO 或注释冒充完成。
5. 不允许修改文档宣称完成但代码未接入。
6. 不允许把 v10 内容混入 v9 主线。
7. 每个修复必须从主生产链或 validation runner 可达。
8. 每个阶段完成后必须记录：修改文件、修复内容、测试命令、测试结果、残余风险。
9. 如果测试失败，不得进入下一阶段，除非明确记录失败原因并修复。
10. CPU-only 必须完整可运行；GPU 只作为 optional acceleration backend。

## 2. 总体执行顺序

建议严格按以下顺序执行：

1. 阶段 0：建立基线和工作约束。
2. 阶段 1：P0-H validation runner 最小闭环。
3. 阶段 2：P0-A 几何路径、介质和 Snell 闭环。
4. 阶段 3：P1-E 搜索候选、去重和 top-K 配置化。
5. 阶段 4：P0-B 复矢量 EM、Rx Jones 和天线姿态。
6. 阶段 5：P1-C 宽带 CFR/CIR/PDP 主链路接入。
7. 阶段 6：P1-F CPU/GPU 查询 fallback 和一致性。
8. 阶段 7：P1-D 绕射 diagnostics。
9. 阶段 8：P2-G 输入 preflight、材质和缓存。
10. 阶段 9：P2-J SBR coverage diagnostics 和收敛性。
11. 阶段 10：论文图表数据和 validation summary。
12. 阶段 11：最终全量验收。

## 5. 阶段 2：P0-A 几何路径、介质和 Snell 闭环

### 5.1 目标

让透射路径在几何和介质语义上物理成立，避免非法 Snell、TIR 透射和介质不连续进入 EM。

### 5.2 当前重点不足

1. `SnellRefractV2` 已存在，但 `TransmissionExpander.cpp` 仍调用旧 `SnellRefract`。
2. `SnellResult` 的 residual、TIR 和角度诊断没有进入主链路。
3. `PathNode` 没有完整 Snell residual 字段。
4. 透射路径缺少系统化 geometry validation 导出。

### 5.3 实施内容

1. 修改 `core/search/TransmissionExpander.cpp`：
   - 全部改用 `SnellRefractV2`。
   - `SnellResult.valid=false` 时拒绝候选。
   - `total_internal_reflection=true` 时拒绝透射候选。
   - `residual > config.path_search.snell_residual_tol` 时拒绝候选。
   - 使用 `SnellResult.direction` 作为 `nextState.current_direction`。
2. 修改 `core/path/PathNode.h`，增加：

```cpp
double snell_residual = 0.0;
double incidence_angle_rad = 0.0;
double transmission_angle_rad = 0.0;
bool total_internal_reflection = false;
```

3. Transmission `PathNode` 填入：
   - `medium_in_id`
   - `medium_out_id`
   - `entered_from_front_side`
   - `incident_direction`
   - `direction`
   - `surface_normal`
   - `snell_residual`
   - `incidence_angle_rad`
   - `transmission_angle_rad`
4. 检查 `ResolveMediumTransition.cpp`：
   - 当前介质必须等于入射侧介质。
   - medium mismatch 必须拒绝候选。
5. 修改 `ExportPaths.cpp` 或新增 geometry validation export：
   - 导出每个 geometry node 的 medium 和 Snell diagnostics。
6. 在 validation 中增加 `geometry` suite。

### 5.4 自测命令

```text
python validation/run_all.py --suite geometry --backend cpu
python validation/run_all.py --suite analytic --backend cpu
```

### 5.5 必测 case

1. air -> glass，45 度入射，折射角误差小于 `1e-4 rad`。
2. glass -> air，大角度入射，必须触发 TIR，不能生成透射路径。
3. air -> concrete -> air 双面墙，medium sequence 连续。
4. 法向反转或介质不匹配 case，必须失败且报告原因。

### 5.6 验收标准

1. `TransmissionExpander.cpp` 不再调用旧 `SnellRefract`。
2. TIR 不生成非法透射路径。
3. Snell residual 超限路径被拒绝。
4. 透射路径 medium continuity 通过。
5. `geometry_validation.json` 或路径导出中包含 Snell residual 和 medium sequence。
6. geometry suite 通过。

## 7. 阶段 4：P0-B 复矢量 EM、Rx Jones 和天线姿态

### 7.1 目标

让 EM 主链路以复矢量电场传播，并在接收端使用真正的 Rx Jones response 做极化匹配。

### 7.2 当前重点不足

1. `ComplexVec3` 和 `electric_field_world` 已存在。
2. `FinalizeAtReceiver.cpp` 当前使用传播场自身极化作为 Rx 匹配方向，不是真正 Rx 天线 Jones response。
3. 接收端复内积需要明确是否使用共轭匹配。

### 7.3 实施内容

1. 修改 `core/em/FinalizeAtReceiver.cpp`：
   - 使用 `BuildRxAntennaModel` 构造 Rx 天线。
   - 根据最后一段入射方向转换到 Rx local spherical。
   - 查询 Rx pattern gain 和 polarization response。
   - 将 Rx local theta/phi Jones response 转回 world basis。
2. 实现接收极化匹配：

```text
vr = E_inc · conj(h_rx)
```

3. 如 `Complex` 缺少共轭函数，补充 `Conj`。
4. `result.amplitude_real/amplitude_imag` 必须来自接收复标量 `vr`。
5. 保持旧字段派生兼容，但不要让旧字段反向决定复矢量主状态。
6. 检查反射/透射后的 TE/TM 基重构，保证电场横向于出射方向。

### 7.4 自测命令

```text
python validation/run_all.py --suite em --backend cpu
python validation/run_all.py --suite smoke --backend cpu
```

### 7.5 必测 case

1. LOS 同极化：接收功率最大。
2. LOS 交叉极化：Rx 旋转 90 度后接收功率显著下降。
3. 圆极化同手性/反手性差异可验证。
4. two-ray 等幅反相路径产生相干抵消。
5. 天线姿态改变时接收功率和极化匹配变化可复现。

### 7.6 验收标准

1. `FinalizeAtReceiver.cpp` 不再使用传播场自身极化作为 Rx response。
2. Rx Jones response 和姿态真实参与接收端计算。
3. co/cross 极化验证通过。
4. `vector_em_validation.json` 或 `antenna_pose_validation.json` 存在。
5. EM suite 通过。

2. 修改 `RtRealChainRunner.cpp`：

   - `fixed_gain` 调用 `BuildBroadbandCFR_FixedGain`。
   - `frequency_sweep_em` 如果未完整实现，应明确 unsupported 或 fallback 并写 warning，不能假装完成。
3. 修正 `BuildBroadbandCFR_FixedGain`：

## 9. 阶段 6：P1-F CPU/GPU 查询 Fallback 和一致性

### 9.1 目标

保证 GPU 只是加速，不改变 CPU reference 语义。

### 9.2 当前重点不足

1. GPU `QueryFaceHitsInRange` 只返回 closest hit，不是真 all/range hits。
2. SceneQuery batch 接口没有充分检查 capability。
3. 缺少 backend consistency report。

### 9.3 实施内容

1. 修改 `SceneQuery`：
   - 需要 all/range hits 时，如果 `SupportsAllHits=false`，必须 CPU fallback。
   - 需要 object filter 时，如果 `SupportsObjectFilter=false`，必须 CPU fallback。
   - GPU 不支持 ignored object/face 语义时，必须 CPU fallback。
2. `QueryRxHitsBatch` 不能在 CPU fallback 下静默返回空结果。
3. 保持 GPU batch occlusion per-ray `tMax`。
4. 增加 backend consistency suite。
5. 输出：

```text
validation/reports/backend_consistency.json
```

### 9.4 自测命令

```text
python validation/run_all.py --suite backend --backend cpu
python validation/run_all.py --suite backend --backend cpu,gpu
```

### 9.5 必测 case

1. closest hit CPU/GPU 对照。
2. range hits/all hits thin slab 对照。
3. occlusion segment tMax 对照。
4. ignored face/object 对照。
5. boundary ambiguous 分类。

### 9.6 验收标准

1. CPU-only 环境完整运行。
2. GPU capability 不足时自动 fallback，不改变语义。
3. `backend_consistency.json` 存在。
4. GPU 可用时 CPU/GPU 差异在容差内或标记为 `boundary_ambiguous`。

## 11. 阶段 8：P2-G 输入 Preflight、材质和缓存

### 11.1 目标

让错误输入在仿真前暴露，保证材质频变电参可追溯。

### 11.2 当前重点不足

1. `missing_material_policy=strict` 存在，但材质缺失仍 warning + vacuum fallback。
2. 没有完整 scene preflight report。
3. ITU-R P.2040 模型目前偏启发式，不是材料参数表化。

### 11.3 实施内容

2. 输出：

```text
validation/reports/scene_preflight_report.json
```

3. strict 策略：
   - 可传播 face 缺 surface/front/back 绑定时中止。

   - transmission face 的 front/back medium 不完整时中止。

7. 公式明确频率单位，例如：

```text
epsilon_r = a * f_GHz^b
sigma = c * f_GHz^d
```
