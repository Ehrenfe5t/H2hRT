# v11 开发文档

更新日期：2026-06-27  
定位：在 `v11基线文档.md` 已冻结的 P2P 全链路基础上，进入面向论文实验的 v11 正式开发。本文只描述开发重点、阶段路线和验证闭环，不改变当前基线结论。

当前基线主链为：

`配置读取 -> OBJ/材质导入 -> 场景预处理 -> SceneQuery -> SBR P2P 寻径 -> GeometricPath -> SolveSinglePathEM -> CIR/PDP/APS/路径导出`

## 1. 论文算法要求的核心

`document/论文相关/大论文.md` 的第三章不是单纯要求一个 RT 程序能跑通，而是要求用自研 RT 引擎证明“天线精细化建模”对信道指标有可量化影响。

论文要求可以压缩成五句话：

1. 几何寻径要生成与天线无关的多径集合，即 `GeometricPath` 只描述空间路径、交互节点和材料/面元/楔边信息。
2. 电磁计算层要把天线作为可替换模块接入，对同一组几何路径施加不同 Tx/Rx 增益方向图、Jones 极化和姿态响应。
3. 路径传播中要保留复振幅、相位和极化演化，反射/透射使用 TE/TM 分解，绕射使用 soft/hard 或 UTD 相关极化分解。
4. 输出必须支撑论文实验图表：路径级复振幅、delay、AoD/AoA、co/cross-pol、XPR、CIR、PDP、APS、MEG、K 因子、RMS delay spread。
5. 需要围绕 E1/E2/E3/C1-C3 做控制变量实验；coverage 热图 E4 可以后置。

论文实验对应关系：

| 实验 | 目的 | v11 是否应优先支持 |
|---|---|---|
| E1 天线模型对比 | ideal/dipole/patch/horn 对 PDP/APS/XPR/总功率的影响 | 必须优先 |
| E2 姿态扫描 | Rx yaw 改变接收功率、APS 中心、XPR | 必须优先 |
| E3 极化模型对比 | Fixed/Jones 极化模型对 XPR 和 CIR 幅相的影响 | 必须优先 |
| E4 覆盖热图 | 大规模 Rx coverage 下天线影响 | 后置 |
| C1 射线数收敛 | 50k/100k/200k/500k 对功率、PDP、XPR 的影响 | 必须优先 |
| C2 绕射敏感性 | 分析绕射路径与绕射功率贡献 | 需要适配当前独立 Tx-D-Rx 绕射机制 |
| C3 接收球半径敏感性 | 验证 `rx_sphere_radius_m` 对路径命中和结果稳定性的影响 | 必须优先 |

## 2. 当前代码已经具备的能力

### 2.1 P2P 全链路已经跑通

当前基线已经具备少量 Tx/Rx P2P 全流程能力：

- `config.json`：meeting 基线。
- `config412.json`：412 基线。
- `V11UserConfigAdapter.cpp`：读取 v11 简化配置，生成 P2P Tx-Rx 任务。
- `RtPipeline.cpp`：串联场景导入、预处理、SBR P2P、EM 和导出。
- `SbrEngine::RunPointToPoint()`：执行 P2P SBR 几何寻径。
- `RunA1RealChain()`：将几何路径逐条送入 EM。

这部分当前不应大改。v11 开发应尽量在“EM 指标补全、天线验证、实验自动化”上做增量开发，避免破坏已经与独立 SBR 对齐的寻径基线。

### 2.2 天线三属性已有雏形

当前代码已经具备：

- 方向图 CSV 加载：`AntennaPattern::LoadCsv()`。
- Jones 极化 CSV 加载：`AntennaPattern::LoadPolarizationCsv()`。
- 天线局部角查询：`WorldToAntennaSpherical()`。
- Ludwig-3 球基矢量转世界坐标：`AntennaSphericalBasisToWorld()`。
- Tx/Rx 独立配置：`tx_antenna`、`rx_antenna`、以及 v11 `antenna_file` 中每个 Tx/Rx 的 pattern/polarization/forward/up。
- per-Rx 天线覆盖：`BuildRxAntennaModelWithOverride()`。

`configs/antennas` 下已有论文实验所需的方向图数据：

- `dipole_gain.csv`
- `dipole_jones.csv`
- `patch_gain.csv`
- `patch_jones.csv`
- `horn_gain.csv`
- `horn_jones.csv`

### 2.3 EM 复矢量链路已有主干

当前 EM 主链为：

- `InitializeTxField()`：根据几何路径首段方向查询 Tx 天线增益/Jones 极化，构造世界坐标复电场。
- `ApplyFreeSpaceSegment()`：对复矢量电场施加传播相位和介质衰减。
- `ApplyReflectionInteraction()`：反射节点做 TE/TM 分解、Fresnel 系数、复矢量重构。
- `ApplyTransmissionInteraction()`：透射节点做 TE/TM 透射、介质状态更新和衰减常数更新。
- `ApplyDiffractionInteraction()`：绕射节点做 soft/hard 分解、UTD 系数、复矢量重构。
- `FinalizeAtReceiver()`：接收端做 Rx 天线方向图/Jones 响应、共轭匹配、co/cross-pol 分解和 XPR 计算。

这说明 v11 的理论方向是对的：不是标量功率链，而是已经有复矢量 EM 链的基础。

### 2.4 路径级导出已经包含论文所需的一部分字段

`ExportPaths.cpp` 当前已经导出：

- `delay_s`
- `phase_rad`
- `power_linear`
- `amplitude_real/amplitude_imag` 目前 JSON 未直接导出，但 `precise_paths.json` 中已有 power/phase，CIR 内部使用复振幅。
- `polarization_vector`
- `polarization_imag`
- `aod_theta_deg/aod_phi_deg`
- `aoa_theta_deg/aoa_phi_deg`
- `co_pol_power_linear`
- `cross_pol_power_linear`
- `xpr_dB`
- `geometry_nodes`
- face/wedge/object/medium/direction/normal/residual 等几何信息

这已经能支撑一部分路径级人工检查，但还不足以直接支撑论文图表。

## 3. 当前与论文要求的主要差距

### 3.1 指标聚合和导出不足

结构体中已经预留了很多论文指标，但实现并未完全落地：

- `APSResult` 有二维网格字段，但 `BuildAPS.cpp` 目前只输出一维 `aoa_theta_deg` entry。
- `XPRStatistics` 已定义，但没有构建函数，没有写入 `EMAggregateResult.xpr_stats`。
- `meg_linear/meg_dB` 已定义，但没有计算。
- `ChannelStatistics` 里有 `rms_delay_spread_s`、`k_factor_dB`、`effective_path_count` 等字段，但 `BuildChannelStatistics.cpp` 当前只计算基础路径数、总功率、最强路径功率、平均 delay、透射路径数。
- `ExportChannel.cpp` 当前只输出 `channel_summary.json`，没有完整导出 `cir.json/csv`、`pdp.json/csv`、`aps_2d_grid.json/csv`、`xpr_stats.json`、`channel_stats.json`。

结论：当前能算路径 EM，但还不能完整产出论文实验所需的数据产品。

### 3.2 Tx 发射功率语义需要补齐

v11 配置和天线 JSON 中有 `power_dBm`，`RtPipeline.cpp` 会写入 `config.sbr.tx_power_dBm`，SBR 几何统计也会用它算一个几何 proxy power。

但 EM 主链 `InitializeTxField()` 当前以单位场幅度初始化，未显式把 `tx_power_dBm` 转成发射功率幅度尺度注入 EM。

这意味着：

- 当前 `power_linear` 更接近“单位发射功率归一化接收功率”。
- 同一 Tx 功率下做天线相对对比问题不大。
- 若论文需要绝对接收功率、Friis 绝对校验、不同 Tx 功率对比，则必须补齐 Tx power 进入 EM 的定义。

v11 应明确并实现其中一种语义：

1. 所有 EM 输出均为相对于 0 dBm / 1 mW 的归一化线性功率。
2. 或者将 `power_dBm` 转成 W 或 mW，进入初始场幅度，使 `power_linear` 可转换为绝对接收功率。

建议采用第 2 种，并在导出中同时给出：

- `power_linear`
- `power_dBm`
- `tx_power_dBm`
- `received_power_dBm`

### 3.3 矢量接收后的路径相位需要统一

在 `FinalizeAtReceiver()` 的复矢量分支中，最终接收复电压为：

`vr = E_inc · h_rx*`

随后写入：

- `result.amplitude_real = vr.re`
- `result.amplitude_imag = vr.im`
- `result.power_linear = |vr|^2`

但 `result.phase_rad` 在函数开头先从 `field.phase_rad` 赋值，复矢量接收后没有显式更新为 `atan2(vr.im, vr.re)`。

影响：

- `BuildCIR()` 使用 `amplitude_real/amplitude_imag`，因此相干叠加本身不一定受影响。
- 但 `precise_paths.json/csv` 中导出的 `phase_rad` 可能不是最终接收电压相位。
- 论文 E3 要比较固定极化和 Jones 极化下路径级复振幅相位差，因此必须修正。

建议：复矢量分支中在写入 `amplitude_real/imag` 后同步：

`result.phase_rad = atan2(result.amplitude_imag, result.amplitude_real)`

### 3.4 天线方向图周期插值需要重点复核

`AntennaPattern::QueryGainDBi()` 和 `QueryPolarization()` 目前对 `phi` 做了 `[0, 360)` 归一化，并使用 `(pi + 1) % Nphi` 处理周期。

但当 `phiDegVal` 落在最后一个采样点到 360° 的区间时，例如网格最后一点为 355°、查询为 359°，当前查找循环可能找不到正确 `pi`，默认 `pi=0`，导致插值落到错误区间。

这会影响：

- 天线方向图主瓣/旁瓣查询。
- Jones 极化查询。
- 姿态扫描中接近 0/360° 方位的结果。

建议 v11 优先修正：

- 若 `phiDegVal >= phiDeg.back()`，应使用最后一个 phi 样点和第一个 phi 样点加 360° 做周期插值。
- 同时为 `QueryGainDBi()` 和 `QueryPolarization()` 加最小单元测试或脚本验证。

### 3.5 APS/MEG 还缺“天线无关 vs 天线观测后”的区分

论文强调传播信道与天线响应解耦。为了讲清楚“天线是角度-极化滤波器”，最好能同时输出两类角谱：

- 几何/传播信道 APS：只根据路径几何和传播损耗，不含 Rx 天线滤波。
- 可观测 APS：经过 Tx/Rx 天线方向图和极化匹配后的路径功率。

当前 `EMPathResult.power_linear` 已经是可观测结果，适合输出“观测 APS”。但若要解释 MEG，最好保留一个“天线无关传播 APS”或至少在实验脚本中用同一几何路径集构造参考。

短期可先做可观测 APS；论文分析时用 ideal 天线作为参考传播 APS。

### 3.6 当前 C2 绕射敏感性表述需适配现实现

论文草稿里写到扫描 `diffraction_rays_per_event = 2/4/8/16`。但当前 v11 P2P 绕射已经简化为独立 `Tx-D-Rx`，不是在 R/T 路径中对 Keller cone 继续发射多条绕射采样射线。

因此 v11 如果不改变当前寻径机制，C2 应改成：

- 绕射开启/关闭对比。
- `max_diffraction=0/1` 对比。
- 绕射棱边判据、可见性和 Keller residual 分布检查。
- 绕射路径功率占总接收功率比例。

不要在当前 P2P 主链中引入 R/T-D 混合或 Keller cone 多采样，否则会破坏此前敲定的寻径简化原则。

## 4. v11 开发总原则

v11 不是“大规模 coverage 版本”，也不是“GPU 加速版本”。v11 应先成为论文第三章可用的 P2P 精细信道仿真版本。

开发原则：

1. 不破坏当前 SBR P2P 几何寻径基线。
2. 几何路径保持天线无关，天线影响只进入 EM 层。
3. 优先补齐论文实验数据产品，而不是扩展更多寻径模式。
4. 所有理论修正必须有可运行的最小验证。
5. 所有实验输出必须能被 Python 脚本直接读取画图。
6. coverage、RxHashGrid、GPU、宽频带扫频继续后置。

## 5. v11 推荐开发阶段

### 阶段 V11.1：EM 语义校准与最小理论修正

目标：先把“路径级 EM 结果”定义清楚，避免后续所有实验建立在不清晰的功率/相位语义上。

开发项：

- 修正矢量接收后的 `phase_rad`，使其等于最终接收复电压相位。
- 明确并实现 `tx_power_dBm` 到 EM 初始场幅度/功率的映射。
- 在 `EMPathResult` 和导出中增加绝对/相对功率字段：
  - `tx_power_dBm`
  - `power_linear`
  - `power_dBm`
  - `received_power_dBm`
- 检查 `free_space_loss_db` 与 Friis 公式的一致性。
- 对 LOS + ideal 天线构造最小解析校验：
  - 无墙、无交互、单 LOS。
  - 对比 `Pr/Pt = Gt Gr (λ/4πd)^2`。

验收：

- Release 编译通过。
- LOS 解析场景中 ideal 天线功率误差可解释。
- `phase_rad` 与 `atan2(amplitude_imag, amplitude_real)` 一致。
- 现有 meeting/412 几何路径数不变。

### 阶段 V11.2：天线方向图与 Jones 查询可信化

目标：确保天线三属性查询本身可信，避免实验结果是插值或坐标系错误造成的。

开发项：

- 修正 `AntennaPattern` 的 phi 周期插值。
- 增加方向图查询验证脚本或小测试：
  - 查询 0°、5°、355°、359°、360° 附近的 gain/Jones 连续性。
  - 验证 `|E_theta|^2 + |E_phi|^2` 归一化。
- 明确 forward/up/right 约定：
  - `right = forward × up`。
  - `theta=0` 为 forward。
  - `phi=0` 为 right。
  - `phi=90°` 指向 up。
- 检查 `BuildIdealAntennaModel()` 与 `BuildAntennaFromBlock()` 的 right 方向是否一致。
- 为 ideal/dipole/patch/horn 建立最小 antenna.json 模板。

验收：

- 同一方向图文件在 0/360° 边界无跳变。
- ideal 天线不随姿态改变功率。
- patch/horn 在主瓣和背瓣方向查询结果符合方向图预期。

### 阶段 V11.3：论文指标聚合与导出补全

目标：让一次 P2P 仿真直接产出论文画图所需数据。

开发项：

- 修改 `BuildAggregateResult()`，让其能够使用 `AppConfig` 或显式参数。
- `BuildPDP()` 使用 profile/config 中的 delay bin，输出：
  - per-path PDP
  - binned coherent PDP
  - binned incoherent PDP
- `BuildAPS()` 扩展为二维 AoA theta-phi 网格：
  - 使用 `em_solver.aps_theta_bins`
  - 使用 `em_solver.aps_phi_bins`
  - 输出线性功率网格和 dB 网格
- 新增 `BuildXPRStatistics()`：
  - mean/median/P10/P90/min/max
  - power-weighted mean
  - CDF 所需数组
- 新增或补全 `ComputeMEG()`：
  - 短期以观测 APS 和 Rx gain 计算相对 MEG。
  - 长期可拆分传播 APS 与可观测 APS。
- 补全 `BuildChannelStatistics()`：
  - RMS delay spread
  - K factor
  - effective path count
  - strongest path delay
  - total power dB/dBm
- 扩展 `ExportChannel()`，至少导出：
  - `cir.json/csv`
  - `pdp.json/csv`
  - `aps_2d_grid.json/csv`
  - `xpr_stats.json`
  - `channel_stats.json`
  - `meg.json`

验收：

- E1/E2/E3 所需字段不需要再从 `precise_paths.json` 手工二次统计。
- Python 画图脚本能直接读取 channel 输出。
- `channel_summary.json` 保留，但不再是唯一信道输出。

### 阶段 V11.4：实验配置与批处理体系

目标：让论文实验可重复、可批量运行、可对比。

开发项：

- 新建实验配置目录，例如：
  - `configs/v11_experiments/e1_antenna_model/`
  - `configs/v11_experiments/e2_pose_yaw/`
  - `configs/v11_experiments/e3_polarization/`
  - `configs/v11_experiments/c1_ray_convergence/`
  - `configs/v11_experiments/c3_rx_radius/`
- 每个实验配置都使用 v11 简化 config + antenna.json。
- E1：
  - ideal / dipole / patch / horn。
  - 同一场景、同一 Tx/Rx、同一 SBR 参数。
- E2：
  - 固定 Tx。
  - Rx yaw = 0/30/60/90/120/150/180°。
- E3：
  - Fixed-Fixed：pattern 可开，polarization_file 为空。
  - Jones-Fixed：Tx 使用 jones，Rx 固定极化。
  - Fixed-Jones：Tx 固定极化，Rx 使用 jones。
  - Jones-Jones：Tx/Rx 都使用 jones。
- C1：
  - ray_count = 50k/100k/200k/500k。
- C3：
  - rx_sphere_radius_m = 0.03/0.06/0.10/0.20 m。
- 增加批处理脚本：
  - 逐个运行配置。
  - 汇总输出路径。
  - 提取关键指标为 CSV。

验收：

- 一条命令可以跑完 E1 或 E2。
- 每个 run_id 可追溯到配置快照。
- 输出指标表可直接进入论文绘图脚本。

### 阶段 V11.5：论文图表脚本与结果解释闭环

目标：把仿真结果转成论文所需图表。

开发项：

- 新建 Python 分析脚本目录，例如 `analysis/v11/`。
- 图表脚本：
  - PDP 叠加图。
  - APS 二维热图。
  - XPR 直方图/CDF。
  - 接收功率 vs yaw。
  - XPR mean vs yaw。
  - ray_count 收敛曲线。
  - rx_sphere_radius 敏感性曲线。
- 汇总脚本：
  - 读取多个 run 的 `channel_stats.json`。
  - 生成论文表格 CSV。
  - 标记 run_id、天线类型、姿态、射线数、接收半径。

验收：

- E1/E2/E3 至少各输出一组论文可用图。
- 图表和数据可由 run_id 追溯到配置。
- 结果变化能用天线方向图、极化或姿态解释，而不是只能说“数值变了”。

### 阶段 V11.6：几何敏感性与独立 SBR 对照

目标：保证论文中几何寻径部分不是黑箱。

开发项：

- C1 射线数收敛：
  - 对比路径数、总功率、PDP 主峰、XPR 均值。
- C2 绕射贡献：
  - 比较 `max_diffraction=0/1`。
  - 输出绕射路径数、绕射功率占比、Keller residual 分布。
  - 当前不引入 R/T-D 混合。
- C3 接收球半径：
  - 对比路径数、LOS 是否稳定、主径功率、总功率。
- 对关键配置继续与独立 SBR 输出对照：
  - 几何路径数。
  - 路径序列。
  - face/wedge id。
  - total_length。

验收：

- 可以给出 200k 或 500k 射线数是否足够的证据。
- 可以解释当前简化绕射在室内场景中的贡献比例。
- 可以给出接收球半径选择的依据。

### 阶段 V11.7：coverage 后置开发

目标：支持论文 E4 或后续大规模场景覆盖。

前置条件：

- P2P EM 指标已经可信。
- 天线模型已经验证。
- 输出指标体系已经完善。

后续开发方向：

- Rx 网格自动生成。
- RxHashGrid 加速多 Rx 命中。
- 几何路径即时送入 EM。
- EM 对每个 Rx 在线累加 coverage 结果。
- 默认不保存全量路径，只保存 TopN 或调试路径。

注意：coverage 不应在 v11 早期抢优先级，否则会把问题从“理论正确性”转移到“内存和效率工程”，不利于论文第三章先闭环。

## 6. 近期最推荐的开发顺序

最合理的顺序是：

1. 修正 EM 相位和 Tx power 语义。
2. 修正天线方向图 phi 周期插值。
3. 补齐 `BuildChannelStatistics`、XPR stats、2D APS、MEG。
4. 扩展 `ExportChannel`，导出论文数据产品。
5. 建 E1/E2/E3 的配置模板和批处理脚本。
6. 做 ideal LOS / Friis / antenna query 的最小验证。
7. 跑 E1，确认四种天线模型确实产生可解释差异。
8. 跑 E2，确认姿态扫描曲线合理。
9. 跑 E3，确认 Jones 极化对 XPR/CIR 有可解释影响。
10. 再做 C1/C2/C3 敏感性实验。

## 7. 当前不建议优先做的事情

以下内容虽然有价值，但不应作为 v11 初期重点：

- GPU/OptiX 加速。
- coverage 大规模 Rx。
- R/T-D 混合路径。
- 宽频带重新寻径。
- 复杂 GUI 或可视化前端。
- 大规模代码重构。

理由很简单：论文第三章首先需要可信地证明天线三属性对 P2P 精确信道指标的影响。只要 P2P 结果、指标导出和实验闭环可靠，当前阶段就足够支撑论文主体。

## 8. v11 完成判据

v11 可以认为达到论文实验可用状态，当且仅当：

- meeting/412 基线仍能通过 Release 回归。
- SBR P2P 几何路径与独立 SBR 基线仍基本一致。
- ideal LOS 场景能通过 Friis 解析校验。
- patch/horn/dipole 的方向图查询通过边界和主瓣验证。
- P2P 输出包含路径级 delay、复振幅、功率、AoD/AoA、co/cross-pol、XPR。
- channel 输出包含 CIR、PDP、2D APS、XPR stats、MEG、RMS delay spread、K factor。
- E1/E2/E3/C1/C3 能批量运行并自动汇总。
- 论文图表脚本能直接消费输出文件。
- 文档中明确当前绕射是独立 `Tx-D-Rx`，不宣称支持 R/T-D 混合。

## 9. 一句话结论

当前 H2hRT v11 已经具备“少量 P2P 几何寻径 + 复矢量 EM”的基础，接下来最该做的不是继续扩寻径或上 GPU，而是把天线三属性耦合、路径级复电磁结果、CIR/PDP/APS/XPR/MEG 指标导出和论文实验批处理彻底打通。

## 10. 给新对话 AI 使用的 v11 全流程开发指导话术

本节不是普通开发计划，而是给后续新对话 AI 接手项目时使用的“工作话术 + 执行手册”。目标是避免新对话在不了解现状时凭记忆、凭直觉、凭局部文件直接改代码。

### 10.1 接手时第一句话应该明确的项目状态

新对话 AI 接手时，应先把当前项目状态压缩成下面这段话，再开始行动：

> 当前 H2hRT v11 的主线目标不是重新设计 RT，也不是做 coverage/GPU，而是在已经跑通的 P2P SBR 几何寻径基础上，补齐论文第三章需要的天线精细建模、复矢量 EM、CIR/PDP/APS/XPR/MEG 指标输出和实验闭环。开发时必须保护当前 meeting/412 基线，任何改动都要确认不破坏 `配置读取 -> 场景导入 -> 预处理 -> SceneQuery -> SBR P2P -> GeometricPath -> SolveSinglePathEM -> 导出` 这条主链。

如果新对话 AI 无法确认这句话中的任何一个环节，就不应该直接修改代码，而应该先读代码和文档。

### 10.2 每次开发前必须执行的理解流程

每次进入一个具体任务前，先按顺序完成四步：

1. 明确目标：这次到底是修理论错误、补输出字段、做实验配置、还是整理代码。
2. 定位链路：找出这个目标在主链中的位置，是配置层、场景层、寻径层、EM 层、统计层，还是导出层。
3. 查证现状：阅读对应代码，不允许只凭文档或记忆判断。
4. 定义验收：修改前先说清楚“完成后结果应该是什么”和“用什么命令或文件证明成功”。

推荐话术：

> 我先不急着改代码。这个任务属于 EM/天线/导出/配置中的哪一层？它的输入是什么，输出是什么，谁消费这个输出？我先从调用链往下读到具体函数，再决定最小修改点和回归方式。

### 10.3 开发全过程必须保持的注意事项

v11 开发中时时刻刻要注意：

- 不要破坏几何寻径基线。SBR P2P 已经和独立 SBR 对齐，除非任务明确要求，否则不要改 `SbrEngine` 的核心寻径机制。
- 不要把天线影响混入几何寻径。几何路径只负责空间路径、交互点、面元、楔边、材质和长度；天线方向图、极化、姿态只在 EM 层生效。
- 不要把 coverage 问题提前引入 P2P 主线。大规模 Rx、RxHashGrid、路径流式 EM 是后续任务。
- 不要只看输出路径数判断对错。路径数只是几何正确性的一部分，EM 还要检查复振幅、相位、功率、AoA/AoD、XPR、PDP/APS。
- 不要为了“看起来完整”引入未经验证的大公式。涉及 Fresnel、UTD、Jones、MEG、APS、K 因子时，先确认定义，再对应代码。
- 不要在没有回归的情况下整理代码。代码清理必须证明 meeting/412 基线仍能跑通，且关键输出没有非预期变化。
- 不要让配置重新变复杂。v11 的配置目标是主配置清晰，细节默认隐藏，实验差异通过小配置文件和 antenna.json 表达。

### 10.4 需要掌握的基本知识

新对话 AI 在动手前至少要理解以下知识点。如果不确定，应先调研或和用户讨论：

- RT 基本链路：几何寻径生成多径，EM 层计算每条路径的传播、反射、透射、绕射和接收。
- P2P SBR：发射射线、确定性 R/T 分裂、接收球命中、路径去重、独立 Tx-D-Rx 绕射。
- 几何路径语义：`GeometricPath` 是天线无关的路径描述，不是电场结果。
- 复矢量电场：每条路径需要保留复振幅、相位和极化方向。
- TE/TM 分解：反射和透射系数要作用在局部极化分量上。
- UTD/绕射基础：当前实现是独立绕射，不支持 R/T-D 混合路径。
- Jones 极化：天线不仅有增益方向图，还有复数极化响应。
- 天线姿态：forward/up/right 定义会直接影响方向图查询。
- CIR/PDP/APS：CIR 要考虑复数相干叠加，PDP 可有相干/非相干统计，APS 需要 AoA/AoD 角度网格。
- XPR/MEG/K 因子/RMS delay spread：这些是论文结果指标，不是简单打印字段。

### 10.5 什么时候必须头脑风暴或先和用户确认

以下情况不要直接落地，应先分析并和用户讨论：

- 涉及理论定义选择：例如 Tx 功率到底作为绝对 W、dBm，还是归一化功率进入 EM。
- 涉及论文指标口径：例如 MEG 用观测 APS 还是传播 APS，PDP 使用相干还是非相干统计。
- 涉及主链改变：例如修改 `GeometricPath` 结构、改变 SBR 路径生成方式、改变路径去重标准。
- 涉及配置结构变化：例如新增必填字段、删除字段、改变 `config.json`/`antenna.json` 语义。
- 涉及 coverage、大规模 Rx、GPU、OptiX、R/T-D 混合路径。
- 涉及可能改变 meeting/412 基线路径数或路径序列的修改。

推荐话术：

> 这里不是单纯代码问题，而是仿真口径问题。我先列出两个或三个可选定义，说明它们对论文结果、配置和导出的影响，再和你确认选哪一个。

### 10.6 如何进行必要调研

调研不是找一段网上文字贴进代码，而是为了确认公式、指标定义和工程口径。调研时按这个顺序：

1. 优先读项目已有文档：`document/论文相关`、`v11基线文档.md`、`v11配置字段完全参考手册.md`、本文件。
2. 再读当前代码调用链，确认项目实际实现，而不是只相信文档。
3. 涉及标准公式时，优先查 ITU、IEEE、经典教材或开源成熟 RT/信道建模实现。
4. 将调研结论落成“项目采用的定义”，不要同时保留多个互相冲突的口径。
5. 把结论写回文档，保证下一次新对话不用重新猜。

涉及网络或论文资料时，应明确来源；如果没有可靠来源，应标注为工程假设。

### 10.7 每一步开发都应回答的三个问题

任何 v11 子任务都必须回答：

1. 目的是什么？
   - 是为了修正理论口径、补齐论文指标、提升可验证性，还是改善开发环境。
2. 完成后结果应该怎样？
   - 是新增文件、字段、图表、测试，还是让某个解析校验通过。
3. 如何检验成功？
   - 是 Release 编译、基线仿真、单元测试、Python 脚本、JSON 字段检查，还是和解析公式对照。

如果这三个问题答不清，就说明还不能进入代码修改。

### 10.8 推荐的单任务执行模板

每个具体任务建议按下面模板推进：

1. 读文档：确认任务与 v11 哪个阶段对应。
2. 读代码：从入口函数读到实际实现函数。
3. 写计划：列出最小修改点、预期影响和回归方式。
4. 修改代码：只改与任务直接相关的文件。
5. 编译验证：优先 Release。
6. 最小功能验证：用小配置或已有 baseline 检查输出。
7. 基线回归：确认 meeting/412 没有非预期变化。
8. 写文档：把新字段、新口径、新限制写回 v11 文档或配置手册。
9. 汇报结果：说明改了什么、为什么、如何验证、剩余风险。

推荐话术：

> 这一步我只改最小闭环：先确认输入字段，修正核心计算，再把输出导出，最后用一个可复现配置验证。不会同时重构无关模块。

### 10.9 V11.1 开发话术：EM 语义校准

目的：

- 让路径级 EM 输出的功率、相位、复振幅语义明确。
- 为后续 CIR/PDP/XPR/MEG 实验打基础。

完成后应该是：

- `phase_rad` 等于最终接收复电压的相位。
- `tx_power_dBm` 是否进入 EM 初始幅度有明确实现和文档说明。
- 路径导出中能看出每条路径的复振幅、功率和相位。
- ideal LOS 场景可以用 Friis 公式解释。

检验方式：

- Release 编译通过。
- 检查 `amplitude_real/amplitude_imag` 与 `phase_rad` 是否一致。
- 构造或使用最小 LOS 场景，对比 `Pr/Pt = Gt Gr (lambda / 4 pi d)^2`。
- meeting/412 几何路径数不应改变。

需要注意：

- 如果只改 EM，不应影响 `precise_paths.json` 中几何节点数量。
- 不要把路径筛除阈值强行加入 P2P EM。P2P 精确模式默认保留全量几何路径，后续分析再筛。

### 10.10 V11.2 开发话术：天线查询可信化

目的：

- 确保天线方向图、Jones 极化和姿态查询没有坐标系或周期插值错误。

完成后应该是：

- `phi=359°` 和 `phi=0°` 附近查询连续。
- ideal 天线不随姿态改变结果。
- patch/horn/dipole 的主瓣、背瓣趋势符合方向图文件。
- Tx/Rx 的 forward/up/right 约定在文档、配置和代码中一致。

检验方式：

- 写一个最小查询脚本或测试程序，查询 0/5/355/359/360 度。
- 检查 gain 和 Jones 响应是否连续。
- 使用同一几何路径更换天线配置，确认输出变化有物理解释。

需要注意：

- Excel、CSV 或商业软件导出的天线数据必须先统一角度定义，再导入。
- 不要只看 gain，Jones 复数相位同样重要。

### 10.11 V11.3 开发话术：指标聚合与导出

目的：

- 让一次仿真输出直接服务论文画图，不再依赖手工从路径 JSON 里临时统计。

完成后应该是：

- `channel_summary.json` 保留基础摘要。
- 新增或补全 `cir`、`pdp`、`aps_2d_grid`、`xpr_stats`、`channel_stats`、`meg` 等输出。
- 所有输出都带单位、字段名清晰、可被 Python 直接读取。

检验方式：

- 对一个 Rx 运行基线配置，检查输出文件是否齐全。
- 用 Python 读取输出并画出至少一张 PDP 和一张 APS。
- 检查总功率、最强路径、RMS delay spread、K 因子是否数值合理。

需要注意：

- CIR 是复数相干量，PDP 可以有相干和非相干两类统计，不能混淆。
- APS 应明确是 AoA 还是 AoD，是观测后功率还是传播参考功率。

### 10.12 V11.4 开发话术：实验配置与批处理

目的：

- 把论文实验变成可重复运行的配置集合，而不是手动改配置。

完成后应该是：

- E1/E2/E3/C1/C3 都有独立配置目录。
- 每个配置有明确 `run_id`。
- 一条命令可以批量运行某组实验。
- 输出可以被汇总脚本整理成 CSV。

检验方式：

- 跑一组最小 E1，例如 ideal 与 patch 两个配置。
- 检查输出目录、配置快照、日志和指标表。
- 确认相同几何参数下，不同天线只改变 EM 指标，不改变几何路径集合。

需要注意：

- run_id 必须可追溯实验变量，例如天线类型、yaw、ray_count。
- 不要通过改 main 入参或重编译切换实验。

### 10.13 V11.5 开发话术：论文图表闭环

目的：

- 将仿真输出变成论文可用图表，并能解释图中的物理原因。

完成后应该是：

- 有 PDP 叠加图、APS 热图、XPR CDF、功率-yaw 曲线、收敛曲线等脚本。
- 每张图能追溯到 run_id 和配置。
- 图中的变化能用方向图、极化匹配、姿态或路径结构解释。

检验方式：

- 至少完成 E1/E2/E3 各一张可用图。
- 图表脚本不依赖硬编码绝对路径，或在文档中明确路径约定。
- 数据异常时能追溯到路径级 JSON 检查。

需要注意：

- 不要只追求图好看。论文图首先要可解释、可复现、字段定义清楚。

### 10.14 V11.6 开发话术：几何敏感性与寻径可信度

目的：

- 证明当前 SBR 几何参数选择是合理的，尤其是射线数、接收球半径、绕射开关。

完成后应该是：

- C1 给出 ray_count 收敛趋势。
- C2 给出绕射开启/关闭对功率和路径数的贡献。
- C3 给出 rx_sphere_radius 对路径数和主结果的影响。
- 对关键场景可继续与独立 SBR 对照。

检验方式：

- 比较路径数、总功率、主径功率、PDP 主峰、XPR。
- 检查 LOS 是否被去重为唯一主 LOS。
- 检查绕射路径数量和功率占比是否符合场景直觉。

需要注意：

- 当前绕射是独立 `Tx-D-Rx`，不能在论文或文档中写成已经支持 R/T-D 混合。
- 接收球半径变化导致路径数变化是正常的，关键是主指标是否收敛。

### 10.15 开发完成后的汇报格式

每次完成一个任务，建议按以下格式汇报：

```text
完成内容：
- 改了哪些文件，解决了什么问题。

验证结果：
- Release 是否编译通过。
- 跑了哪些配置。
- 关键输出文件在哪里。
- 关键数值是否符合预期。

对基线影响：
- meeting/412 几何路径数是否变化。
- EM 指标变化是否属于预期理论修正。

剩余风险：
- 哪些公式或口径还需要用户确认。
- 哪些输出还只是工程近似。
```

### 10.16 最重要的工作习惯

v11 开发最怕的是“看见一个字段就改一个字段，看见一个函数就补一个函数”。正确习惯是：

1. 先想清楚仿真物理含义。
2. 再确认项目当前实现。
3. 再决定最小修改。
4. 再用基线和解析用例验证。
5. 最后把结论写回文档。

一句话给新对话 AI：

> v11 的核心不是把代码堆得更多，而是让每一个输出字段都知道自己从哪里来、物理上代表什么、能被哪一个论文实验使用、又能用什么基线或解析公式证明它没有跑偏。
