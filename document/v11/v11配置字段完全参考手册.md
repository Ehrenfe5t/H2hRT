# v11 配置字段完全参考手册

> 日期: 2026-06-26  
> 依据: 当前 `AppConfig.h`、`AppConfigJsonCodec.cpp`、`AppConfigValidator.cpp`、`RtPipeline.cpp`、`RtRealChainRunner.cpp`、`SbrEngine.cpp`、`MaterialDatabase.h` 实际代码。  
> 用途: 一方面记录当前 H2hRT v11 CPU P2P 主链真实配置字段；另一方面记录下一轮“统一 config.json + 子配置文件”的配置收敛方案。

## 0. 当前主链事实

当前主链是:

```text
配置读取
  -> OBJ/材质映射导入
  -> 材质库绑定
  -> 场景预处理
  -> FaceBVH + WedgeAcceleration
  -> CPU SceneQuery
  -> SBR P2P 几何寻径
  -> GeometricPath
  -> SolveSinglePathEM / A1 real chain
  -> CIR/PDP/APS/路径导出
```

当前阶段不使用旧 `SearchEngine` 精确寻径，不启用 coverage 主链，也不启用 CUDA/OptiX。

OptiX/CUDA 说明:

- OptiX 以前只是 `SceneQuery` 的可选 GPU 求交/遮挡后端，不是 SBR 寻径算法本身。
- CPU P2P SBR 不需要 OptiX。
- 当前工程已移除 CUDA/OptiX 编译、链接和配置入口。
- CPU FaceBVH 仍然必要，它是少量 Tx/Rx 准确仿真也需要的场景求交结构。

## 1. 配置收敛总原则

你的总体思路是合理的: 配置文件应成为 `main` 和 `RtPipeline` 的统一输入，用户不应该为了换场景、换 Tx/Rx、换仿真需求去改 main 入参或重编译。

建议长期形态:

```powershell
RT.exe config.json
```

`config.json` 是总入口，只放每次仿真必须显式修改的顶层信息，以及若干子配置文件路径。复杂对象，如天线列表、Rx 表格、coverage 网格、日志细节，可以放入子 JSON 或外部表格。

建议目标结构草案:

```json
{
  "run_id": "case_001",
  "scene": {
    "obj_file": "scene.obj",
    "material_binding_file": "scene_material_map.json",
    "material_database_file": "ItuMaterial.csv"
  },
  "antenna_file": "antenna.json",
  "simulation": {
    "enable_p2p": true,
    "enable_coverage": false
  },
  "p2p": {
    "tx_ids": ["tx01"],
    "rx_ids": ["rx01", "rx02"],
    "ray_count": 1000000,
    "max_depth": 6,
    "max_reflection": 4,
    "max_transmission": 1,
    "max_diffraction": 1,
    "power_threshold_dB": -180.0,
    "rx_sphere_radius_m": 0.1
  },
  "em": {
    "frequency_list_hz": [2400000000.0]
  }
}
```

注意: 上面是建议目标形态，不是当前代码已经完全支持的 JSON schema。

## 2. 顶层 `run_id`

当前代码:

- 字段位置是 `app_runtime.run_id`。
- 输出目录、日志上下文、配置快照都依赖它。

建议:

- `run_id` 上移到总入口头部，成为每次仿真的必填项。
- `app_runtime` 其余字段降级为隐藏的日志/运行细节配置。

原因:

- `run_id` 是仿真任务身份，必须显式、醒目、稳定。
- `mode`、`log_level`、`enable_console_logging`、`enable_file_logging`、`log_file_path`、`config_snapshot_directory`、`cache_directory` 都不是电波传播仿真的物理参数。

日志策略后续单独整理:

- 控制台只输出关键阶段、路径数量、EM 数量、失败原因、输出目录。
- 文件日志保留详细诊断。
- 当前控制台和日志确实偏混乱，应后续单独设计 `logging` 默认策略。

## 3. 场景输入应收敛为三件套

当前代码:

- `scene_import.source_file`: 必用，OBJ 场景路径。
- `scene_import.source_format`: 解析，但当前只支持 OBJ。
- `scene_import.scene_material_map_file`: 必用，场景材质绑定文件。
- `material.material_database_file`: 必用，材料数据库 CSV。
- `scene_import.coordinate_transform`、`normalize_object_names`: 已解析，但普通仿真不应强制暴露。

建议目标:

```json
"scene": {
  "obj_file": "scene.obj",
  "material_binding_file": "scene_material_map.json",
  "material_database_file": "ItuMaterial.csv"
}
```

建议删除或隐藏:

- 删除用户级 `source_format`，由文件扩展名判断；当前只接受 `.obj`。
- 隐藏 `coordinate_transform`，默认 `"none"`，作为高级导入选项保留。
- 隐藏 `normalize_object_names`，默认开启或按导入器内部策略处理。

## 4. 场景预处理配置建议隐藏

当前代码:

- `scene_preprocess.enable_wedge_build` 控制是否构建绕射楔边。
- `scene_preprocess.enable_scene_cache` 控制 scene cache。
- `scene_preprocess.bvh_leaf_size` 控制 CPU FaceBVH 叶节点大小。
- `scene_preprocess.rebuild_normals` 已解析。

建议:

- `scene_preprocess` 整块从普通用户配置中隐藏。
- 是否构建绕射楔边由寻径需求推导，不再手动暴露:

```text
if p2p.max_diffraction > 0 or coverage.max_diffraction > 0:
    build wedges
else:
    skip wedge build when no module needs diffraction
```

`rebuild_normals` 的作用:

- 当 OBJ 法向缺失、不可信或方向混乱时，按几何面重新生成法向。
- 它会影响反射、透射、绕射的侧向判断。
- 如果 OBJ 本身已经有正确法向和内外语义，打开它反而可能引入差异。
- 建议默认隐藏并保持 `false`，只作为模型导入调试项。

`bvh_leaf_size` 是否需要动态:

- 它是性能参数，不是物理精度参数。
- 叶子太小，树更深；叶子太大，叶内求交多。
- 当前少量 Tx/Rx CPU P2P 不应让用户手动配。
- 后续可以内部动态设置，例如小场景 8-16，大场景 16-32。

## 5. 材料数据库与 ITU-R P.2040 实现

当前代码:

- `material.material_database_file` 必用。
- `material.missing_material_policy` 默认 `strict`。
- `MaterialDatabase::Interpolate()` 已实现 log-log 插值，本质上是两频点之间的幂律插值:

```text
epsilon_r(f) = 10 ^ lerp(log10(epsilon_r0), log10(epsilon_r1))
sigma(f)     = 10 ^ lerp(log10(sigma0),     log10(sigma1))
```

- `MaterialDatabase::ExtrapolateITU()` 也实现了 ITU-R P.2040 风格幂律外推:

```text
epsilon_r(f) = epsilon_r(f0) * (f/f0)^b
sigma(f)     = sigma(f0)     * (f/f0)^d
```

重要差异:

- 当前 `QueryByName()` 在频率落在材料库两个频点之间时使用 log-log 幂律插值。
- 当前 `QueryByName()` 在频率低于最小频点或高于最大频点时，直接返回边界频点值。
- 也就是说，`ExtrapolateITU()` 已存在，但当前查询链路没有自动用于越界外推。

建议:

- `material_database_file` 合并到 `scene` 块。
- `missing_material_policy` 保持默认 `strict`，普通配置隐藏。
- `missing_material_policy` 仍有必要，因为场景绑定缺失材料时必须明确是失败还是 fallback。正式仿真应 strict。
- 后续单独决定是否启用频率越界 ITU 幂律外推。

## 6. 天线配置建议放入 `antenna.json`

当前代码:

- `antenna` 是全局回退配置。
- `tx_antenna`、`rx_antenna` 可覆盖全局配置。
- `RxTarget` 结构已有 per-Rx 天线覆盖字段。
- `SolveSinglePathEM(..., const RxTarget*)` 支持 per-Rx 天线覆盖。
- 但当前 `RunA1RealChain` 的公开入口没有 `RxTarget` 参数，`RtPipeline` 多 Rx 循环也没有把 `rxTarget` 传入 EM，因此 per-Rx 覆盖目前是“结构和函数预留存在，主链尚未完全打通”。

建议目标:

`config.json` 只写:

```json
"antenna_file": "antenna.json"
```

`antenna.json` 写 Tx/Rx 详细信息:

```json
{
  "tx_list": [
    {
      "id": "tx01",
      "position_m": [2.38, 1.5, -9.61],
      "power_dBm": 0.0,
      "pattern_file": "tx_pattern.csv",
      "polarization_file": "tx_pol.csv",
      "forward": [1, 0, 0],
      "up": [0, 0, 1]
    }
  ],
  "rx_list": [
    {
      "id": "rx01",
      "position_m": [1.58, 1.5, -8.81],
      "pattern_file": "rx_pattern.csv",
      "polarization_file": "rx_pol.csv",
      "forward": [1, 0, 0],
      "up": [0, 0, 1]
    }
  ]
}
```

这样可以省略 `source_type`:

- `pattern_file` 为空且 `polarization_file` 为空: 理想全向/默认极化。
- `pattern_file` 非空: 使用增益方向图。
- `polarization_file` 非空: 使用极化方向图。

Tx 发射功率建议放在每个 Tx 中:

- 多 Tx 时每个 Tx 可能功率不同。
- 当前 `sbr.tx_power_dBm` 是全局字段，不适合作为长期接口。

后续预留:

- 少量 Tx/Rx: 直接写 list。
- 大量 Rx: 支持 CSV/Excel 转换后的表格文件，如 `rx_table_file`。
- 网格虚拟 Rx: 放在 coverage 配置中生成，不塞进 P2P `rx_list`。

## 7. P2P 与 Coverage 应拆成两个仿真需求块

当前代码:

- 当前主链只稳定支持 P2P SBR。
- 多 Rx 是外层循环逐个 Rx 做 P2P。
- coverage 主链冻结。
- 旧字段中 `path_search` 和 `sbr` 混在一起，用户不容易理解。

建议目标:

```json
"simulation": {
  "enable_p2p": true,
  "enable_coverage": false
},
"p2p": {
  "tx_ids": ["tx01"],
  "rx_ids": ["rx01", "rx02"],
  "ray_count": 1000000,
  "max_depth": 6,
  "max_reflection": 4,
  "max_transmission": 1,
  "max_diffraction": 1,
  "power_threshold_dB": -180.0,
  "rx_sphere_radius_m": 0.1
},
"coverage": {
  "enabled": false
}
```

规则建议:

- `antenna.json` 只描述有哪些 Tx/Rx。
- `simulation.enable_p2p` 决定是否对这些 Tx/Rx 做点对点多径。
- 多 Tx 多 Rx 默认做 Tx/Rx 组合，先机械并行，后续再优化。
- coverage 不能复用 P2P 的“全部路径都保存”模式，后续应走 RxGrid + 流式 EM 累加。

P2P 和 coverage 参数必须分开:

- P2P 追求完整路径、可复现实验、完整 EM 诊断。
- Coverage 追求大规模统计，不应默认保存全量路径。
- 二者的射线数、交互阶数、接收半径、路径裁剪策略都应独立配置。

## 8. P2P 必要寻径参数

你提出的必要参数是合理的。普通配置建议只暴露:

| 目标字段 | 当前对应字段 | 说明 |
|---|---|---|
| `ray_count` | `sbr.ray_count` | Fibonacci 射线数 |
| `max_depth` | `sbr.max_ray_depth` | 总交互深度 |
| `max_reflection` | `sbr.max_reflection_count` | 反射上限 |
| `max_transmission` | `sbr.max_transmission_count` | 透射上限 |
| `max_diffraction` | `sbr.max_diffraction_count` | 绕射上限；当前只做独立 Tx-D-Rx |
| `power_threshold_dB` | `sbr.ray_power_threshold_dB` | 射线分支终止阈值 |
| `rx_sphere_radius_m` | `sbr.rx_sphere_radius_m` | Rx 命中球基础半径 |

默认隐藏:

- 去重开关: P2P 默认严格去重开启。
- 路径保存: P2P 默认保存完整路径；coverage 默认不保存完整路径。
- 相似压缩/TopN: P2P 默认关闭；coverage 后续按需打开。
- `ray_tube_*`: 动态接收球细节。
- 残差字段: 诊断字段。

## 9. 动态接收球半径

当前实现:

```text
if enable_dynamic_rx_radius:
    rayTubeAngle = ray_tube_angle_rad
    if rayTubeAngle <= 0:
        rayTubeAngle = sqrt(4*pi/ray_count)
    effectiveRadius = max(rx_sphere_radius_m,
                          pathDistance * rayTubeAngle * ray_tube_radius_scale)
    effectiveRadius = clamp(effectiveRadius,
                            max(rx_sphere_radius_m, ray_tube_min_radius_m),
                            ray_tube_max_radius_m or large_number)
```

大白话:

- 射线越远越稀，固定 Rx 小球容易漏检。
- 动态半径把射线看成有角宽度的射线管，用来减少远距离漏检。
- 它是工程补偿，不是严格点命中。
- 当前对 P2P 对齐独立 SBR 是合理的，但会增加远距离命中数量。

建议:

- P2P 默认开启。
- 普通配置只暴露 `rx_sphere_radius_m`。
- `ray_tube_radius_scale`、`ray_tube_min_radius_m`、`ray_tube_max_radius_m`、`ray_tube_angle_rad` 作为高级隐藏项。

## 10. 绕射候选楔边

“候选楔边”不是最终绕射路径，而是搜索时先筛出来的一批可能发生绕射的边。

流程:

```text
场景预处理:
  找共享边 -> 建楔边 -> 判断是否可绕射 -> 建 WedgeAcceleration

寻径:
  查询可能相关的楔边候选
  -> 对候选楔边求 Fermat/Keller 绕射点
  -> 检查 Tx-D、D-Rx 可见性
  -> 生成真正 Tx-D-Rx path
```

当前相关字段:

- `sbr.wedge_max_distance_m`: 候选楔边最大搜索距离。
- `sbr.wedge_max_candidates`: 最多取多少个候选楔边。

建议:

- 普通 P2P 配置先隐藏这两个参数。
- 默认值由场景尺度和接收半径推导，或先保留当前默认。
- 只有绕射漏检或候选过多时再作为高级调试项暴露。

## 11. 残差字段

残差是“这条几何路径是否满足物理几何约束”的诊断量，不是普通仿真用户必须配置的参数。

当前会记录:

- `reflection_residual_m`: 反射几何误差诊断。
- `max_snell_residual`: 透射 Snell 定律误差诊断。
- `max_keller_residual`: 绕射 Keller cone 误差诊断。
- `geometry_residual`: 上述残差的综合最大值。

当前状态:

- SBR 路径会计算并导出这些诊断。
- 当前主链主要用 `ValidateSbrPathForEmReady` 做 EM-ready 检查。
- `enable_path_residual_filter` 等字段尚未形成清晰的用户级过滤策略。

建议:

- 残差配置全部隐藏。
- debug/validation 模式导出即可。
- 后续如果需要严格验证，再设计统一 `debug.validation` 子块。

## 12. EM 频率配置

当前代码:

- 单频点仿真使用 `em_solver.frequency_hz`。
- `solver_mode` 当前主要用于排除 `"Coverage"`，使 v11 主链走 P2P。
- `frequency_sweep.enabled=true` 时会生成 CFR/观测 CIR。
- `BuildBroadbandCFR_FixedGain` 是固定中心频点路径复振幅，再按不同频率旋转相位的窄带近似。
- `BuildBroadbandCFR_FrequencySweep` 当前直接返回 unsupported，逐频点重算材料/Fresnel/EM 尚未实现。

建议目标:

```json
"em": {
  "frequency_list_hz": [2400000000.0, 2500000000.0]
}
```

含义:

- 单频点: list 只有一个频点。
- 多频点: 对同一几何路径逐频点计算 EM，材料参数按频点查询。
- 默认不逐频点重寻径，因为几何路径主要由场景几何决定。

建议取消用户级 `solver_mode`:

- 当前主链本来就是 P2P precise。
- 是否做 coverage 应由 `simulation.enable_coverage` 控制，而不是 `em_solver.solver_mode` 字符串。

`frequency_sweep` 建议隐藏:

- 当前 fixed-gain CFR 可以作为后处理工具。
- 逐频点 EM 未完成，不应作为正式用户配置暴露。

## 13. APS 与 MEG 字段

当前字段:

- `aps_theta_bins`: APS theta 分箱数。
- `aps_phi_bins`: APS phi 分箱数。
- `aps_export_2d_grid`: 是否导出二维角功率网格。
- `compute_meg`: 是否计算 Mean Effective Gain。

当前代码注意:

- `BuildAPS` 当前主要用每条 EM 路径的 AoA theta 和功率生成 APS entry。
- 2D APS/MEG 完整工程链路仍需要继续审查。

建议:

- 单频点 P2P 基础仿真不需要用户显式设置。
- 作为输出分析选项隐藏在高级 `output.analysis` 或默认策略中。

## 14. 其余块是否可以隐藏

建议:

- `output`: 普通用户只需要 `run_id`。路径、CIR、PDP、APS 默认按主链导出；细分开关隐藏。
- `validation`: 默认开启基本校验，自检作为开发模式选项隐藏。
- `experiment`: 如果只用于 cache/meta，可隐藏或并入 run metadata。
- `numeric_tolerance`: 全部隐藏，保持稳定默认；仅几何鲁棒性调试时暴露。
- `pipeline`: 全部隐藏。用户不应关心 stage0/stage4 等内部阶段名。

## 18. 第二轮确认结论

本节记录针对绕射、材料频率模型、天线文件格式、多频点和输出命名的第二轮讨论结论。

### 18.1 P2P 绕射边不应被少量候选截断

确认结论:

- 对少量 Tx/Rx 的 P2P 准确仿真，绕射寻径应检查全部可绕射楔边，至少不能因为候选数量上限漏掉本应存在的 Tx-D-Rx。
- 当前激活的 P2P 代码满足这一点。

当前代码事实:

- `TracePointToPointDiffraction()` 遍历 `scene.wedges` 中所有 `wedge.diffractable == true` 的楔边。
- 对每条可绕射楔边依次做:
  - Tx 和 Rx 是否都在大侧。
  - Fermat 绕射点是否落在边段内。
  - Tx 到绕射点是否可见。
  - 绕射点到 Rx 是否可见。
  - Keller residual 诊断记录。

因此当前 P2P 绕射不是“只取少数候选楔边”。之前文档中“候选楔边”的说法容易误导，应理解为未来 coverage/加速场景中的工程预筛，而不是当前 P2P 准确寻径的物理截断。

后续配置建议:

- P2P 普通配置不暴露 `wedge_max_candidates`。
- P2P 默认 exhaustive over diffractable wedges。
- coverage 或大规模 Rx 后续可引入空间候选，但必须保证候选半径/索引策略不会破坏可接受的准确性，必要时提供 validation 模式做全量楔边对照。

### 18.2 材料频率模型与 ITU-R P.2040

当前实现已经包含两层能力:

1. 表内频点之间的 log-log 插值:

```text
epsilon_r(f) = 10 ^ lerp(log10(epsilon_r0), log10(epsilon_r1))
sigma(f)     = 10 ^ lerp(log10(sigma0),     log10(sigma1))
```

这等价于假设两个材料数据库频点之间满足幂律变化。

2. `ExtrapolateITU()` 中已有 ITU-R P.2040 风格幂律外推函数:

```text
epsilon_r(f) = epsilon_r(f0) * (f/f0)^b
sigma(f)     = sigma(f0)     * (f/f0)^d
```

但当前 `QueryByName()` 未自动调用 `ExtrapolateITU()`。如果查询频率低于数据库最小频点或高于最大频点，当前行为是直接返回边界频点值。

分析结论:

- 若材料 CSV 已经提供多个频点，且这些点来自 ITU-R P.2040 或同类模型，那么当前表内 log-log 插值是合理的。
- 若只提供单频点或查询频率超出表格范围，当前“边界钳位”更保守，但不等同于完整 ITU-R P.2040 幂律外推。
- 正式仿真不建议悄悄外推；外推应显式可控并输出警告。

后续建议:

```json
"material_frequency_policy": {
  "interpolation": "log_log_power_law",
  "out_of_range": "strict"
}
```

初期可以隐藏这个块，默认策略建议:

- 表内: log-log 幂律插值。
- 表外: strict 报错或 warning+clamp，二选一需要后续确认。
- 若启用外推: 明确写 `out_of_range = "itu_power_law_extrapolate"`，并在报告中记录外推材料、原频点、目标频点。

### 18.3 先搁置宽频带，做好单频点

确认结论:

- 当前阶段先做好单频点全链路仿真。
- `frequency_sweep`、宽带 CFR、IFFT 观测 CIR 暂时不作为主接口暴露。
- 当前多频点需求先按 `frequency_list_hz` 预留接口讨论，但第一阶段只要求 list 中一个频点。

后续策略:

- 单频点: 跑一次几何寻径 + 一次 EM。
- 多频点: 默认复用同一组几何路径，只逐频点重算 EM。
- 逐频点重寻径暂不作为默认行为。

### 18.4 天线方向图/极化方向图格式

不建议让 C++ 主链直接读取 Excel 作为核心仿真输入。

原因:

- Excel 适合人工编辑，不适合作为仿真引擎的稳定交换格式。
- `.xlsx` 解析依赖重、版本细节多、可重复性不如纯文本。
- 工程软件和商业仿真软件通常导出文本类方向图文件，例如 CSV、TXT、MSI/Planet、NSMA、FEKO `.ffe`、CST/HFSS far-field ASCII/CSV 等。

建议:

- `antenna.json` 作为天线元数据入口。
- 方向图和极化方向图用 CSV/TXT 或后续支持的专业文本格式。
- Excel 可以作为前处理输入，但应先转换成 CSV/JSON，再进入 RT 主链。

推荐基础 CSV 格式:

```csv
theta_deg,phi_deg,gain_dBi
0,0,2.15
5,0,2.10
```

推荐极化/复远场 CSV 格式:

```csv
theta_deg,phi_deg,gain_dBi,Etheta_re,Etheta_im,Ephi_re,Ephi_im
0,0,2.15,1,0,0,0
```

后续可扩展:

- 支持 FEKO `.ffe`。
- 支持 CST/HFSS 导出的 far-field table。
- 支持 Excel 到 CSV/JSON 的转换工具，但不把 Excel 作为核心运行时格式。

### 18.5 多频点默认复用同一组几何路径

确认结论:

- 多频点默认复用同一组几何路径。
- 每个频点重新查询材料参数并重新计算 EM、相位、Fresnel、透射损耗等。
- 几何寻径不随频点重复执行，除非后续显式打开高级选项。

这符合当前阶段目标:

- 减少重复寻径成本。
- 保证不同频点结果差异主要来自 EM 频率响应。
- 避免频点越多导致几何寻径时间线性爆炸。

### 18.6 多 Tx/Rx 输出命名

确认结论:

- 多 Tx/Rx 输出目录采用 `Tx*-Rx*` 命名方式。

建议目标:

```text
output/{run_id}/Tx01-Rx01/
output/{run_id}/Tx01-Rx02/
output/{run_id}/Tx02-Rx01/
```

这样比当前仅按 `rx_id` 命名更清楚，尤其适合后续多 Tx。

### 18.7 Coverage 继续搁置

确认结论:

- coverage 详细配置先保留接口或搁置。
- 等后续真正引入 RxHashGrid、大规模 Rx、流式 EM 累加时，再设计 coverage 的完整配置。
- 当前不要让 coverage 字段污染 P2P 主配置。

### 18.8 控制台输出稍后讨论

确认结论:

- 先最终确定配置 JSON 输入。
- 完成对应代码整理。
- 用户自测闭环验证不影响原有仿真寻径结果后，再讨论控制台输出内容。

因此当前文档只记录日志/控制台需要整理，不在本轮定义最终输出格式。
