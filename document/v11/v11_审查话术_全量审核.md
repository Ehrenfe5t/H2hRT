### 2.6 审查：关键 Bug 修复

除 V11.1-V11.6 计划内的修改外，本轮还修复了以下开发过程中发现的问题：

#### Bug A：天线 CSV 文件路径解析

**问题**：V11UserConfigAdapter 中 `ResolvePath` 使用 `lexically_normal()` 返回相对路径，但 `BuildAntennaFromBlock` 中 `LoadCsv` 从 CWD 解析，导致路径不一致。

**修复**：所有实验 antenna JSON 文件中的 `pattern_file` 和 `polarization_file` 使用绝对路径。

**审查点**：检查 `configs/v11_experiments/e1_antenna_model/antenna_dipole.json` 中的 `pattern_file` 是否为 `G:/RT/H2hRT-7.1-SBR-/configs/antennas/dipole_gain.csv`。

#### Bug B：per-path 重建天线导致带方向图时 EM 阶段卡死

**问题**：`SolveSinglePathEM` 每次调用都重建 `AntennaModel`（含 `LoadCsv` 读取 6644 行文件）。55959 条路径 → 7.4 亿行磁盘 I/O。

**修复**：
- `RtRealChainRunner.cpp`：`BuildRealPathResultSet` 中在循环外构建天线一次
- `RtRealChainRunner.h`：`SolveSinglePathEM` 新增接受 `AntennaModel*` 参数的重载
- `SolveSinglePathEM` 公共入口（5 参数版）构建天线后调用新的重载

**审查清单**：
1. [ ] `BuildRealPathResultSet` 中天线构建是否在 `#pragma omp parallel for` 之前？
2. [ ] `SolveSinglePathEM` 的两个重载是否声明一致？
3. [ ] 公共 5 参数入口是否正确构建天线后转发到 7 参数重载？
4. [ ] 验证：带方向图（dipole）的仿真时间应与 ideal 相当（~20s for 500k rays）

**验证命令**：
```powershell
# 快速对比测试
Remove-Item output\v11_quick_* -Recurse -Force -ErrorAction SilentlyContinue
Measure-Command { .\x64\Release\RT.exe config_quick_test.json } | Select-Object TotalSeconds
# 应在 2 秒内完成
```

#### Bug C：C3 接收球半径无效果

**问题**：动态射线管半径下限 `ray_tube_min_radius_m = 0.5m` 覆盖了 0.03-0.20m 的测试范围。

**状态**：已识别并在附录 E 中记录。如需真正测试接收球半径敏感性，需修改配置关闭动态半径。
