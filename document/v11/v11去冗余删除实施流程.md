# v11 去冗余删除实施流程

> 目标：在不影响现阶段完整 P2P RT 主链功能的前提下，冻结、隔离、拆分、删除 H2hRT 中与 v11 主链无关的历史寻径/coverage/实验功能，形成一个干净、直观、可验证的 v11 开发环境。
>
> 核心原则：**先定义主链，再建立基线，再断开入口，再拆分整理，最后物理删除。任何一步删除后必须能用同一套场景完成几何路径与 EM 输出闭环验证。**

---

## 1. v11 保留的唯一主链

v11 第一阶段只保留少量 Tx/Rx 的 P2P RT 仿真主链：

```text
AppConfig JSON
  -> LoadAppConfigFromJsonFile / ValidateAppConfig
  -> BuildSceneForBatch2: OBJ + material_map
  -> BuildSceneForBatch3: Edge/Wedge/BVH/Diagnostics
  -> BuildSceneForBatch4: SceneQuery
  -> RunSbrPointToPointSearch
  -> SbrEngine::RunPointToPoint
  -> GeometricPath / PathNode
  -> SolveSinglePathEM
  -> BuildCIR / BuildPDP / BuildAPS / BuildChannelStatistics
  -> ExportPaths / ExportChannel / ExportCoverage / ExportVisualization
```

当前主链配置判据：

```json
{
  "pipeline": {
    "precise_path_source": "SBR",
    "enable_stage4_precise_em": true
  },
  "sbr": {
    "enabled": false,
    "store_paths": true
  }
}
```

说明：

- `pipeline.precise_path_source="SBR"` 表示 precise/P2P 几何路径来自迁移后的独立 SBR。
- `sbr.enabled=false` 表示不启用旧 coverage 通道。
- P2P 结果必须保存完整 `GeometricPath`，并送入 `SolveSinglePathEM()`。
- coverage、大规模 Rx、RxHashGrid 流式 EM、TopN coverage 存储策略，全部推迟到 v11 后续阶段。

## 2. 去冗余总策略

### 2.1 不直接“大删”的原因

当前 H2hRT 不是一个单纯代码量大的问题，而是历史功能、实验功能、现阶段主链混在一起：

- `SearchEngine` precise 搜索仍存在，但 v11 主链已经走 SBR P2P。
- `SbrEngine.cpp` 同时包含 P2P、legacy coverage、wavefront、megakernel、coarse pass 等多套逻辑。
- `RtPipeline.cpp` 同时编排 precise、coverage、stage0/1/2/3、path reuse、legacy SBR power 等分支。
- 配置文件暴露大量与当前 P2P 无关的字段，导致用户难以判断哪些参数真正生效。

如果直接删除，很容易误伤：

- 当前能跑通的 P2P + EM 主链。
- 输出格式。
- 可视化/对照脚本。
- 材质、天线、EM 模块间传值。

所以正确流程是：

1. 先做基线。
2. 冻结旁支入口。
3. 拆分主链文件。
4. 删除无入口、无依赖、无测试价值的资产。
5. 每一批删除后做闭环验证。

### 2.2 删除判据

一个文件/函数/配置字段只有同时满足以下条件，才允许物理删除：

- v11 P2P 主链不调用。
- 工程编译不依赖。
- 输出结果不依赖。
- 文档和配置中已有替代说明。
- 删除后 412/meeting 对照场景几何路径数量、路径序列、EM 有效结果不退化。

若只满足部分条件，先标记为 `deprecated` 或移动到 `archive`，不立即删除。

## 4. 分阶段实施

## Phase 0：只标记，不删除

### 删除/冻结对象

不做物理删除，只在文档和代码注释中标记以下内容不是 v11 P2P 主链：

- `SearchEngine` precise 搜索主线。
- `SbrEngine::Run()` legacy coverage。
- `SbrEngine::RunWavefront()`。
- `SbrEngine::RunMegakernel()`。
- `SbrEngine::RunCoarsePass()`。
- `PathReuseEngine`。
- `RxSeedSampler`。
- `pipeline.enable_stage0_precompute`、`enable_stage1_coarse_sbr`、`enable_stage2_constrained_search`、`enable_stage3_path_reuse`。
- coverage 自动网格配置。

### 为什么先标记

这一步的价值是建立“主链边界”，让后续删除有依据。

标记后，开发者能明确：

- v11 当前只保证 P2P。
- coverage 功能没有消失，但不参与当前主链。
- 旧 SearchEngine 不再作为论文主算法路径。

### 自测闭环

只改注释/文档时：

- 不需要全量场景跑。
- 至少执行一次 Debug 编译，确认注释修改没有误碰代码。

通过标准：

- 编译通过。
- 运行任意一个 P2P smoke 配置通过。

- 少量 P2P 用户只填直观字段。
- coverage 字段不干扰 P2P。
- SearchEngine 字段不干扰 SBR。

### 自测闭环

用新模板生成 412/meeting 配置后，与旧 compare 配置对比：

- 路径总数一致或只因明确参数差异而变化。

通过标准：

- 新旧配置同参数下输出路径数基本一致。
- `precise_paths.json` 格式不变。
- A1 EM 输出不退化。

## Phase 2：拆分 `SbrEngine.cpp`，不改变行为

### 整理对象

当前 `core/search/SbrEngine.cpp` 是 v11 最大混乱源。建议先做无行为变化拆分：

```text
core/search/
  SbrEngine.h
  SbrEngine.cpp                    // 只保留入口分发和公共 facade
  SbrPointToPointTracer.cpp/h       // RunPointToPoint 主流程
  SbrRayState.cpp/h                 // P2P TraceState、分支生成
  SbrDiffractionTracer.cpp/h        // Tx-D-Rx 绕射
  SbrPathDeduplicator.cpp/h         // 严格签名、二级校验、编号
  SbrPathValidator.cpp/h            // EM-ready 校验
  SbrLegacyCoverage.cpp/h           // 暂存 Run/RunWavefront/RunMegakernel
```

### 为什么先拆分

这一步不是删功能，而是先把“当前主链”和“历史功能”物理分区：

- 后续查 P2P 时只看 P2P 文件。
- coverage 代码还在，但不会污染主链阅读。
- 删除风险降低。

### 不允许改变的接口

```cpp
SbrCoverageResult SbrEngine::RunPointToPoint(const SbrContext& context) const;
SbrCoverageResult SbrEngine::Run(const SbrContext& context) const;
SbrCoarseResult SbrEngine::RunCoarsePass(const SbrCoarseContext& context) const;
```

第一轮拆分只移动代码，不改签名、不改算法、不改配置。

### 自测闭环

拆分前后对比：

- Debug 编译通过。
- 412 三个 Rx 路径数一致。
- meeting 路径数一致。
- `TdR` 数量一致。
- 输出 `path_signature` 稳定。
- EM 有效结果数一致。

通过标准：

- 路径数不变。
- 序列分布不变。
- 总长度差异小于 `1e-6` 或完全一致。
- 输出 JSON schema 不变。

## Phase 3：冻结 SearchEngine precise 主线

### 冻结对象

- `core/search/SearchEngine.*`
- `ReflectionExpander.*`
- `TransmissionExpander.*`
- `DiffractionExpander.*`
- `PathState.*`
- `PathReuseEngine.*`
- `RxSeedSampler.*`
- `ConstrainedSearchConfig` 相关分支。

### 为什么冻结

v11 主算法已经切换到独立 SBR 迁移版。SearchEngine 存在以下问题：

- 高阶反射/透射有启发式截断。
- 与独立 SBR 输出机制不一致。
- 配置字段复杂，容易误导主链调试。

但它仍可能作为历史对照或论文中“旧版本方法”存在，所以第一步是冻结而不是删除。

### 具体操作

1. 在 `RtPipeline` 中保留 `precise_path_source="SearchEngine"` 的兼容入口，但 v11 模板不再暴露。
2. 文档标记 SearchEngine 为 legacy precise。
3. 若要进一步清理，移动到 `core/search/legacy/`，保持工程可编译。

### 自测闭环

冻结后主测 SBR P2P：

- `precise_path_source="SBR"` 完全不受影响。
- 旧 SearchEngine 配置如仍保留，应至少能编译，不要求作为 v11 主功能验证。

通过标准：

- SBR P2P 全链路不变。
- 工程编译不因为移动文件而失败。

## Phase 4：冻结 coverage 主线

### 冻结对象

- `SbrEngine::Run()`。
- `RunWavefront()`。
- `RunMegakernel()`。
- coverage 自动 Rx 网格生成。
- coverage `sbr_coverage.json` 标量功率输出。
- SBR coverage 后处理里的 SBR→Precise EM 批处理。

### 为什么冻结

coverage 后续设计方向应是：

```text
Rx 网格生成
  -> RxHashGrid
  -> SBR 命中
  -> 路径即时送入 EM
  -> 每个 Rx 在线累加
  -> 只保存 TopN/调试路径
```

而不是当前这种：

```text
几何层保存大量路径
  -> coverage 后处理再整体交给 EM
```

因此旧 coverage 通道不能作为 v11 P2P 主链的一部分继续扩展。

### 自测闭环

冻结 coverage 后只要求：

- P2P 配置不受影响。
- `sbr.enabled=false` 时不再经过 coverage 代码。
- 文档明确 coverage deferred。

若仍要保留旧 coverage 配置：

- 编译通过。
- 可以运行 smoke，但不作为 v11 正式验收。

通过标准：

- P2P 运行时间、路径数、EM 结果不变。
- coverage 相关字段从 v11 P2P 模板消失。

当前 `RtPipeline.cpp` 同时包含：

- 材质数据库加载。
- SearchEngine precise。
- SBR P2P。
- 多 Rx 循环。
- coverage。
- stage0/1/2/3 加速链。

建议拆分为：

```text

  RtConfigStage.cpp/h              // 配置读取/校验/日志
  RtSceneStage.cpp/h               // 场景导入/预处理/query
```

- SBR 给 EM 提供哪些字段。

### 自测闭环

- 命令行入口不变。
- 配置路径参数不变。
- 输出目录结构不变。

通过标准：

- Debug 编译通过。

## 6. 删除失败时的判断规则

如果删除某批后出现路径差异：

1. 先判断是否是配置字段变化导致。
2. 再判断是否是线程顺序导致 path_id 顺序变化。
3. 再比较 path_signature、交互序列、total_length。
4. 若路径数量变化但序列分布一致，检查去重或 TopN 是否被意外打开。
5. 若绕射消失，优先检查 WedgeBuilder、large-side 判断、visibility ignored faces。
6. 若几何路径存在但 EM 结果减少，检查 `ValidateSbrPathForEmReady()` 和 `PreparePathForEM()`。
7. 若只有功率不同，检查天线配置、材质数据库、介质衰减、FSPL 是否被重复或漏算。

如果无法解释，停止继续删除，回到上一阶段。

### 7.2 配置形态

v11 推荐配置只表达 P2P 必需信息：

```text
runtime
scene
material
tx
rx_list
antenna_tx
antenna_rx
sbr_p2p
em
output
```

不再让用户在 P2P 配置中看到：

- coverage grid。
- stage0/1/2/3。
- path reuse。
- legacy SBR power。
- GPU megakernel。
- SearchEngine keep_limit。

### 7.3 功能形态

最终 v11 第一阶段应明确支持：

- 少量 Rx P2P。
- LOS。
- 多阶 R/T。
- 独立 Tx-D-Rx 绕射。
- R/T 不与 D 混合。
- 完整 `GeometricPath` 输出。
- 完整逐路径 EM。
- 复数相位相干 CIR。
- 天线方向图/极化/姿态注入。

明确不承诺：

- 大规模 coverage。
- R/T/D 混合绕射路径。
- GPU 加速。
- 频率相关天线方向图。
- 阵列互耦。
- 时变/Doppler。

## 8. 推荐实施顺序

推荐顺序如下：

1. 建立 412/meeting 基线。
2. 新增 v11 干净 P2P 配置模板。
3. 修正天线配置生效语义和配置快照。
4. 拆分 `SbrEngine.cpp`，保持行为不变。
5. 拆分 `RtPipeline.cpp`，保持命令行和输出不变。
6. 冻结 SearchEngine 和 coverage 入口。
7. 归档旧配置。
8. 清理 `.vcxproj/.filters`。
9. 建立最小自动回归脚本。
10. 再考虑物理删除 legacy 文件。

一句话总结：

> v11 去冗余不是为了“少代码”，而是为了让主链唯一、配置直观、结果可复现、理论分析能落到真实代码路径上。
